# Basic —— Web安全基本功

> 你有没有想过：一个简单的网页背后，浏览器和服务器之间说了多少"悄悄话"？
> Basic 要学的就是偷听这些对话、修改对话内容，甚至伪造身份去骗过服务器。
> 这是所有 CTF Web 题的地基，也是 Web 安全的起点。

---

## 场景一：一个网页的"源代码"里藏了什么

**场景描述**：你打开一个 CTF 题目，页面干干净净什么都没有。但 flag 可能就藏在你看不到的地方。

**原理**：浏览器从服务器拿到的 HTML 本质就是一个文本文件，里面除了可见的内容，还有 HTML 注释（`<!-- -->`）、隐藏的 JS 脚本路径、或者出题人直接写在注释里的 flag。这些东西不会显示在页面上，但都在源代码里。

**实战：四种姿势看源码**

| 方法 | 操作 | 什么时候好用 |
|------|------|-------------|
| 快捷键 | `Ctrl + U` | 最快，大部分网站都能用 |
| 开发者工具 | `F12` → Elements 面板 | 还能同时看样式、调试 JS |
| 右键菜单 | 右键 → 查看网页源代码 | 最直观 |
| URL 前缀 | `view-source:http://目标.com` | 右键被禁用时的救命稻草 |

**重点盯什么（源码里搜这三个）**：
- `<!--` —— 找 HTML 注释
- `flag{` —— 如果 flag 直接放页面里
- `.js` —— 找外部 JS 文件路径，可能有逻辑在里面

**新手避坑**：有的页面用 JS 拦截了右键菜单和 `F12`，让你打不开开发者工具。但浏览器已经下载下来的内容，没有任何脚本能阻止你查看。用 `view-source:` 前缀绕过，或者先在其他页面打开 F12，再输入目标网址，就可以了。

## 场景二：HTTP 对话里翻线索

**场景描述**：flag 不直接显示在页面上，而是藏在服务器返回的某个响应头、或者某种中间跳转过程里。你得学会"监听"浏览器和服务器之间的每一句对话。

**原理**：浏览器和服务器通信用的是 HTTP 协议，每次通信包含一个**请求**（你这边发出去）和一个**响应**（服务器返回来）。这两部分都有"头"和"体"。头里放元信息（类型、时间、状态），体里放实际内容（HTML、JSON、图片）。

**实战：三大监听手段**

| 工具 / 命令 | 怎么用 | 特点 |
|-------------|--------|------|
| F12 → Network | 刷新页面，逐条点请求看详情 | 零配置，开箱即用 |
| BurpSuite | 设置代理抓包，拦截修改 | 最专业，能改包重发 |
| `curl -v` | 命令行发请求，看完整交互 | 脚本化，适合批量操作 |

**响应里要盯紧的四个地方**：

```
 响应头（Response Headers）
   - 出题人可能在自定义头里塞 flag：X-Flag: flag{...}
   - 或者给个提示：Hint: 看一下Cookie

 状态码（Status Code）
   200 → 正常，资源存在
   301/302 → 重定向了，中间可能有信息
   403 → 被拦了（可能是WAF，也可能是真权限）
   500 → 服务器炸了，报错信息可能泄露路径

 响应体（Response Body）
   - JSON 数据的某个字段可能就是 flag
   - HTML 注释、隐藏的 `<div>` 别忘了看

 Set-Cookie 头
   - 服务器给你的"身份令牌"，可能藏着加密或编码的数据
```

**请求里也要看**：
```
 Cookie → 可能就是 base64 编码的用户名密码
 请求参数 → URL 里的 ?xxx=yyy，试着手动改改
 请求头 → User-Agent、Referer、X-Forwarded-For 可能被题目用来做校验
```

**新手避坑**：
- 状态码 403 不一定代表"这里有东西但没权限"——也可能是统一拦截/WAF。跟随便一个不存在的路径比比看，如果也是 403，那就是统一拦截，别浪费精力。
- 看到 301/302 别直接跳过，看看跳转前那个页面的响应体有没有东西。

## 场景三：改改参数，看服务器反应

**场景描述**：URL 尾巴上的 `?uid=1001` 或者表单里的 `role=member`，改一个数字、换一个单词，返回的内容就全变了——可能看到了别人的隐私数据，可能触发了隐藏功能。

**原理**：参数是告诉服务器"我要什么"的方式。GET 参数直接放在 URL 的 `?` 后面，POST 参数在请求体里。服务器拿到参数后去查数据库或者执行逻辑。如果服务器没有检查"你是否有权限访问这个参数对应的数据"，那就出现了越权漏洞。

**实战：参数修改三板斧**

| 工具 | 操作 |
|------|------|
| 浏览器地址栏 | 直接改 URL 里的值，回车就行 |
| F12 → Network → Edit and Resend | 找到请求，右键选"编辑并重发" |
| BurpSuite Repeater | 抓到包，右键 Send to Repeater，随便改 |

**参数测试清单（照着这个试一遍）**：

| 参数类型 | 你原来的值 | 改成什么试试 | 期望结果 |
|---------|-----------|-------------|---------|
| 数字型 ID | `uid=1001` | `uid=1000`、`uid=1002`、`uid=-1`、`uid=0` | 越权看别人数据 |
| 权限标志 | `level=1` | `level=9`、`level=999`、`level=admin` | 提权 |
| 金额数量 | `amount=50000` | `amount=-1`、`amount=0`、`amount=99999999` | 负数/溢出漏洞 |
| 字符串角色 | `role=member` | `role=admin`、`role=vip`、`role=root` | 角色伪造 |
| 布尔开关 | `debug=0` | `debug=1`、`debug=true` | 打开调试模式 |
| 隐藏参数 | 没有这个参数 | 手动加上 `&admin=true`、`&isVip=1` | 触发隐藏逻辑 |
| 同名传两次 | `cat=1` | `cat=1&cat=2` | 参数污染，看后端取哪个 |

**不同位置的参数**：

```
GET 参数：http://目标.com/page?key=value&key2=value2
                └───────────────────┘
                看到 ? 后面都是，浏览器地址栏直接改

POST 参数：在请求体里，地址栏看不见
           写法：username=admin&password=123
           用 BurpSuite 抓包改，或者 F12 里编辑重发
```

**新手避坑**：
- 改完参数页面没变化不一定是没漏洞——可能参数名不对，或者后端根本没用这个参数。
- 参数污染（传两个同名参数）在不同框架下行为不同：有的取第一个，有的取第二个，有的拼在一起。试试不同顺序。
- 看到数字就试试负数、零、极大值，这三个在 CTF 里出现频率超高。

## 场景四：伪造 Cookie 和 Session

**场景描述**：网站让你登录后给了一个 Cookie，你把这个 Cookie 的值改一改，就变成了管理员。

**原理**：Cookie 是服务器让你浏览器保存的一小段文本，每次请求自动带上。服务器靠它认出"你是谁"。如果 Cookie 里的数据是明文或者可预测的，你就可以伪造任意身份。常见三种情况：

1.**明文编码型**：`Cookie: userinfo=eyJ1c2VyIjoiZ3Vlc3QifQ==`（这是 base64 编码的 JSON `{"user":"guest"}`），解码改 `admin` 再编码回去。
2.**标志位型**：`Cookie: member=0` 改成 `member=1`，就能访问会员功能。
3.**Session ID 可预测**：`Cookie: session=abc` 改成已知的其他用户 session ID。

**实战：四种改 Cookie 的方法**

| 方法 | 步骤 |
|------|------|
| F12 → Application | 左侧点 Cookies，双击值直接改，刷新页面生效 |
| BurpSuite | 抓到包，找到 `Cookie:` 那行直接改，Repeater 重放 |
| 浏览器 Console | `document.cookie="key=value"` 直接设 |
| curl | `curl -H "Cookie: key=value" http://目标.com` |

**常见考点速查**：

```text
Cookie 里看到长得像 base64 的 → 解码看看
  echo "eyJyb2xlIjoiZ3Vlc3QifQ==" | base64 -d
  → {"role":"guest"}
  改成 {"role":"admin"} 再 base64 编码放回去

Cookie 里看到 0/1 的 → 翻转试试
  admin=0 → admin=1
  vip=false → vip=true

Cookie 里有 JWT 的 → 拿去 jwt.io 解码看 payload
  如果签名算法是 none，改 payload 就能直接过
```

**新手避坑**：
- 把 Cookie 整个删掉之后页面还能正常打开——这不代表没有鉴权！可能进入了"游客模式"，或者身份验证放在了其他头（比如 `Authorization`）里。要对比删除前后能访问的数据有没有变化。
- base64 是编码不是加密，`=` 结尾是它的特征。看到奇怪的字符串先试试 base64 解码。
- 有的网站同时检查 Cookie 和某个请求头，只改一个不够。

## 场景五：换种方式跟服务器说话

**场景描述**：一个 URL 用 GET 访问返回 405 不让看，换成 POST 或 PUT 可能就能访问了，甚至能上传文件。

**原理**：HTTP 定义了多种"方法"（也叫动词），每种代表不同的操作意图。同一个 URL 对不同方法的响应可能完全不同。如果服务器只限制了 GET 的权限，但忘了限制 PUT 或 DELETE，你就钻了空子。

**实战：六种请求方法一览**

| 方法 | 干啥的 | CTF 里怎么用 |
|------|--------|-------------|
| GET | 获取数据 | 默认方法，看页面内容 |
| POST | 提交数据 | 登录、传参、提交表单 |
| PUT | 上传/覆盖 | 绕过上传限制，直接写文件 |
| DELETE | 删除 | 可能触发管理操作 |
| OPTIONS | 问服务器支持哪些方法 | 探路，看 `Allow` 响应头 |
| PATCH | 部分更新 | 某些 REST API 专用 |

**怎么改方法**：

```bash
# curl 指定方法（-X 或 --request 都行）
curl --request PUT "http://目标.com/upload" --data-binary "@shell.php"
curl --request DELETE "http://目标.com/admin/delete?id=1"
curl --request OPTIONS "http://目标.com/api" -v

# 看看服务器支持哪些方法
curl -X OPTIONS "http://目标.com/" -v
# 看响应头里的 Allow: GET, POST, PUT, DELETE
```

```javascript
// 浏览器 Console 里换方法发请求
fetch("http://目标.com/api/flag", { method: "PUT" })
  .then(r => r.text())
  .then(console.log)
```

**经典套路**：
1.**GET 改成 POST**—— 有些 WAF 只检查 GET 参数，改成 POST 传同样的值就绕过了。
2.**OPTIONS 探路**—— 先问服务器"你支持哪些方法"，发现支持 PUT 就去上传木马。
3.**PUT 上传 webshell**—— 如果 PUT 方法开放且没有鉴权，直接上传一个脚本文件就能拿到控制权。

**新手避坑**：
- 改方法时注意参数位置：GET 参数放 URL，POST 参数放请求体。方法一变，参数位置也要跟着变。
- 有些后端框架会自动把 POST body 里的参数当成 GET 参数用，有些不会，试的时候两种都传试试。

## 场景六：突破前端校验

**场景描述**：表单要求密码至少 8 位、邮箱必须带 `@`，但你抓包把数据改了，后端直接接受了。说明校验只在浏览器端做了，后端根本没有同样的检查。

**原理**：前端 JS 校验只为了用户体验（少发无效请求），它**不能替代**后端的安全校验。所有从浏览器发出的数据都可以被修改、伪造、跳过。真正决定安全与否的，是服务器的后端代码。

**实战：三种绕过方式**

| 方式 | 怎么做 | 适合场景 |
|------|--------|---------|
| 抓包改数据 | BurpSuite 拦截请求，改完再放行 | 大多数情况 |
| Console 直接调函数 | `checkForm("admin", "123")` 跳过 UI 校验 | 前端校验逻辑独立时 |
| 纯脚本直连 | Python `requests` 发请求，完全不经过浏览器 | 批量操作、自动化 |

**新手避坑**：
- 前端校验绕过去了不等于漏洞存在——关键是看后端有没有做同样的校验。
- 如果后端也做了校验，那前端绕过也没用。但后端经常"忘了"对某些字段做校验。

## 场景七：重定向里翻出 flag

**场景描述**：访问 `/admin` 被自动跳转到了登录页，但跳转发生之前那个中间页面里可能就有 flag。

**原理**：服务器返回 `3xx` 状态码 + `Location` 头告诉你"去别处看"。浏览器默认自动跟着跳，所以你眼睛看到的总是最终页面。如果出题人在跳转之前的那个响应体里放了 flag，你会发现"明明访问了地址，什么都没看到"。

**实战：抓住中间页面**

```bash
# 方法一：curl 不加 -L，不跟随跳转
curl "http://目标.com/admin" -v
# 你会看到 302 响应，里面有 Location 头和可能的响应体

# 方法二：curl 加 -L 跟随跳转，拿到最终页面
curl "http://目标.com/admin" -L
```

**F12 里怎么看**：
- 打开 Network 标签
- 勾上**Preserve log**（保持日志）—— 不然跳转后之前的请求记录会被清掉
- 刷新页面，你会看到多个请求，每个 3xx 中间响应都能点开看

**SSRF + 302 高级玩法**：
> 场景：题目有一个 SSRF 漏洞，但做了域名白名单，只能请求外网地址。
> 你在自己的 VPS 上放一个 302 跳转页面，返回 `Location: http://127.0.0.1:6379`（内网 Redis）。
> SSRF 先请求你的 VPS（合法外网），收到 302 后跟着跳到内网 Redis，就达成了 SSRF 绕过。

**新手避坑**：
- 默认情况下 F12 Network 在页面跳转后会自动清空，**务必勾上 Preserve log**才能看到中间过程。
- 不只是 HTTP 跳转，`<meta http-equiv="refresh">` 和 JS 的 `window.location` 也会产生跳转效果，留意 Sources 面板里的代码。

## 场景八：HTTP Basic 认证的攻防

**场景描述**：访问一个页面，浏览器弹出登录框要用户名密码。这就是 HTTP Basic 认证。

**原理**：最原始的认证方式，流程只有两步：

```
你访问受保护页面
    ↓
服务器返回 401 + WWW-Authenticate: Basic realm="输入密码"
    ↓
浏览器弹出登录框
    ↓
你输入账号密码
    ↓
浏览器把 "账号:密码" 用 base64 编码，塞进请求头发送
    ↓
Authorization: Basic 5L2g5aW9OjEyMzQ1Ng==
```

**关键认知**：base64 是编码，不是加密。任何人都可以解码，不需要密钥。它就像是把密码写在明信片上，只是换了一种写法而已。

**实战一：解码已经抓到的凭证**

```python
# Python 解码
import base64
data = "5L2g5aW9OjEyMzQ1Ng=="
decoded = base64.b64decode(data).decode()
print(decoded)   # 输出：admin:123456
```

```bash
# Linux 命令行解码
echo "5L2g5aW9OjEyMzQ1Ng==" | base64 -d
```

**实战二：爆破 Basic 认证**

用 BurpSuite Intruder 爆破的步骤：

1. 访问目标，抓到带 `Authorization: Basic 一串字符` 的请求
2. 右键 → Send to Intruder
3. 把那串 base64 字符标记成变量（选中后点 Add）
4. Payloads 标签 → Type 选 "Custom iterator"
  - Position 1: 用户名列表（admin, root, user...）
  - Position 2: `:` （冒号分隔符）
  - Position 3: 密码列表（123456, password, admin...）
5. 添加 Payload Processing 规则：** Base64-encode**
6.**关键一步**：在 Payload Encoding 里**取消勾选**URL-encode（否则 `=` 会被编码成 `%3d`）
7. Start Attack → 看响应长度和状态码，长度不同的那个就是成功的

**新手避坑**：
- 流量包（pcap）里的 `Authorization` 头可以直接解码拿到账号密码，这是最省力的方式。
- 爆破时一定要取消 URL 编码，否则 base64 末尾的 `=` 符号会变成 `%3d`，导致认证永远失败。
- HTTP Basic 认证现在已经很少在生产环境用了，但 CTF 里几乎每年都有。

    |      (F12 Sources 看每个 JS 文件内容)     |

    |      (看看响应头有没有蹊跷)               |
    |                                        |
    |--- ⑤ 你点击按钮/提交表单 ---------------->|

## 知识点总结表

| 知识点 | 核心一句话 | 常用工具/方法 | 新人易错点 |
|--------|-----------|-------------|-----------|
| 查看源码 | 服务器返回的所有内容都能看到 | Ctrl+U, F12, view-source: | 以为右键被禁就看不了 |
| 请求响应分析 | 头和体里都可能藏 flag | F12 Network, BurpSuite, curl -v | 只看响应体，忽略响应头 |
| 参数修改 | 改 URL/表单的值，观察反应 | 地址栏直改, BurpSuite Repeater | 只改值不试类型（负数/极大值/删参数） |
| Cookie/Session | 身份凭据可以伪造 | F12 Application, curl -H | 以为 base64 是加密安全的 |
| 请求方法修改 | 换方法访问同一 URL | curl -X, fetch() | 忘了同时移动参数位置 |
| 前端验证绕过 | 前端的限制都可以跳过 | BurpSuite, Python requests | 绕过前端≠有漏洞，看后端 |
| 重定向处理 | 中间页面可能藏信息 | curl 不加 -L, Preserve log | 没勾 Preserve log 看不到中间步骤 |
| HTTP Basic 认证 | base64 编码的密码等于明文 | base64 解码, Intruder | 爆破时忘了关 URL 编码 |
| 前端一切皆可控 | 浏览器里的没有什么是不能改的 | 所有工具的总纲 | 以为"页面看起来正常"就安全了 |
